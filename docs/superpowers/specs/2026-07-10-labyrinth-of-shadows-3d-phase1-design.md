# Labyrinth of Shadows 抜本改修 — Phase 1: 3D描画基盤（R3F）設計

- 作成日: 2026-07-10
- 対象: `src/features/labyrinth-of-shadows/`
- ステータス: 設計確定（実装計画へ）

## 背景と目的

現状の Labyrinth of Shadows は疑似3D（レイキャスティング）の迷路ホラーゲームで、敵・アイテムを**絵文字を丸に乗せて**描画している。ユーザーからの評価は「一定の難易度を超えるとゲームとして成り立たない」であり、具体的な不満は3点:

1. **グラフィックス**が貧弱（絵文字スプライト）
2. **敵から逃げる**駆け引きが破綻する（視界が狭く接近が読めない）
3. **アイテムの意味**が薄い

この改修全体では、本物の3D迷路ゲーム（react-three-fiber, 以下 R3F）へ作り替える。ただし作業は4フェーズに分解し、本ドキュメントは **Phase 1（3D描画基盤）** のみを対象とする。

### フェーズ全体像（本specはPhase 1のみ）

| Phase | 内容 | 解決する不満 |
|-------|------|------------|
| **1. 3D描画基盤（本spec）** | R3F導入・迷路/敵/アイテムを本物の3Dに・FP操作・遅延ロード。**ゲームルールは現状維持（パリティ移行）** | グラフィックス |
| 2. 逃走の駆け引き再設計 | 敵の視界/ロストルール明確化・索敵UI・AIバランス・能動的な一手 | 敵から逃げる破綻 |
| 3. アイテム再設計 | 各アイテムに明確な戦略的意味を与える | アイテムの空虚さ |
| 4. 演出・難易度・仕上げ | ライティング/サウンド演出・難易度カーブ調整 | 全体の完成度 |

各フェーズは個別の spec → plan → PR。

### Phase 1 が「解決すること／しないこと」（正直な整理）

- ✅ **グラフィックスは大きく改善**する（テクスチャ壁・3D敵・ライティング・フォグ・影）
- 🔷 逃走の**読みやすさは自動的にいくらか改善**する（本物の奥行き・指向性オーディオ・敵モデルが見える・広いFOV）。ただし**AIバランスそのものは Phase 2 の領分**
- ❌ **アイテムの意味は変わらない**（Phase 3 の領分）

→ Phase 1 単体では「まともに遊べる」には到達しない。見た目の激変と逃走の読みやすさの土台づくりが目的。

## 設計方針の確定事項（ブレインストーミングの結論）

| 論点 | 決定 |
|------|------|
| 視点/土台 | 疑似3Dを**本物の3Dに作り替え**（R3F） |
| 実装アプローチ | **react-three-fiber + drei**（React 19 と最も相性が良い宣言的バインディング） |
| カメラ視点 | **一人称（FP）** |
| アセット方針 | **プロシージャル中心**（プリミティブ＋発光マテリアル＋ライティング＋フォグ。外部3Dモデル不要） |
| 操作 | **マウスルック（ポインタロック）採用**＋キー旋回フォールバック |
| 進め方 | フェーズ分解、本specはPhase 1（ルールはパリティ維持） |

## アーキテクチャ — 「再利用」と「置換」の線引き

3D化はドメインロジックに一切触れず、**描画層だけを差し替える**。既存の Clean Architecture（domain とレンダリングの分離）がこの移行を可能にしている。

### ✅ 無改修で再利用

- `domain/`（迷路生成 `maze-service` 経由、敵AI `enemy-strategy`、BFS経路探索 `pathfinding`、スコア `scoring`、衝突判定 `collision`、`models/item`）
- `types.ts`（`GameState` ほか）
- `game-logic.ts`（ゲームルール全体）
- `constants.ts`（`CONFIG` / `CONTENT` のバランス値）
- `audio.ts` / `entity-factory.ts` / `minimap-renderer.ts`
- コンポーネント: `HUD` / `Controls` / `Minimap` / `TitleScreen` / `ResultScreen`
- フック: `use-input` / `use-audio`

### 🔄 置換

| 現状 | 置換後 |
|------|--------|
| `renderer.ts`（レイキャスティング描画） | R3F シーンコンポーネント群（`presentation/three/*`） |
| 2D `<canvas>`（メイン描画） | R3F `<Canvas>` |
| `use-game-loop.ts` の rAF 内 `Renderer.render(...)` 呼び出し | R3F `useFrame` 駆動のゲームティック |

**重要なリファクタ**: 現状 `use-game-loop.ts` は単一の rAF ループで「ロジック更新＋描画＋HUD/ミニマップ更新」を回している。R3F の `useFrame` 自体が rAF ループなので:

1. `use-game-loop.ts` の**ロジック更新ステップ（L110〜L148 相当: 時間経過・入力取得・`GameLogic.*` 呼び出し・敵更新・サウンド）を純粋なティック関数として抽出**する（例: `presentation/hooks/use-game-tick.ts` あるいは `game-tick.ts`）。
2. `Renderer.render(...)`（2D描画）呼び出しは**削除**。
3. HUD更新・ミニマップ更新・`onGameEnd` 判定・変化検知ロジックはティック関数内に維持。
4. 抽出したティック関数を `<Canvas>` 内コンポーネントの `useFrame` から毎フレーム呼ぶ。
5. `GameLogic.updatePlayer` が書き込む `g.player.x / y / angle` を毎フレーム FPカメラの `position` / `rotation` に反映する。

これにより、描画技術に依存しないロジックと R3F 描画が分離され、既存の移動・衝突判定・敵AIがそのまま3Dで動く。

### ➕ 新規追加

- 依存: `three`, `@react-three/fiber`, `@react-three/drei`
- `presentation/three/` シーンコンポーネント群（後述）

## 3Dシーン構成（プロシージャル中心）

```
<Canvas> ... presentation/three/LabyrinthScene.tsx
├── <fog>              指数フォグ = 恐怖演出 + 描画距離制限（既存 CONFIG.render.maxDepth 相当・性能対策）
├── ライティング        環境光(極小) + カメラ追従トーチ点光源（既存 renderer の torchFlicker 式を流用して揺らぎ）
├── <MazeWalls>        迷路グリッド(g.maze) → InstancedMesh のボックス群（1ドローコール）。レンガ質感マテリアル
│                      （既存 infrastructure/rendering/brick-texture の getBrickColor を色味に流用可）
├── <Floor> / <Ceiling> 平面メッシュ
├── <ItemMesh> × N     未取得アイテムを発光プリミティブで描画（鍵/回復/加速/地図/罠箱/出口ドア）
│                      emissive + 小点光源 + bob上下動（既存 renderer の bob 相当）
├── <EnemyMesh> × N    active な敵をタイプ別プロシージャル形状で描画:
│                        - 追跡(chaser)   : 威圧的な赤い塊
│                        - 徘徊(wanderer) : 半透明の漂う霊
│                        - テレポート(teleporter): 歪む渦（pulse アニメ）
│                      emissive発光 + 微動アニメ
└── <PlayerCamera>     FPカメラ。useFrame でゲームティック実行 → g.player を反映
```

### 描画の設計原則

- ホラー感は**モデル精度でなく、ライティング＋フォグ＋影**で出す。暗闇 + カメラ追従トーチ点光源が基本。
- 壁は `InstancedMesh` で1ドローコールに集約（迷路サイズ最大14でも軽量）。
- フォグの終端は既存 `maxDepth`(18) 相当に合わせ、描画距離制限と恐怖演出を兼ねる。
- ビネット/スキャンラインなどのポストプロセスは `@react-three/postprocessing`（追加依存）が必要なため **Phase 1 では任意**。まずフォグと発光で成立させ、必要なら後続で追加検討。

## カメラ・操作

- **デスクトップ**: WASD 移動 ＋ **マウスルック（ポインタロック）**。Shift ダッシュ / Space 隠れる / Esc ポーズ（ポインタロック解除）。矢印キー / A・D 旋回もフォールバックとして残す。
  - マウスの水平移動量を `g.player.angle` に反映する（既存 `CONFIG.player.rotSpeed` 相当の感度係数）。
  - ポインタロックは drei の `PointerLockControls` を使うか、`requestPointerLock` を直接扱うかは実装計画で決める（既存の入力/ポーズ制御との整合を優先）。
- **モバイル**: 既存 `Controls.tsx` のタッチUI（移動・旋回・隠れる・ダッシュ）をそのまま流用。
- 壁衝突は既存 `MazeService.isWalkable`（グリッド判定）を無改修で使用。
- カメラ FOV は快適な FP 値（例: 75°）。目線高さは迷路スケールに合わせる。

## バンドル・遅延ロード

- **懸念**: `three` は ~160KB(gzip)。本プラットフォームは13ゲームを抱えるため、他ゲームの初期ロードに影響させてはならない。
- **対策**: Labyrinth of Shadows のゲーム本体を `React.lazy` でルート分割し、`three` 系依存をメインバンドルから切り離す。`<Suspense>` でローディング画面を表示。
- ルーティング元（ページコンポーネント）の薄いラッパーで `lazy` 化する。

## テスト方針

- `domain/` ・ `game-logic` のユニットテスト: **無改修で全 green 維持**。ルールを触らないパリティ移行なので、既存テスト群が移行の安全網になる。
- 抽出するゲームティック関数: 描画に依存しない純粋ロジックとして**テスト可能な形**で切り出す（既存 `use-game-loop` のロジック部の振る舞いを維持）。
- 3Dシーン（R3Fコンポーネント）: WebGL は jsdom で動かないため**ユニットテスト対象外**。`air-hockey`（Canvas/rAF 依存でテスト困難）の前例に倣い、これは許容する。
- E2E（Playwright）: タイトル → プレイ起動が WebGL でエラーなく到達することを確認。ヘッドレス WebGL の有効化フラグが必要になる可能性があり、**リスクとして明記**（E2E が難しければ手動確認で代替する判断も可）。

## Phase 1 のスコープ境界（明確な非目標）

以下は**このフェーズでは変更しない**（後続フェーズの領分）:

- ❌ 敵AIバランス・ロストルール・能動的な一手（→ Phase 2）
- ❌ アイテムの意味の再設計（→ Phase 3）
- ✅ 難易度設定（EASY/NORMAL/HARD の size・敵数・敵速度・時間・ライフ）は**現状と完全に同一**
- ✅ 鍵/罠箱(-12秒)/回復/加速(10秒)/地図(周囲公開) のルールは**現状と完全に同一**
- ✅ ミニマップは既存の2D Canvas（`minimap-renderer.ts`）を継続利用
- ✅ スコア計算・コンボ・ライフ・スタミナ・隠れるエネルギーの各ロジックは無改修

## 受け入れ基準（Phase 1 完了の定義）

1. Labyrinth of Shadows が **本物の3D（R3F, FP視点）** で描画・操作できる。
2. 迷路・壁・敵（3種）・アイテム（全種）・出口が**プロシージャルな3Dオブジェクト**として表示される（絵文字スプライトの廃止）。
3. マウスルック（デスクトップ）＋タッチUI（モバイル）で操作できる。
4. 移動・衝突・敵AI・アイテム取得・鍵/出口・制限時間・ライフ・スコアが**現状と同じルールで動作**する（パリティ）。
5. `domain/` ・ `game-logic` の既存ユニットテストが**全て green**。
6. `three` 系依存が**遅延ロード**され、他ゲームの初期バンドルに影響しない。
7. `npm run ci`（lint:ci → typecheck → test:coverage → build）が通る。

## リスクと留意点

- **WebGL の E2E/テスト**: jsdom で不可。ヘッドレス Playwright の WebGL も不安定な場合がある。ロジックのテスト維持を最優先し、描画は手動確認で補完する。
- **バンドルサイズ**: 遅延ロードを必須とする。build 後のバンドル分割を確認する。
- **プラットフォーム初の3D依存**: `three` 系はプラットフォームに新規導入となる（CLAUDE.md 上「確認してから実行」領域 → 本設計の承認をもって合意とする）。
- **パフォーマンス（モバイル）**: `InstancedMesh` による壁の集約・フォグによる描画距離制限で軽量化。低スペック端末での実測確認は仕上げ段階で行う。
