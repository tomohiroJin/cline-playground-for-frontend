# Labyrinth of Shadows — 捕縛ペナルティ再設計（鍵ドロップ）設計書

- 日付: 2026-07-16
- 対象ゲーム: `src/features/labyrinth-of-shadows/`
- 位置づけ: 4フェーズ抜本改修（P1〜P4 全マージ済）後の**バランス較正ブラッシュアップ**
- 前提記憶: `labyrinth-of-shadows-3d-overhaul`

## 背景と課題

実プレイで「敵が弱い／緊張感がない」と感じる主因は、**ミスの帰結が軽いこと**にあると特定した。

現状の敵接触は「アクションゲーム型のHP制」になっている（`game-logic.ts:201` 付近）:

- 接触で `lives--`、無敵2.5秒、ノックバック、スコア-50、コンボリセット
- ライフ0で初めてゲームオーバー、回復薬でライフ回復

つまり「捕まる＝ライフを1削られるだけの軽い被弾」であり、ステルス/ホラーが本来持つ
「見つかった瞬間の切実さ」が構造的に存在しない。追跡状態機械（patrol→chase→search）・
隠れ・石投げといったインフラは揃っているのに、**接触という山場だけが安すぎる**。

## 目的

敵接触を「軽い被弾」から「**鍵を落とした！**という事件」へ格上げし、
"捕まりたくない" という切実さを作る。既存の隠れ／石投げ／追跡状態機械に牙を持たせる。

## スコープ

一度に複数のパラメータを変えるのはリスク（因果が不明瞭になる）なので、**変更は1点に絞る**。

### やること（これだけ）

- 敵接触時、**鍵を1個ドロップ**する（歩ける近傍セルへ散らばり、再回収可能）

### やらないこと（今回のスコープ外）

- ライフ数の変更（難易度別 5/3/2 のまま）
- 時間ロスの追加
- 敵AI・視野・速度・捜索粘りの変更
- 即死化・ライフ制廃止（迷路の逃げ場制約で理不尽になりやすく、回復薬とも衝突）
- 全鍵ロスト・入口への引き戻し（理不尽側に振れすぎる）
- 新規サウンドアセットの追加（`SoundName` 二重同期のリスク回避。既存音を流用）

### 既存挙動は維持（鍵ドロップを"上乗せ"するだけ）

- ライフ-1／無敵2.5秒／ノックバック／スコア-50／コンボリセット はそのまま

## 詳細設計

### 発動条件

既存の接触分岐（`isPlayerCollidingEnemy` かつ `!g.hiding` かつ `g.invince <= 0`）の中で、
**`g.keys > 0` のときだけ**鍵をドロップする。鍵が0なら従来通り（ライフ-1等）のみ。

### ドロップ処理

1. **1個だけ**ドロップ（`g.keys--`）
2. **落下先セル＝プレイヤー隣接の歩けるセル**を選ぶ。
   - 選定は「**敵と反対方向を優先**」— 取りに戻る動線が即・敵側へ向かわないようにする
     （デススパイラル対策）。
   - `MazeService.isWalkable(maze, x, y)` を再利用して壁/敵側を除外。
   - 歩ける隣接が無ければ**プレイヤー自身のセル**にフォールバック（そこは必ず歩ける）。
3. その位置に鍵 Item を生成して `g.items` に追加:
   `{ x, y, type: 'key', got: false, dropped: true }`
   （アイテム座標はセル整数。既存アイテムと同じ規約に合わせる）

### 再回収の挙動（スコア抜け穴の封じ込め）

鍵回収（`updateItems` の `key` 分岐, `game-logic.ts:118`）は本来 **+100pt＋コンボ加算**を伴う。
落とした鍵をそのまま拾い直せると「被弾-50 → 再回収+100 でスコア純増、しかもコンボ稼ぎ」
という抜け穴になり、ペナルティがボーナスに反転する。これを防ぐ:

- `dropped:true` の鍵 → **`g.keys++` のみ**。スコア加算なし・コンボ加算なし
  （接触時に既に `combo=0`）。メッセージは「🔑 落とした鍵を拾い直した (n/req)」
- 通常の鍵 → 従来通り +100pt＋コンボ
- **進行（reqKeys 達成）には従来通りカウント**される

### 型変更

- `Item` インターフェース（`types.ts`）に `dropped?: boolean` を追加。
  既存の鍵は `undefined`＝通常扱い。

### 多重ドロップの防止

- 接触時に無敵2.5秒がセットされる既存挙動により、同一接触で連続ドロップは起きない
  （1接触＝最大1ドロップ）。

### フィードバック演出

- **メッセージ**: 鍵を落とした場合「🔑 鍵を落とした！」。鍵0で落とせなかった時は
  従来の「💔 ダメージ！」のまま。
- **音**: 既存の `hurt` 音を流用（被弾自体は起きているので自然）。新規音は追加しない。
- **ミニマップ**: 落とした鍵は通常の鍵 Item として自動表示される（再回収の目印になり好都合）。
  特別対応なし。

## テスト計画

### ユニットテスト（TDD, Red→Green）

ロジックは純粋関数（`advanceGame` / `updateEnemyWithStrategy` / `updateItems`）に閉じており、
R3F を介さずユニットテストで挙動をほぼ完全に検証できる。

- 鍵>0で接触 → `g.keys` が1減り、`dropped:true` の鍵 Item が**歩けるセル**に生成される
- 鍵0で接触 → ドロップなし（`g.keys` が負にならない）／従来の被弾のみ
- 落下先が walkable、かつ敵と反対方向を優先／歩ける隣接が無ければプレイヤーセルにフォールバック
- **落とした鍵を再回収 → `g.keys++` のみ、スコア不変・コンボ非加算**
- 通常の鍵回収は従来通り +100pt（リグレッション防止）
- 単一接触で最大1ドロップ（無敵ガード）

### CI

`npm run ci`（lint:ci → typecheck → test:coverage → build）全緑。
ESM依存は追加しないので P4 の postprocessing 罠は無関係だが、対象ページテストは jest で回す。

### 実プレイ検証ゲート（最重要）

「CI 緑 ＝ 面白い」ではない（Falldown の教訓）。ブラウザ実機で必ず確認する:

1. 「鍵を落とす」が**"事件"として効いているか**（怖い／取り返したくなるか）
2. **捕まる頻度は妥当か**（ペナルティが効くにはある程度捕まる必要がある。
   少なすぎたら別途「敵AI軸」の相談へ）
3. **デススパイラルになっていないか**（取りに戻って連続で捕まる地獄になっていないか）
4. 落下先が**理不尽でないか**（壁/敵の中/戻れない位置に落ちていないか）

Devサーバー注意（記憶より）: 反映しない時は `fuser -k 3000/tcp` で3000解放後 `npm start`。
`pkill -f` は自滅するので使わない。ビルド2〜5分。

## リスクと対策

| リスク | 対策 |
|---|---|
| デススパイラル（取りに戻る→連続で捕まる） | 落とすのは1個のみ・敵と反対方向へ落とす・掴まれ直後の無敵/ノックバック維持 |
| スコア抜け穴（再回収で純増） | `dropped` フラグで再回収時はスコア/コンボ非加算 |
| 落下先が理不尽（壁/敵の中） | `isWalkable` で検査・walkable 隣接が無ければ自セルへフォールバック |
| そもそも捕まらずペナルティが効かない | 実プレイ検証ゲートの項目2で確認。効かなければ敵AI軸を別途相談（本設計では触れない） |
