# Falldown Shooter 連鎖崩壊システム（Chain Collapse）設計

- 作成日: 2026-07-15
- 対象: `src/features/falldown-shooter/`
- 目的: ゲーム性ブラッシュアップ。「爽快感の山場がない」という課題に対し、**連鎖（チェイン）**という山場を生む構造を導入する。

## 背景と課題

現状のゲームは機能豊富（ステージ制・難易度・パワーアップ・スキル・コンボ・ランキング・画面シェイク）だが、体験として「爽快感の山場」が弱い。

コード調査で判明した根本原因:

- 消えるのは **完全に揃った行だけ**（`Grid.clearFullLines` はテトリス式の一括行シフト）。
- 撃って開けた"穴"の上のセルは落ちてこない（**セル単位の重力が存在しない**）。
- 弾でセルを撃っても点が入るだけ（10点/セル）でラインは消えず、ラインが揃うのは「ブロック着地」時のみ。

結果、「一撃で連鎖的にドカンと消える」という溜め→解放のリズムが原理的に存在しない。これが山場が生まれにくい構造的要因である。

## 設計方針

**列ごとの重力（ぷよ式）＋連鎖ライン消去**を純粋関数として導入し、「わざと積んで一撃で大連鎖」という溜め→解放の山場を生む。連鎖数に応じてスコア倍率・演出・効果音を段階的に増幅し、体感を盛り上げる。

責務分離を厳守する:

- **ドメイン（純粋関数）**: 連鎖を一瞬で最終盤面まで解決し、順序付きの「連鎖ステップ列」を返す。決定的でテスト可能。
- **プレゼン（フック）**: そのステップ列を時間軸に沿って1段ずつ再生し、演出を発火する。

## セクション1: コアメカニクス「盤面解決（resolveBoard）」

現状の「完全行のみ消去」を、セル重力＋連鎖ライン消去に置き換える。

```
resolveBoard(grid) → { grid, chainSteps[], totalCells }
  chain = 0
  loop:
    grid = applyColumnGravity(grid)   // 各列を独立して下詰め（穴を落として埋める）
    fullRows = 揃った行を検出
    if fullRows が空 → break
    chain += 1
    chainSteps に記録(chain, 消えた行, 消えたセル数, 落下後グリッド)
    grid = fullRows を null 化
  return { grid, chainSteps, totalCells }
```

挙動:

- **列ごとの重力**を導入。撃って開けた穴の上のセルが落ちて、行が揃えば消える→落ちる→また揃えば消える…を1ループ=1連鎖として解決。
- 連鎖は「着地時」だけでなく、**弾・爆弾・スキルでセルを消した直後にも**発動しうる。
- 空中を落下中の**ブロック（shape）は対象外**。着地済みグリッドのみを解決し、責務を分離する。

トレードオフ（既知）:

- 現状のテトリス式一括シフトから**列独立重力に変わる**ため、オーバーハング（せり出し）が崩れるなど手触りが変化する。より直感的になる一方、既存 `clearFullLines` のテスト・挙動を差し替える。
- 盤面が片付きやすくなる可能性 → 難易度が下がる方向。セクション5のバランス調整で吸収する。

## セクション2: 連鎖スコアリングと既存スコアとの統合

連鎖を「明確に最大の報酬」にしつつ、既存倍率と掛け合わせても青天井にならないよう設計する。

新設する連鎖倍率テーブル（`constants.ts` に `CHAIN_BONUS` を追加）:

| 連鎖数 | 1 | 2 | 3 | 4 | 5 | 6+ |
|--------|---|---|---|---|---|-----|
| 倍率 | 1.0 | 1.5 | 2.5 | 4.0 | 6.0 | 8.0（上限） |

ライン得点の式:

```
lineScore =
   Σ(各連鎖ステップ: 消えた行数 × CONFIG.score.line × 同時消しボーナス(その行数))
   × stage × scoreMultiplier(難易度)
   × chainMult(最大連鎖数)      ← 新規
   × comboMult                   ← 既存（コンボ）
```

- 弾でセルを撃った **10点/セルはそのまま**（即時フィードバックとして維持）。
- 連鎖の各ステップ内の「同時消し（複数行）」には既存 `SIMULTANEOUS_LINE_BONUS` をそのまま適用。「1連鎖で複数行」も「多段連鎖」も両方報われる。

暴走防止ガード:

- `chainMult` は6連鎖で頭打ち（8.0上限）。
- `comboMult` は連鎖解決の**開始時点のコンボ数で1回だけ確定**（連鎖の途中で自己増幅しない）。これにより `40点×5×8×…` のような事故を防ぐ。
- 既存の `score-balance.test.ts` に連鎖ケースを追加し、想定スコア帯を回帰テストで固定する。

## セクション3: 演出・フィードバック（連鎖の"体感"設計）

連鎖数に比例して全フィードバックを段階的に増幅する。これが山場の正体。

| 要素 | 実装 | 既存資産 |
|------|------|----------|
| 連鎖カウンター | 「◯ CHAIN!」を大きくポップ、数字が跳ねる。色が段階変化（白→黄→橙→虹） | ComboDisplay を参考に新設 |
| 効果音のピッチ上昇 | 連鎖ごとに消去音を半音ずつ上げる（ぷよ的"ドレミ"上昇） | `audio.ts` に `chain(level)` 追加 |
| 画面シェイク | 連鎖数に比例して強度アップ | `use-screen-shake.ts` 流用 |
| パーティクル／花火 | 高連鎖（4+）で既存 fireworks を発火 | `Effects.tsx` / `EFFECT.fireworks` 流用 |
| フローティングスコア | 「N CHAIN! +1,240」 | `use-floating-scores` 流用 |

連鎖の"再生"アーキテクチャ:

- **ドメイン（純粋）**: `resolveBoard` が順序付き連鎖ステップ列と最終盤面を即座に返す。
- **プレゼン（フック）**: そのステップ列を短い間隔（例120ms）で1段ずつ state にコミットして再生。再生中は `isResolving` ロックで落下・スポーンループを一時停止し、「連鎖中は時間が止まってドカドカ消える」間を作る。

演出強度は `constants.ts` に集約（連鎖ごとのシェイク量・パーティクル数・色）し、調整しやすくする。

## セクション4: アーキテクチャ配置と統合ポイント

実行経路は **flat モジュール**（`game-logic.ts` / `grid.ts` / `block.ts`）を使用しており、`domain/services`・`domain/models` はランタイムから参照されていない並行構造（テスト専用・実質死蔵）。既存の active パターンに合わせ、flat 側に一貫して追加する。`domain/` 層には手を入れない（整理は本スコープ外）。

新規追加（flat・純粋関数）:

| 追加先 | 関数/定義 | 責務 |
|--------|-----------|------|
| `grid.ts` | `Grid.applyColumnGravity(grid)` | 各列を下詰め（純粋） |
| `game-logic.ts` | `GameLogic.resolveBoard(grid)` | 重力＋連鎖ループ → `{ grid, chainSteps[], totalCells }` |
| `constants.ts` | `CHAIN_BONUS`, 連鎖演出強度テーブル | 倍率・演出パラメータ |
| `audio.ts` | `Audio.chain(level)` | ピッチ上昇消去音 |
| `types.ts` | `ChainStep`, `ResolveResult` | 型定義 |

統合ポイント（`resolveBoard` を差し込む3箇所）:

1. **落下ループ（`use-game-loop.ts` 着地時）**: 現在の `clearFullLines` を `resolveBoard` に置換。
2. **弾ループ（`processBullets` 後）**: 撃ってセルを消した後に `resolveBoard` を実行 → 撃って引き金を引く連鎖が可能に（山場の主役）。
3. **スキル／爆弾（laser/blast/clear/bomb）**: セル消去後に `resolveBoard` を通す → スキルから大連鎖も狙える。

再生シーケンス（新設フック `use-chain-resolver.ts`）:

- `resolveBoard` の `chainSteps[]` を受け取り、`isResolving` ロック下で120ms間隔で1段ずつ state コミット＋演出発火。
- 完了後にロック解除し、通常ループ（落下・スポーン）を再開。ステージクリア／ゲームオーバー判定は再生完了後に評価。

## セクション5: バランス調整とテスト戦略

バランスの勘所:

- 撃つ＝セルを"減らす"行為であり、撃っても行は埋まらない（むしろ空く）。行が揃うのは主にブロック着地時。連鎖は"揃った行が消えて上が落ち、また揃う"ときに発生。よって難易度は激変せず、新しい報酬レイヤーが乗るという読み。実測で確認する。
- **一次リリースは難易度パラメータを据え置き**。連鎖導入後に実プレイ／シミュレーションで測り、必要なら spawn/fall を後追いで微調整。先回りの過剰チューニングはしない。
- テストモード（既存 `use-test-mode`）で連鎖セットアップを手動生成し、体感チューニングに活用。

テスト戦略（TDD: 純粋関数から先にテスト）:

1. `applyColumnGravity`: 穴・オーバーハング・複数列・空グリッドのユニットテスト。
2. `resolveBoard`: 0連鎖／1連鎖／多段連鎖／満盤・単一行のエッジ。
3. スコア（`score-balance.test.ts` 拡張）: 連鎖倍率の適用、コンボ二重暴走しないガード、想定スコア帯の回帰固定。
4. 統合（`integration.test.tsx`）: 弾トリガー連鎖／スキルトリガー連鎖／`isResolving` ロックが落下・スポーンを止める。
5. **回帰移行**: 現行 `clearFullLines` のテトリス式挙動を前提にしたテストを洗い出し、列重力＋連鎖仕様に更新。

## 実装フェーズ（段階マージ）

| Phase | 内容 |
|-------|------|
| P1 | `applyColumnGravity`＋`resolveBoard`（純粋関数）＋着地統合で `clearFullLines` 置換。最小の連鎖成立 |
| P2 | 弾／スキル／爆弾への統合＋連鎖スコアリング＋コンボガード |
| P3 | `use-chain-resolver` によるステップ再生＋`isResolving`ロック＋演出／連鎖SE |
| P4 | バランス実測・微調整＋回帰テスト仕上げ |

## スコープ外（YAGNI）

- フィーバー／オーバードライブモード（別案 B）。連鎖演出で山場が十分なら不要。
- 土壇場デンジャー・クリア（別案 C）。
- `domain/` 層（死蔵構造）の整理・統合。
- 難易度パラメータの先回りチューニング（実測後に別途）。
