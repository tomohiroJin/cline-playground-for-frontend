# IPNE ビジュアル・モーション ブラッシュアップ 設計ドキュメント

- 作成日: 2026-07-02
- 対象: `src/features/ipne/`
- 方式: 3フェーズ分割・各フェーズ独立 PR（DSI ビジュアル刷新 #147〜#149 と同運用）

## 背景と目的

IPNE は PR #126 で戦士・盗賊スプライトの v3 刷新（個別デザイン化・自動補正・全身斬撃モーション）を完了したが、
現状調査で以下の伸びしろが確認された。

1. **移動もカメラもタイル単位スナップで位置補間なし** — 歩行バウンス（bob/squash）で緩和しているが実体は瞬間移動。動きの硬さの最大要因
2. **アニメ枚数不足** — 攻撃2枚・待機2枚。被弾はプレイヤー＝idle 流用、敵＝単発フレーム差し替えのみ
3. **ヒットストップ未実装** — シェイク・フラッシュはあるが打撃の「重み」を出す時間凍結がない
4. **環境演出が皆無** — ステージ差はタイルのパレット替えのみ。光源・霧・装飾なし
5. **グリッド線が常時表示**でドット絵の質感を弱めている
6. ソースコメントの陳腐化（敵「2フレーム/16×16」表記が実データ 4フレーム/32〜56px と不一致）

本ブラッシュアップは「動きの滑らかさ・打撃感・生命感・ワールドの質感」を全面的に底上げする。

## 設計原則（全フェーズ共通）

- **ドメイン層には触れない**: タイル座標・移動ロジック・戦闘ロジックは既存のまま。変更はすべて
  `presentation/` 層（描画・エフェクト・スプライト）で完結させる
- **純粋関数は TDD**: 補間・シェイク・タイル選択などは Red→Green→Refactor、カバレッジ 90%+ 目標
- **Canvas 描画はスモークテスト**＋既存テストの維持。各 PR で `npm run ci` 全パス必須
- ドット絵の質感維持: `imageSmoothingEnabled=false`・整数ピクセルスナップを崩さない

## Phase 1: 動きの基盤（滑らかさ＋打撃感）

ブランチ: `feature/ipne-visual-p1`

### 1-1. 描画位置の補間

新規モジュール `presentation/screens/render/visualPosition.ts`。

- エンティティ ID ごとに「前回論理位置 → 現在論理位置・遷移開始時刻」を記録する視覚位置トラッカー
- 描画時に `now` ベースで ease-out 補間（約 120ms。プレイヤー移動クールダウンに整合させる）
- プレイヤー・敵の両方に適用。`drawPlayer` / `drawEnemies` は論理位置の代わりに補間位置を使う
- **2タイル以上の跳躍**（ステージ遷移・テレポート・リセット）は補間せず即スナップ
- 消滅したエンティティのエントリは掃除する（メモリリーク防止）
- 補間計算は純粋関数として実装し TDD 対象

### 1-2. カメラのスムーズ追従

- `viewportService.calculateViewport` の整数タイル計算は既存のまま維持
- 描画時にピクセル単位のカメラオフセットを補間位置ベースで算出し、`toScreenPosition` 相当の変換に加算
- カメラはプレイヤーの補間位置を追うため、移動時に画面全体が滑らかに流れる
- マップ端クランプは既存挙動を踏襲（クランプ後の座標を補間）

### 1-3. ヒットストップ

- `effectManager` に凍結ウィンドウを追加（攻撃ヒット 60〜80ms、ボス撃破 150ms 程度）
- 実装は**描画層の `now` を凍結する方式**: 凍結中はアニメーション・補間・パーティクルの経過時間が止まる
- ゲームループ（`useGameLoop`）・入力処理・ドメイン更新には一切触れない

### 1-4. 方向性シェイク

- 現在の乱数シェイクを「ヒット方向ベクトル優勢の減衰振動」に変更
- 被弾時は攻撃が来た方向へ弾かれる感覚。方向情報がない場合は従来の乱数シェイクにフォールバック
- `getShakeOffset` のシグネチャ拡張（方向ベクトルをオプション引数で受ける）

**実装ノート（spec からの乖離）**:
- 被弾時の「攻撃が来た方向」は、presentation 層に攻撃元の座標・方向情報が存在しないため未実装。方向性シェイクはプレイヤー攻撃がヒットした際の前方キック（damage: 2）にのみ適用し、被弾側は引き続き無方向のシェイクのままとした。攻撃元方向を presentation 層まで配線する対応は将来課題とする。
- 「方向情報がない場合は従来の乱数シェイクにフォールバック」という仕様は、実装時に決定論的な2軸非整合（周波数の異なる）正弦振動へ置き換えた。乱数（`Math.random`）ベースだとユニットテストでの再現・検証が困難であり、ヒットストップの凍結時間との整合を取りやすくするための判断。

## Phase 2: キャラ・敵の生命感

ブランチ: `feature/ipne-visual-p2`

### 2-1. プレイヤー攻撃の4枚化

- 攻撃アニメを 2→4 枚（予備動作 → 振り抜き → フォロースルー → 復帰）に拡張
- 戦士・盗賊 × 4方向。既存の `computeAttackTransform`（変形）と重ねて適用
- 既存の v3 不変条件テスト（32×32／パレット整合）を新フレームにも適用

### 2-2. 被弾専用フレーム

- 戦士・盗賊に被弾専用フレームを追加（現在は idle 流用）— のけぞり＋白点滅と整合するポーズ
- 全敵 7 種にも被弾フレームを追加（ボス系は既存 `_DAMAGE_FRAME` の品質を底上げ）

### 2-3. 敵の予備動作と種別個性

- 攻撃前の「溜め」フレーム（発光・膨張）を追加し、攻撃が読めるようにする
- 種別個性の強化: CHARGE＝突進残像、RANGED＝詠唱光、SPECIMEN＝変異の脈動 など
- 状態→フレーム対応は `drawEnemies` の既存分岐を拡張

### 2-4. コメント是正

- `enemySprites.ts` 等の陳腐化コメント（「2フレーム」「16×16」）を実データに合わせて修正

## Phase 3: ワールドの質感

ブランチ: `feature/ipne-visual-p3`

### 3-1. 床タイルのバリエーション

- 座標ハッシュ（決定論的・純粋関数）で床タイルに 2〜3 種のバリエーションを混ぜ、単調さを解消
- ステージ別装飾ドット: S1 小石／S3 水晶片／S4 闇のルーン／S5 深淵の亀裂 など
- `getStageFloorSprite` の返却をバリエーション対応に拡張（既存パレット差し替え機構は維持）

### 3-2. 環境演出

- プレイヤー中心の光円（ラジアルグラデーション）＋画面端ヴィネット
- ステージ別アンビエントカラーオーバーレイ（S1 暖色／S3 青緑／S4 は光円を狭めて緊張感）
- 描画順: タイル → エンティティ → 光円・ヴィネット（`drawOverlays` 付近に追加）

### 3-3. グリッド線の抑制

- 常時表示の白グリッド線（`rgba(255,255,255,0.1)`）を大幅に淡くするか、床タイルの目地表現に置き換え

## データフロー（Phase 1 の中核）

```
useGameRender (renderTime)
  └─ renderGameFrame(now)
       ├─ effectManager: 凍結ウィンドウ判定 → frozenNow を算出
       ├─ visualPosition: 論理位置(タイル) + frozenNow → 補間位置(px)
       ├─ カメラ: プレイヤー補間位置 → ピクセルオフセット
       ├─ drawWorld / drawEnemies / drawPlayer: 補間位置で描画
       └─ drawOverlays: 方向性シェイクの translate/restore
```

## エラーハンドリング・エッジケース

- 補間トラッカーに未知のエンティティ → 初回はスナップ（補間しない）
- ステージ遷移・ゲームリセット時 → トラッカー全クリア
- ヒットストップの多重発火 → 凍結ウィンドウは延長のみ（重複加算しない）
- 低フレームレート環境 → 補間は経過時間ベースなので追加対応不要

## テスト方針

| 対象 | 方式 |
|------|------|
| visualPosition（補間・スナップ判定・掃除） | TDD・純粋関数ユニットテスト |
| ヒットストップ（凍結ウィンドウ・延長） | TDD・effectManager テスト拡張 |
| 方向性シェイク | TDD・純粋関数ユニットテスト |
| スプライト新フレーム | 不変条件テスト（サイズ・パレット整合）を既存パターンで追加 |
| タイルバリエーション（ハッシュ選択） | TDD・決定論性のユニットテスト |
| 描画パイプライン | 既存スモークテスト維持＋回帰確認 |
| CI | 各 PR で `npm run ci` 全パス。E2E は CI 実行 |

## スコープ外

- ドメイン層・ゲームバランスの変更
- ズーム機能・視点変更（トップダウン維持）
- 外部画像アセットの導入（コードスプライト方式を維持）
- サウンド・BGM の変更
