# 灰燼の城壁 — 盤面の空間パズル化（集中スライス）設計書

- **作成日**: 2026-07-24
- **対象**: `src/features/ashen-rampart/`（プラットフォーム14本目）
- **前提**: P1 ウォーキングスケルトンが main にマージ済み（PR #169）
- **本スペックの位置づけ**: P2 コンテンツ拡充に入る前の「面白さの核」検証スライス

## 1. 背景と課題

P1 は決定的シミュレーション（乱数なし）＋ウェーブ固定という良い土台を持つが、実プレイの体感は「**判断に意味がない**」。ユーザー検証で最も強く指摘された欠落である。

### 根本原因（コードで裏取り済み）

| # | 原因 | 該当箇所 |
|---|------|---------|
| 1 | 弓兵の塔 `range 2.5` が 9×7 マップの経路をほぼ全カバー → どこに置くか問われない | `card-pool.ts` `arrow-tower.tower.range` |
| 2 | 設置スロット12個が**均質**（角・チョーク・砦際の価値差がない） | `stage-map.ts` `PLAINS_MAP.buildSlots` |
| 3 | タワー同士が**位置で相互作用しない**（隣に置く意味がない） | `simulate-wave.ts` タワー処理 |
| 4 | マナ毎ウェーブ満タン＋タワー維持コスト0で盤面が一方的に増える | `run-state.ts` `INITIAL_MANA_MAX` |

「ウェーブ固定だから」ではなく、**制約が噛まない＋選択が相互作用しない**ことが主因。ウェーブ固定は「盤面の答え合わせが運で崩れない」という良い設計思想なので**維持する**。

## 2. 方針（合意済みの4判断）

1. 課題の主軸 = **盤面の空間パズル**（TDの醍醐味を判断の源泉にする）
2. 盤面モデル = **固定経路を豊かに**（決定的sim・ウェーブ固定を壊さない。迷路構築型TD＝動的経路探索は不採用）
3. 初回スコープ = **既存の平原1面で核を実プレイ検証**してから量を足す
4. 隣接シナジーは初回**かがり火1枚のみ**（最小変更で核を確認）

過去の教訓を反映：`falldown-shooter`（テスト全緑≠面白い／盤面性質を変える大改修は早期実プレイ確認必須）、`primal-path-fun-verification`（sim較正の限界→早期実プレイ）。

## 3. 空間パズル化の3機構

### 機構1：射程の希少化（土台）

射程を「経路全体を覆う万能資源」から「局所的な希少資源」に変える。

- 弓兵の塔 `range 2.5 → 1.6`（設置スロットから覆える経路は約3セル窓）
- 火砲台 `range 2 → 1.5`（splash は維持）
- 結果：経路11セルを守るには**要所に3〜4基**が必要 → 「どのセグメントを先に守るか」が問いになる
- ウェーブ固定＝既知なので「W3で重装が長い直線に来る → 先行投資 vs チョークで受ける」の**先読みパズル**が成立

### 機構2：マス価値の差（地形）

`StageMap` に地形メタを追加し、均質なスロット／経路に序列をつける。

- **高台（highground）**：2スロットに +射程 または +火力ボーナス。希少ゆえ争奪が起きる
  - **設計意図**：高台と滞留ゾーンを**あえて完全一致させない**。例：高台の1枚は砦際、もう1枚は角付近。→「チョークに最良塔を置く vs 砦際を固める」の二律背反を作る
- **滞留ゾーン（choke）**：S字の角セル `(4,3)(4,2)(4,1)` を減速（速度×0.6）。ここを覆うタワーは**1体あたりの射撃回数が増える**＝高価値カバー域
  - sim は per-tick 移動を計算済みなので、セル別速度倍率の追加は純粋関数の小改修

### 機構3：隣接シナジー（選択が掛け算になる）

配置が「他の配置に依存」して初めて空間が知恵になる。核となる1枚のみ導入。

- **かがり火（beacon／オーラ塔）**：周囲8マスの味方タワーに +火力（初期案 +25%）。攻撃はせずバフのみ
  - 「カバレッジ優先で散らす vs バフ優先で固める」の緊張が生まれ、**位置取りそのものが意思決定**になる

## 4. 可読性（この機構では必須。juice ではなく前提条件）

空間パズルは**射程が見えないと遊べない**。よって初回スライスに含める。

- 手札のタワーカード選択／盤面ホバーで、そのタワーが**覆う経路セルをハイライト**（射程オーバーレイ）
- **高台・滞留ゾーンを別タイルで明示**（色だけに依存せずアイコン等も併用）
- （任意）どの経路セルも覆っていない「穴」の可視化

## 5. カードプール調整

- **かがり火を追加**（`type: 'tower'` の派生。攻撃せずオーラのみ）
- 初期デッキを見直し、毎ウェーブの手札が「同じ弓兵6枚」でなく**種別の異なる空間的選択**になるよう配分
  - 初期案：弓兵の塔×4・火砲台×1・かがり火×1・業火×2・補給×1・棘罠×1（計10枚）
  - 正確な配分はプレイ検証で較正

## 6. アーキテクチャ適合

Clean Architecture + DDD の依存方向・決定的sim を維持。**全て決定的純粋関数で完結し `RandomFn` を要さない**。

| 層 | 変更 |
|----|------|
| `domain/board/stage-map.ts` | `BuildSlot` に `terrain?: 'highground'`、`StageMap` に減速セル情報を追加 |
| `domain/combat/simulate-wave.ts` | ①セル別速度倍率を移動計算に適用 ②タワー実効値 = 基礎値 × 高台ボーナス × かがり火隣接バフ × 全体戦術倍率（配置固定なので戦闘開始時に一括算出） |
| `domain/cards/card-pool.ts` | かがり火カード追加、初期デッキ配分見直し |
| `presentation/BoardGrid.tsx` | 射程オーバーレイ＋地形タイル描画 |

### タワー実効値の算出

placement は1ウェーブ中不変なので、`simulateWave` 冒頭でタワー配列を構築する際に実効 `damage`/`range` を確定させる：

```
実効damage = round(baseDamage × highgroundBonus × beaconBonus × towerAttackMultiplier)
実効range  = baseRange × highgroundRangeBonus
```

かがり火バフは「各タワーの8近傍にかがり火があれば適用」を全タワーに対して一度だけ評価。

## 7. テスト方針（TDD: Red → Green → Refactor）

決定性保証を核に、以下の決定的ケースを `simulate-wave.test.ts` に追加：

- **高台ボーナス**：同一タワーを通常スロットと高台に置いた撃破tickの差が期待通り
- **滞留減速**：減速セルを通る敵の到達tickが期待通り遅くなる
- **かがり火隣接バフ**：隣接タワーの実効damageが上がり、非隣接は不変
- **境界**：かがり火自身は攻撃しない／8近傍の対角も隣接に含む
- 既存テスト（発射周期・罠・splash・漏れ判定）は不変で通ること

`stage-map` / `card-pool` の追加定義にも純粋関数テストを付す。presentation は射程オーバーレイの表示ロジックを補完的にテスト。

## 9. スコープ外（YAGNI・後続フェーズへ）

- 新マップ／飛行敵（2軸カバレッジ）／分岐／ボス（P2残り）
- Canvas エフェクト層・打撃演出（P3）
- meta 進行・ラン間アンロック
- 隣接シナジー2枚目（氷結×重撃コンボ）
- 経路の動的化（迷路構築型TD、本スペックで不採用）
- マナ経済の抜本改修（維持コスト等）は本スライスでは扱わない

## 10. 次のフェーズへの接続
