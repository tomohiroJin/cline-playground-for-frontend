# Deep Sea Interceptor ビジュアル刷新 設計書

- 作成日: 2026-07-01
- 対象: `src/features/deep-sea-interceptor/`
- 目的: ゲームを「もっと派手に・見やすく・敵を個性的に」する

## 背景と目的

Deep Sea Interceptor は深海を舞台にした縦スクロールシューティング。現状には以下の課題がある。

- **敵が没個性**: 通常敵 4 種（`basic` / `fast` / `shooter` / `tank`）は色違いの同一スプライト（楕円＋赤い目 2 つ）で描かれ、動きも spawn 時ランダムで型と無関係。ボス・ミッドボスには固有 SVG があるのに、道中の雑魚は見分けがつかない。
- **派手さの伸びしろ**: 描画は DOM + インライン SVG。敵弾は単なる赤い円、パーティクルは単色の四角/丸で、グロー（発光）や弾トレイルが未使用。
- **見やすさの担保が必要**: 派手にするほど「敵弾がエフェクトに埋もれて避けにくい」問題が起きやすい。

本設計では、この 3 つ（派手・見やすい・個性的）を**バランスよく**底上げする。ビジュアルテイストは**ネオン生物発光系**（暗い深海を基調に、鮮やかな発光色で光らせる）とする。

## 設計方針（全体アーキテクチャ）

- 既存の **DOM + インライン SVG 描画**と **Strategy 構造**を壊さない。Canvas 化などの大改修はしない（YAGNI）。
- その上に「**ネオン発光レイヤー**」と「**型別の見た目・挙動メタデータ**」を足す。
- **見た目メタデータの一元化**が設計のキモ。型ごとの「シルエット種別・基調色・危険度・動きパターン」を 1 つのテーブルに集約し、描画・挙動・危険度表示が同じ真実を参照する（DRY / 単一責任）。
- ネオンは **SVG フィルタ＋CSS の再利用ヘルパー**で表現し、個々のスプライトに散らさない。
- **`prefers-reduced-motion` 対応**で過剰アニメをオプトアウト可能にする（プロジェクトの UI/UX ルール準拠）。

### 現状把握（重要な事実）

- 描画は Canvas ではなく DOM + インライン SVG（README の記述は古い）。各エンティティが絶対配置の div/svg。
- 通常敵 4 種は `EnemySprite` のフォールバック汎用 SVG（楕円＋赤い目）で描画。ボス/ミッドボスのみ固有 SVG。
- 敵の動き: `getMovementStrategy`（`game-logic.ts:244`）が通常敵に `[straight, sine, drift]` を **`movementPattern`（`entities.ts:97` で spawn 時に 0〜2 のランダム）** で割り当て。→ 動きは型と無関係。
- 自弾は既にシアン/白（寒色, `BulletSprite`）、敵弾は赤（暖色, `DeepSeaInterceptorGame.tsx` にインライン）。色役割の土台は既にある。
- `EnemyConfig`（HP・速度・点数）と `EnemyType` は数値バランスの基準。本設計では**変更しない**。

## フェーズ構成

各フェーズ = 独立レビュー可能な 1 PR。P1 の基盤に P2/P3 が依存するため、前フェーズのマージ後に次へ着手する。

| Phase | テーマ | 主対象 |
|---|---|---|
| P1 | 見やすさ基盤＋ネオン描画基盤＋配色整理 | `constants.ts`(ColorPalette), `BulletSprite`, `PlayerSprite`, 敵弾描画, 発光ヘルパー |
| P2 | 通常敵 4 種の個性化（見た目＋挙動） | `EnemySprite`, `game-logic`(getMovementStrategy), `entities`, 新規 `EnemyVisual` テーブル |
| P3 | 派手さ演出強化 | パーティクル/撃破/被弾/トレイル, styles |

## Phase 1: 見やすさ基盤＋ネオン描画基盤＋配色整理

**狙い**: 以降で派手にしても弾・敵が埋もれない足場を先に作る。見やすさ 4 項目のうち「敵弾の視認性」「自機の当たり判定」「自弾と敵の分離」をここで担保。

### 1-1. 配色の役割整理（自弾と敵の分離）

`ColorPalette` を「役割」で再整理し、寒色＝味方 / 暖色＝敵の脅威 を徹底する。

| 役割 | 色方針 | 現状との差分 |
|---|---|---|
| 自弾 | シアン/白（維持） | 変更なし |
| 敵弾 | 白コア＋暖色グローリングの二層 | 単色赤円 → コア＋輪郭で常時視認可 |
| 敵本体 | 型ごとに異なる発光色（P2 で確定） | ここでは基調パレットのみ整備 |
| アイテム | 現状維持（既にグロー有り） | 変更なし |

### 1-2. ネオン発光ヘルパー（再利用可能な描画基盤）

- 再利用可能な SVG グローフィルタ（`feGaussianBlur` ベース）と、CSS `drop-shadow` ユーティリティを 1 箇所に定義。
- 各スプライトは「発光色」を渡すだけで統一トーンのグローが乗る。強度は控えめ既定値＋強調用の 2 段階。
- `prefers-reduced-motion` で発光アニメ（パルス等）を停止できるようにする。

### 1-3. 敵弾リデザイン（視認性の主役）

- 現状 `DeepSeaInterceptorGame.tsx` にインライン記述の敵弾（赤い円）を、専用コンポーネント `EnemyBulletSprite` に切り出す。
- 構造: 白コア（中心の高輝度点）＋暖色グローリング＋わずかな縁取り。背景・ネオン発光に埋もれない「必ず見える弾」を最優先。サイズ感も微調整。
- コンポーネント分離により P3 の弾トレイル追加が差し込みやすくなる。

### 1-4. 自機の当たり判定の可視化

- `PlayerSprite` に、実当たり判定（`hitboxRatio 0.4`）に対応する中心の小さな高輝度コア点を常時表示。
- 「機体の見た目は大きいが、当たるのは中心の点だけ」を直感的に伝え、回避を公平化する。

### Phase 1 スコープ外

- 敵本体のシルエット（P2）／撃破・被弾・トレイル演出（P3）。

### Phase 1 テスト方針

- 主に描画（presentational）。純粋ロジックは配色ヘルパーの色決定関数程度 → 該当あれば単体テスト。
- コンポーネントは `describe` 単位でスモークレンダリング（既存テストパターン踏襲）。

## Phase 2: 通常敵 4 種の個性化（見た目＋挙動）

**狙い**: 没個性の核心を解消。「敵の種別・危険度が一目で分かる」＝見やすさにも直結。

### 2-1. 見た目メタデータの一元化（設計のキモ）

新規テーブル `EnemyVisual: Record<EnemyType, {...}>` を導入し、型ごとに以下を定義する。

- `silhouette`: シルエット種別（描画分岐のキー）
- `glowColor`: ネオン基調色（P1 のグローヘルパーに渡す）
- `danger`: 危険度（`low` / `mid` / `high`）— 色・縁取り・アイコンで表現
- `movement`: 動きパターンのキー

描画（`EnemySprite`）・挙動（`getMovementStrategy`）・危険度表示が同一テーブルを参照する（DRY）。

### 2-2. 通常敵 4 種のキャラクター設計

| 型 | シルエット | 動き（挙動差別化） | 危険度／識別 |
|---|---|---|---|
| basic | クラゲ（ドーム＋触手、ゆらぎ発光） | ゆっくり漂う（drift＋パルス） | low・シアン系 |
| fast | 矢尻型の深海ダーツ（尖った前傾シルエット） | 素早い直進＋軽い蛇行、残像トレイル | mid・青紫、鋭い形で「速い」を伝達 |
| shooter | 提灯アンコウ風（発光ルアー＝目印） | サイン波でホバリングし自機を狙う | high・マゼンタ、発射前にルアーが光る予兆（テレグラフ） |
| tank | 甲殻ユニット（装甲板の重厚シルエット） | 重々しいドリフト（低速） | mid〜high・琥珀、被弾で装甲フラッシュ |

### 2-3. 挙動差別化の実装方針

- 現状 `movementPattern` は spawn 時ランダム（`entities.ts:97`）。これを型ベースに変更する。
  - `getMovementStrategy` を「型 → 動き」対応に拡張、または `EnemyVisual.movement` を参照。
  - 既存の `MovementStrategies`（straight/sine/drift）を活かしつつ、必要なら型別の軽い変種を追加。
- shooter のテレグラフ: 発射クールダウンの残り時間を使い、発射直前にルアーを発光させる予兆表現（危険度の伝達＝見やすさ）。ロジックは既存 `EnemyAI.shouldShoot` / `fireRate` の情報から導出し、新規状態は最小限にする。

### 2-4. 後方互換・影響範囲

- `EnemyType` 型・`EnemyConfig` の数値バランス（HP・速度・点数）は変更しない。見た目と動きの割り当てのみ変更。
- ボス・ミッドボスの既存 SVG はそのまま（P2 の対象は通常 4 種）。

### Phase 2 テスト方針（TDD 対象あり）

- 純粋ロジックを先にテスト: 型 → 動きパターンの対応（`getMovementStrategy` の型別分岐）、危険度・シルエットのメタデータ参照。
- ランダム依存を型決定に置き換えることでテストが安定化する（従来は乱数依存でテストしづらかった）。
- 既存の `movement.test.ts` / `enemy-ai.test.ts` の回帰を確認。

## Phase 3: 派手さ演出強化

**狙い**: 爽快感・迫力の底上げ。P1 で視認性の足場を作った上で、埋もれない範囲で振り切る。

### 3-1. 撃破エフェクト強化（爽快感の主役）

- 現状の単色パーティクル（四角/丸）を、発光バースト＋衝撃波リングに強化。
- 通常敵撃破: 小さめの発光飛散。tank/shooter など高危険度は一回り大きく。
- ミッドボス／ボス撃破: 既存の画面シェイク・フラッシュに、拡大する衝撃波リングを追加（既存 `screenShake` / `screenFlash` に乗せる）。

### 3-2. 被弾フラッシュ（ヒットの手応え）

- 敵に自弾がヒットした瞬間、敵本体を一瞬白く発光させる（tank の装甲フラッシュと統一表現）。
- 「硬い敵に効いている」感触を伝え、tank の HP 差を体感的に理解しやすくする（見やすさにも寄与）。

### 3-3. 弾トレイル（スピード感）

- チャージショットと fast 敵の残像に軽いトレイルを付与（P1 で分離した弾コンポーネントに差し込み）。
- 通常自弾は数が多いので、トレイルは控えめ or 付けない（視界のノイズ回避＝見やすさ優先）。

### 3-4. 環境・発光の底上げ

- P1 のネオングローを、ステージ発光ギミック（Stage 4 生物発光帯など）と馴染ませて全体の一体感を出す。
- 過剰にならないよう、既存ギミック演出の範囲内で調整。

### 3-5. 非機能

- パルス・トレイル・衝撃波などの装飾アニメを、reduced-motion 設定時は静的表現にフォールバック（UI/UX ルール準拠）。

### Phase 3 スコープの線引き（YAGNI）

- 新しいゲームメカニクス（新敵・新武器・スコア変更）は入れない。既存挙動の視覚的強化に限定。
- 音の刷新は今回のスコープ外（`audio.ts` は触らない）。

### Phase 3 テスト方針

- 演出は presentational 中心。純粋ロジック（撃破エフェクト生成のパーティクル数決定など）に切り出せる部分があれば単体テスト。
- 既存テストの回帰確認（`game-logic.test.ts` 等）。

## テスト戦略（横断・プロジェクト TDD 規約準拠）

- 純粋ロジック優先で TDD（Red→Green→Refactor）。
  - P2: 型 → 動きパターン対応、危険度/シルエットのメタデータ参照、shooter テレグラフ判定。
  - P3: 撃破エフェクトのパーティクル数決定など、切り出せた純粋関数。
- presentational はスモークレンダリング: 各スプライトが型ごとに例外なく描画されることを `describe` 単位で確認。
- 回帰: `movement.test.ts` / `enemy-ai.test.ts` / `game-logic.test.ts` を各 Phase で確認。
- カバレッジ目標: 新規ビジネスロジック 90%+、新規コード全体 80%+。

## 非機能・品質

- `prefers-reduced-motion` 対応（全 Phase 共通）。
- パフォーマンス: エンティティ数上限（`Config.limits`）は変更しない。グロー/トレイルは DOM 数を増やしすぎない範囲で。
- 型安全: `any` 禁止。`EnemyType` を網羅する `Record` で型漏れを防止。
- CI: 各 Phase で `npm run ci`（lint:ci → typecheck → test → build）を通す。E2E はローカル実行不可のため CI/PR 側で確認。

## 完了基準（Definition of Done）

各 Phase の PR がすべて満たす。

- [ ] 対象スコープの実装完了（P1 基盤 / P2 4 種個性化 / P3 演出）
- [ ] 新規純粋ロジックにテストがあり通過
- [ ] `npm run ci` グリーン
- [ ] `prefers-reduced-motion` で装飾が無効化される
- [ ] 見やすさ 4 項目（敵弾視認 / 当たり判定 / 種別・危険度 / 自弾と敵の分離）が維持されている
