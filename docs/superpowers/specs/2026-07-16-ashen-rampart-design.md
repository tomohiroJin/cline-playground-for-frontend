# 灰燼の城壁 — ASHEN RAMPART 新規ゲーム設計書

- 日付: 2026-07-16
- 対象: `src/features/ashen-rampart/`（新規 Feature モジュール、プラットフォーム14本目）
- 位置づけ: 未開拓ジャンルへの挑戦。**デッキ構築 × タワーディフェンスのハイブリッド・ローグライト**
- ジャンル空白地帯（リズム/カード/TD/物理パズル）のうち、カード＋TDを一本に統合

## コンセプト

滅びかけた王国の最後の城壁を、押し寄せる魔物の軍勢から守るデッキ構築タワーディフェンス。
ダークファンタジーの世界観。手札のカードでタワー・罠・スペルを繰り出し、
ウェーブを凌ぎながらデッキを育て、3ステージ先のボスウェーブ突破を目指す。

企画時に決定した4本柱:

1. **ハイブリッド型**: カードで防衛手段を配置する TD（純TD・純カードバトルではない）
2. **フェーズ分離型**: 準備フェーズでじっくり考え、戦闘フェーズは自動進行を観戦
3. **中型ラン**: 1ラン 15〜20分、ステージ3〜4面構成のローグライト
4. **経路型TD（案A）**: 敵は固定経路を進軍、道沿いの設置マスにタワーを建てる古典構造

## 1ランの構造（15〜20分）

```
ラン開始
 └ 初期デッキ10枚（基本タワー×6、基本スペル×3、罠×1）
 └ ステージ1（平原）   … ウェーブ×3 + 報酬
 └ ステージ2（森/廃村）… 分岐選択 → ウェーブ×4 + 報酬
 └ ステージ3（城門）   … ウェーブ×4 + ボスウェーブ
 └ リザルト（スコア・記録保存）
```

- ステージ間に**分岐選択**: 「安全だが報酬が渋い道」vs「危険だが高レア率の道」
- ステージごとにマップ（経路・設置マス）が変わり、デッキの答え合わせが3回起きる
- 砦のライフは**ステージをまたいで持ち越す**。序盤の漏らしが終盤に効く

## 1ステージ内のループ（フェーズ分離）

```
[準備フェーズ] 手札5枚。マナを消費してタワー/罠/スペル(事前型)を使用
      ↓ 「ウェーブ開始」ボタン
[戦闘フェーズ] 敵が経路を進軍、タワーが自動攻撃。プレイヤーは観戦
      ↓ 全滅させたら
[報酬フェーズ] カード3枚から1枚選んでデッキに追加（スキップ可）
      ↓ 手札を引き直して次の準備フェーズへ
```

- **タワーは設置したら盤面に残る**（同一ステージ内で永続）。カードは使い切りの「投資」
- 使わなかった手札は持ち越さず毎ウェーブ引き直し → 「今引いた手でどう凌ぐか」の即興性
- ステージが変わると盤面はリセット（新マップ）。デッキだけが持ち越される成長資産

## カード・デッキシステム

### カード4タイプ

| タイプ | 効果 | 例 |
|--------|------|-----|
| タワー | 設置マスに建てる。盤面に永続 | 弓兵の塔（単体速射）、火砲台（範囲・遅延）、聖光の尖塔（貫通） |
| 罠 | **経路マス**に置く設置型。踏んだ敵に発動し消費 | 落とし穴（即死・単体）、棘罠（通過ダメージ×3回） |
| スペル | 使い切りの即時/次ウェーブ効果 | 業火（範囲ダメージ）、時泥（敵全体減速）、補給（マナ+2） |
| 戦術 | 永続的なルール変更 | 鍛冶の加護（全タワー攻撃+15%）、血の契約（ライフ-3、マナ上限+1） |

- 初期デッキ10枚 → ラン終了時 18〜22枚を想定
- レアリティ: コモン/レア/エピック。危険な分岐ほど高レア率

### マナ経済

- 準備フェーズ開始時に全回復。基本上限 **4**
- タワー 2〜4 / スペル 1〜3 / 戦術 2〜3 マナ
- 余りマナは持ち越さない（毎フェーズ使い切るテンポ）
- マナ上限の拡張が成長軸の一つ

### ビルドの3軸（最初からカードプールで成立させる）

1. **要塞型**: タワー多め。序盤に盤面資産を作り後半は観戦
2. **魔導型**: スペル多め。盤面は薄いが危機をスペルで捌く
3. **圧縮型**: 報酬スキップ＋除去でデッキを薄くしキーカードを毎回引く

PRIMAL PATH の教訓: ビルド多様性は後付けで増やすと調整が苦しい。初期カードプール設計時に3軸の成立を確認する。

## 敵とウェーブ

基本5種で個性を出す（DSI 敵個性化の知見を流用）:

- **雑兵**（数で押す）/ **俊足**（速いがもろい）/ **重装**（遅いが硬い）/ **飛行**（対空タワーのみ有効）/ **ボス**（ステージ3のみ、特殊ギミック）

ウェーブ構成は**ステージごとに事前定義**（乱数はカード報酬・分岐側に寄せる）。
盤面の答え合わせが運で崩れないようにするため。

## アーキテクチャ

### Feature モジュール構成

```
src/features/ashen-rampart/
├── domain/
│   ├── cards/          # カード定義・デッキ・手札・マナ（値オブジェクト）
│   ├── board/          # マップ（経路・設置マス）・タワー盤面
│   ├── combat/         # ウェーブ進行シミュレーション（純粋関数）
│   ├── run/            # ラン状態・ステージ分岐・報酬生成
│   └── scoring/        # スコア計算
├── application/
│   ├── use-cases/      # start-run, play-card, start-wave, choose-reward, …
│   └── ports/          # rng-port（乱数注入）, storage-port（記録保存）
├── infrastructure/     # seeded RNG 実装、localStorage アダプタ
├── presentation/       # React コンポーネント + Jotai アトム
└── hooks/
```

### 設計の要点

1. **戦闘は決定的シミュレーション**: `domain/combat/` の純粋関数で
   `(盤面, ウェーブ定義, シード) → tick列` を返す。UI は結果の再生のみ。
   ヘッドレスで数百ラン回して破綻検出が可能（迷宮の残響の sim 基盤の思想）。
   ただし sim は破綻検出用と割り切り、**面白さは早期実プレイで確認**（PRIMAL PATH の教訓）。
2. **乱数は全て ports 経由の seeded RNG**: シャッフル・報酬・分岐すべて注入。テスト再現可能。
3. **カードはデータ駆動**: `CardDefinition` のリテラル配列＋少数の効果プリミティブ
   （damage / slow / buff / gainMana …）の組み合わせ。カード追加＝データ追加。

## ビジュアル方針（こだわりポイント）

**DOM/SVG ＋ Canvas エフェクト層の3層構成**。派手さと保守性を両立する。

| 層 | 技術 | 内容 |
|----|------|------|
| 背景層 | CSS / 静的描画 | ダークファンタジーの雰囲気（黄昏の空、遠景の廃墟、霧のグラデーション） |
| 盤面層 | DOM/SVG | 経路・設置マス・タワー・敵。敵移動は tick データを CSS transform で補間再生 |
| エフェクト層 | Canvas (rAF) | 弾道・爆発・パーティクル・グロー・画面フラッシュ。DSI 演出強化の知見を流用 |

- エフェクト層は**描画専用でゲームロジックを一切持たない**（air-hockey の「Canvas はテスト困難」
  という地雷を、ロジック分離で回避する）
- カード UI は手札のホバー浮き上がり・ドラッグ配置・使用時の燃え尽きアニメーション等、
  「カードを切る気持ちよさ」を重視
- ライト/エフェクトの動的生成数は上限を設ける（labyrinth-of-shadows の
  「描画リソース数変動によるカクつき」の教訓。パーティクルはプール方式）

## テスト戦略（TDD）

| 層 | テスト内容 | カバレッジ目標 |
|----|-----------|--------------|
| domain/combat | tick 進行・攻撃・移動・撃破判定（最重要） | 90%+ |
| domain/cards | マナ検証・シャッフル・ドロー・デッキ追加 | 90%+ |
| domain/run | ステージ遷移・分岐・報酬・ライフ持ち越し | 90%+ |
| application | ユースケース単位 | 80%+ |
| presentation | フェーズ遷移の UI フロー（RTL） | 主要フロー |
| E2E | 1ラン完走スモーク（Playwright、シード固定） | 1本 |

- エフェクト層（Canvas）はユニットテスト対象外。ロジックを持たせないことで担保

## 実装フェーズ分割（想定）

大きめの新規ゲームなので、PR を分割して段階的にマージする:

1. **P1 基盤**: domain 全層＋ユースケース＋最小 UI（プレースホルダ描画で1ラン遊べる）
2. **P2 コンテンツ**: カードプール拡充（3ビルド軸成立）・ステージ3面・ボス・分岐
3. **P3 ビジュアル**: Canvas エフェクト層・カードアニメーション・背景アート
4. **P4 較正**: 実プレイ＋ヘッドレス sim によるバランス調整・スコア/記録・仕上げ

各フェーズは個別の実装プラン（`docs/superpowers/plans/`）を作成してから着手する。

## スコープ外（今回やらないこと）

- メタ進行（ラン間の永続アンロック）— 初回リリース後の拡張候補
- 難易度選択 — まず標準難度1本を較正してから
- リアルタイムでのカード使用（戦闘フェーズ中の介入）— フェーズ分離を崩すため不採用
- 2P・対戦要素
- 効果音/BGM の作り込み — Tone.js での最小限の SE のみ P3 で検討
