# Picture Puzzle ブラッシュアップ設計書 ──「あなただけの美術館」

- 作成日: 2026-07-07
- 対象: `src/` 直下の picture-puzzle（このプラットフォームの基盤ゲーム）
- 目的: UI/UX の質感・達成感・快適性・面白さを、一貫したコンセプトのもとで底上げする

## 1. 背景と課題

picture-puzzle は 6テーマ・9段階難易度・スコア＆ランク・クリア演出・動画再生・履歴・シェアと機能は充実しているが、**見た目と見せ方が弱い**。現状のビジュアルは「アニメーショングラデーション背景＋グラスモーフィズム＋シアンのグロー」で、これは AI/テンプレ系のデフォルトに近く、**このゲーム固有の個性が乗っていない**。一方でコンテンツ側は浮世絵・世界の風景・ノスタルジー等、**画像そのものが主役の強い素材**を持つ。この素材を引き立てる「額装＝ギャラリー」の器へ作り替えることが課題。

### 優先度（ユーザー合意）

1. UI/UX の質感（見た目・見せ方）── 最優先
2. 達成感・継続動機
3. アクセシビリティ・快適性
4. 面白さ（新モード）

## 2. コンセプト（体験を束ねる背骨）

> **プレイヤーは学芸員（キュレーター）。パズルを解くたびに、その絵を自分の美術館に「収蔵」していく。**

既存要素を美術館の言葉で再定義する（新規データは原則増やさず、既存を読み替える）:

| 既存要素 | 美術館の言葉 | 継続動機への効果 |
|---|---|---|
| テーマ | 展示室 / コレクション | 展示室を埋めたくなる |
| 画像をクリア | 作品を収蔵 | 収集欲・コンプリート欲 |
| ランク（★） | 鑑定評価 | ★★★での再収蔵＝リプレイ動機 |
| テーマ解除 | 新館オープン | 次の目標の可視化 |
| クリア履歴 / ベストスコア | 収蔵目録 | 積み上げの実感 |

## 3. トーン & デザイントークン（A案：ギャラリー/美術館）

- 基調: 温かいオフホワイト（美術館の壁）。UI を黒子にして絵を主役に、広い余白・静謐
- パレット（暫定・フェーズ3で WCAG AA 準拠に最終調整）:
  - クリーム `#f4f1ea` / インク（文字）`#2b2620` / サブ文字 `#7a7062`
  - ゴールド（アクセント）`#a8894e` / セージ（進捗）`#8a9a7b` / マット（額の台紙）`#fffdf9`
- タイポ: 見出し＝セリフ体（Georgia／和は明朝系）、ラベル＝Inter の小型オールキャップス
- 額装: 白マット＋細フレーム＋控えめな影で画像を「額装」する共通コンポーネント
- グロー装飾（シアンの脈動等）は廃止し、光は「正解ピースの定位置光（ゴールド）」に集約

## 4. スコープと安全境界（★重要・厳守）

picture-puzzle は基盤ゲームゆえ、共有フォルダに専用コードが同居している。**変更を picture-puzzle 配下に閉じ込め、他12ゲームへ波及させないこと**を最優先の制約とする。

### 触ってよい（picture-puzzle 専用）

- `src/pages/PuzzlePage.tsx` / `PuzzlePage.styles.ts`
- `src/components/{TitleScreen,PuzzleSections,SetupSection,GameSection}.tsx`
- `src/components/molecules/*`（ThemeSelector, DifficultySelector, ResultScreen, ClearHistoryList, ShareButton 等）
- `src/components/organisms/PuzzleBoard*` / `PuzzlePiece*` / `VideoOverlay` / `ConfettiOverlay`
- `src/presentation/*`（hooks / store）
- `src/domain/puzzle/*` / `src/application/use-cases/*`（フェーズ4のみ最小限）
- `src/data/themes.ts` / puzzle 系 `src/utils/*`（puzzle-utils, storage-utils, score-utils）
- `src/store/atoms.ts`（外部参照ゼロを確認済み）

### 触らない（全ゲーム共有・波及リスク）

- `src/styles/GlobalStyle.ts` / `src/styles/tokens/*`（`--bg-gradient` 等のグローバル CSS 変数）
- `src/App.tsx`（全体レイアウト・ヘッダ）
- `src/pages/GameListPage.tsx`
- `src/features/*`（他ゲーム）

### 実現方法（局所テーマ注入）

ギャラリーのトーンは、グローバルの `GlobalStyle.ts` を触らず、**`PuzzlePageContainer` 上で CSS カスタムプロパティを再定義**して配下だけに適用する（カスケードで他ゲームの `:root` は無傷）。背景も `PuzzlePageContainer` 自身に美術館色を敷き、グローバル背景を覆う。

```
PuzzlePageContainer {
  --bg-gradient / --accent-color / --glass-bg / --text-primary ... を局所上書き
  background: <museum cream>;
}
```

### 検証（回帰チェック）

各フェーズ完了時、picture-puzzle 以外のゲーム1本以上をブラウザで開き、背景・配色・タイポが従来どおりであることを確認する。

## 5. フェーズ構成

各フェーズを独立 PR とする（本リポジトリの運用に準拠）。

### フェーズ1: ギャラリーの器（見た目）

全4画面に A案トーンを一貫適用する。**情報量は増やさず、額装と言葉で意味を与え直す**（既存のスコア/タイム/手数/履歴はそのまま）。

- **共通**: `PuzzlePageContainer` への局所テーマ注入、額装コンポーネント（マット＋フレーム）、タイポ刷新
- **① タイトル**: 余白＋額装ヒーロー、明朝＋Inter、「入館する」ボタン。「収蔵目録（n/15点）」への導線（中身はフェーズ2で実装。ここでは器＝空き枠のみ用意）
- **② セットアップ**: テーマ＝展示室カード化。収蔵率バー・アンロック条件の表示枠を用意（数値はフェーズ2で接続）。難易度は「額のサイズ」として整理
- **③ プレイ中**: 盤面をマット＋額縁で額装。HUD（時間・手数）を解説プレート風に。正解ピースは緑フラッシュ→ゴールドの定位置光へ
- **④ リザルト**: クリア＝「収蔵しました」の儀式。★＝鑑定評価表示。動画演出は「作品が動き出す展示」として活かす

受け入れ基準: 4画面がギャラリートーンで一貫。既存の全テストがグリーン。他ゲームに視覚的変化なし。

### フェーズ2: 収蔵コレクション（達成感）

既存の「クリア履歴／ベストスコア」を、美術館メタファーの**収蔵目録**へ再設計。新規永続データは足さず、既存の `PuzzleRecord`・ベストスコア・累計クリアを読み替える。

- **収蔵目録ビュー**（`ClearHistoryList` を発展）: 展示室ごとの壁に作品を並べる。収蔵済み＝画像＋鑑定評価（★）、未収蔵＝空フレーム、未開館テーマ＝「あと○点で開館」
- **展示室ごとの収蔵率**をセットアップ／目録に表示
- **最終目標**: 「名誉学芸員（全15点★★★収蔵）」を最上位ゴールとして可視化
- フェーズ1で用意した「収蔵率バー」「収蔵目録導線」の空き枠に実データを接続

受け入れ基準: 既存の履歴・ベストスコアが目録として正しく表示。★★★再収蔵で評価が更新。データ移行（既存 localStorage）を壊さない。

### フェーズ3: 快適性（アクセシビリティ・操作感）

明るい配色にするからこそコントラストと色依存を見直す。

- **色覚・コントラスト**: パレットを WCAG AA 準拠に最終調整。★ランクは色＋文字ラベル併記、正解フィードバックは色のみに頼らず動き＋定位置光で伝達
- **キーボード**: フォーカスリング可視化、操作キーヒントの常設
- **モバイル**: スワイプ閾値調整、タップ領域 44px 以上、小画面での盤面・展示室カードのレスポンシブ最適化
- **ロード体感**: 画像ロード中は額縁スケルトン、次作品の先読み、レイアウトシフト防止
- **モーション配慮**: `prefers-reduced-motion` で紙吹雪・ディゾルブ・背景アニメを抑制

受け入れ基準: 主要導線がキーボードで完結。コントラスト AA。reduced-motion で過剰演出が止まる。

### フェーズ4: 面白さ（新モード）

コンセプトに合う**2モードに限定**。既存の盤面ロジックへの侵襲を最小化する。

- **本日の一枚（デイリー）**: 日替わりで固定の作品＋難易度を出題（日付シード固定）。`shuffle-service` にシード引数を追加するのみでドメイン侵襲は最小。自己ベスト更新の習慣化フック
- **鑑定チャレンジ（タイムアタック / 手数制限）**: 既存作品に達成条件（例: 90秒以内、最適手数+10以内）を付与し、達成で特別な鑑定メダル。既存スコア資産を活用（達成判定の追加のみ）
- **backlog（今回は名前のみ・実装見送り）**: 回転ピース変種、ぼかしミステリー展、企画展ローテーション

受け入れ基準: デイリーは同日同一シードで再現。チャレンジ達成判定が正しくメダル付与。既存の通常プレイに影響なし。

## 6. 設計原則の遵守（プロジェクト規約）

- **TDD**: 各フェーズはテストファースト。新規コード 80%+、ドメインロジック 90%+
- **Clean Architecture / DDD**: 見た目はプレゼンテーション層に閉じる。フェーズ4のシード・達成判定はドメイン／ユースケースへ適切に配置し、UI 都合を持ち込まない
- **DbC**: シード・難易度・達成条件は型とバリデーションで事前条件を表現
- **禁止事項の厳守**: `any` 不使用、`domain/` からの外側参照禁止、`dangerouslySetInnerHTML` 不使用
- **CI**: `npm run ci`（lint:ci → typecheck → test:coverage → build）＋ E2E を各 PR でパス

## 7. リスクと緩和

| リスク | 緩和策 |
|---|---|
| 共有スタイルへの波及で他ゲームが崩れる | 局所テーマ注入に限定、各フェーズで他ゲーム回帰チェック |
| 明るい配色でコントラスト不足 | フェーズ3で AA 準拠を確定、色依存を排除 |
| 既存 localStorage データの破壊 | 目録は読み替えのみ、`migrateClearHistory` 等の既存移行を尊重 |
| フェーズ4のドメイン侵襲 | シード引数追加・達成判定追加に限定、盤面ロジックは不変 |
| E2E フレーキー（過去 Issue #131 等の前例） | 演出は待機ベースで検証、`prefers-reduced-motion` を活用 |
