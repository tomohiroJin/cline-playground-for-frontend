# PRIMAL PATH 面白さ検証レポート

- 日付: 2026-06-21
- 対象: `src/features/primal-path/`（原始進化録）
- ブランチ: `chore/primal-path-fun-verification`
- 関連: [Phase3 設計](../specs/2026-06-20-primal-path-phase3-design.md) / [ビルド多様性設計](../specs/2026-06-18-primal-path-build-variety-design.md)

## 1. 背景と目的

PRIMAL PATH の「ビルド多様性・リプレイ性」ブラッシュアップ（Phase1〜3）は全フェーズが main にマージ完了した。
その直後に提起された問い:

> **「全フェーズ完了して、最初の目的通りの面白さができたかのテストはどうなっている？ 完成して終わり?」**

既存の自動テスト（655+ スイート）は **correctness（メカニズムが正しく発火するか）のみ**を証明しており、
当初の目的＝**「毎ランの手触りが違う／リプレイ性が上がったか」という体験・面白さは未検証**だった。
本レポートはこの宿題に 3 つの手法で答える。

合否を測る設計目標（[build-variety 設計 E 節] より）:
1. 横軸(アーキタイプ)×縦軸(パワーカーブ ⚡即効/🌱晩成/🔗コンボ/🃏ワイルド)×コンボの 3 次元で**毎ランの手触りが違う**か。
2. 各トーテム/コンボが**成立はするが必勝ではない**か。
3. 即効型と晩成型の**パワーカーブ交差点が中盤**に来るか。
4. 上位トーテム解放(2/5/10 クリア)の動機付けが効いているか。

## 2. 検証手法

| # | 手法 | 問い | 実装 |
|---|------|------|------|
| ① | バランス・多様性シミュレーション | 必勝でないか / カーブ交差 / 多様性 | 本番 `gameReducer` 駆動モンテカルロ |
| ② | ブラウザ実機プレイ | UI で本当に成立するか・手触り | Playwright + 実機 |
| ③ | UX レビュー | 認知負荷・発見しやすさ | design-review スキル |

## 3. 検証① バランス・多様性シミュレーション

### 3.1 手法

本番の `gameReducer` をそのまま駆動するモンテカルロ。**検証対象と実行コードが同一**なので結果は本番挙動に忠実。

- `Math.random` をシード付き Mulberry32 に一時差し替えて完全決定論化（reducer 内部の `rollE`/`applyEvo` まで固定）。
- `process.env.NODE_ENV='production'` を一時設定し、DbC コントラクトを本番同様に無効化（実プレイヤーが遊ぶのは production ビルド）。
- 戦闘ループは `use-battle.ts` と同じイベント判定（`enemy_killed`→`AFTER_BATTLE` 等）を同期再現。
- 進化選択ポリシー: `greedy-atk`（実効 ATK 最大）/ `balanced`（最弱系統を伸ばす）/ `random`。
- メタ進行（ツリー）を `none`〜`FULL_TREE` で可変。各条件 120〜150 シード。

実装: `src/features/primal-path/__sim__/run-simulator.ts`

### 3.2 結果

**メタ進行が体験に与える影響（greedy-atk, トーテム平均勝率）**

| 解放段階 | di1(氷河期) | di2(大災厄) | di3(神話世界) |
|---------|:---:|:---:|:---:|
| 無強化 | 7% | 0% | 0% |
| T1 | 16% | 0% | 0% |
| T1-2 | 33% | 0% | 0% |
| T1-3 | 85% | 3% | 0% |
| T1-4 | 100% | 65% | 3% |
| フル強化 | 100% | 100% | 100% |

**フル強化でのパワーカーブ（di0, bc 開始時 平均実効 ATK）**

| トーテム | bc0(序盤) | bc1(中盤) | bc2(終盤) |
|---------|:---:|:---:|:---:|
| 🩸血(即効) | 115 | 219 | 361 |
| 🌰種火(晩成) | 85 | 183 | 322 |

### 3.3 所見

- **A. メタ進行が勝敗を二値支配**: 無強化はほぼ全敗、フル強化は全難易度(0〜3)で**一律 100% 必勝**。→ 設計目標②「必勝でない」に反する。
- **B. 横軸(トーテム)が勝敗に出ない**: フル強化では全 6 トーテム 100%、無強化では全滅。個性は最終 ATK(488〜531)の微差にしか出ない。
- **C. パワーカーブが交差しない**: 血(即効)が全区間で種火(晩成)を上回り続け、**交差しない**。→ 設計目標③が未達。晩成型が終盤も即効型に追いつけない。
- **D. ラン長が完全固定**: 平均進化 15.0 / 到達 bc 3.0 が全条件で不変。ランごとの構造的伸縮がない。
- **際どい帯（勝率 5 割の歯ごたえ）はメタ進行の特定 1 段階にしか存在しない**。di1=T1-2〜T1-3、di2=T1-4 のみ。**di3 は中間帯が消失**（T1-4 で 3%→フルで 100% の崖）。

### 3.4 シミュレーションの限界

`greedy-atk` という機械的戦略基準であり、実プレイヤーが楽しむ**コンボ/シナジー狙いの非 ATK 進化**は再現していない。
手触り・演出・コンボの楽しさは体感領域であり、②③で補完する。

## 4. 検証② ブラウザ実機プレイ

炎の祖（🔗コンボ）で原始難易度を実機プレイ（localStorage に `clears:10` をシードし上位トーテムを全解放）。

**全機能が実機 UI で破綻なく成立・コンソールエラー 0**。確認した要素:

1. **ドラフトキーストーン混入**（原始の咆哮/永久凍結が進化の 4 枚目に 💠 付きで出現）— Phase2 のドラフト混入機能。
2. **節目提示の curve+tag 重み付け**（火傷ビルドに 🔥連鎖の業火 が提示）— Phase3 の提示重み付け。
3. **シナジー発動**（🔥灼熱の魂 / 🏹鋭い爪、相性◎ ×1.3）。
4. **火傷コンボの戦闘ログ**（⚔→55 / 🔥火傷19 / 🩸13 / ❄️環境-3）。
5. **小覚醒 → 大覚醒の段階演出**（🔥炎の目覚め → ⚡炎王の始祖、背景グラデ付き）。

スクショは検証時にプロジェクトルートへ取得（レビュー後クリーンアップ）。

## 5. 検証③ UX レビュー（design-review）

### 良い点
- 60-30-10 配色（紺黒 60% / カード 30% / 金アクセント 10%）が統一。
- アーキタイプタグの色彩心理（即効=赤 / コンボ=青 / 晩成=緑 / ワイルド=白）が適切。
- 選択中の黄枠ハイライト（図と地）が明確。
- ステータスバーの増減色分け（橙 ATK / 緑 HP / 青 DEF）が直感的。
- 大覚醒の背景グラデ演出でクライマックスを差別化。

### 必須修正（Must Fix）
1. **広大なデッドスペース**: 幅 640px の縦長カラムが画面上部に集中し、デスクトップでは下半分以上が黒い空白。第一印象で「モバイル幅の流用・未完成」に見え品質感を損なう。
2. **極小テキスト＋低コントラスト**: シナジータグ・覚醒進捗・数値が WCAG AA(4.5:1) を下回る。

### 推奨改善
- アーキタイプタグが小さく発見しにくい（ビルドの軸を決める最重要情報なのに）。
- 進化カードの情報密度過多（1 カードに 7〜8 種）。
- 節目 KS の演出的な軽さ（重要選択なのに視覚的な重みが乏しい）。

## 6. 統合結論：3 つの壁

> **ビルド多様性の「素材」（横軸トーテム×縦軸カーブ×コンボ）は実装・動作とも成立している。
> しかし、それが「毎ランの手触りの違い」や「歯ごたえ」としてプレイヤーに伝わる度合いは限定的。**

体験を薄める 3 つの壁:
- **(a) メタ進行（ツリー解放度）が勝敗を二値支配** — トーテム/コンボ選択の妙が勝敗に乗らない（①A・B・際どい帯）。
- **(b) パワーカーブ交差が機能せず即効が常に優位** — 晩成型が終盤も追いつけない（①C）。
- **(c) UI の余白・可読性で個性が視覚的に埋もれる** — アーキタイプタグが小さく、選択画面が間延び（③）。

**「完成して終わり」ではなかった。** メカニズムは正しく動くが、設計意図（リプレイ性・毎ランの違い）が体験に結実しきっていない、というのが 3 手法一致の診断である。

## 7. 改善提案（優先度付き・未着手）

| 優先 | 領域 | 内容 |
|:---:|------|------|

| 高 | バランス(b) | 晩成型(種火/群れ)の終盤スケールを強化し、即効型とのパワーカーブ交差を中盤に作る。 |

## 8. 副産物：dev ビルドの HP 負クランプ漏れ

シミュレーションで数千ラン回した際、dev モードの事後条件 `ensureTickResult`（`contracts/tick-postconditions.ts`）が
**HP が負の値**を検出して例外を投げるケースが稀に発生した（終盤の高 ATK 敵による）。
production ビルドではコントラクトが無効化されるため実害は限定的だが、反射タンク等で dev ビルドが稀にクラッシュしうる。
HP を負許容ではなく 0 クランプすべき箇所がある可能性。要追跡。

## 9. 付録：シミュレータの使い方

```bash
# 健全性テスト（CI で常時実行・軽量）
npx jest src/features/primal-path/__sim__/run-simulator.test.ts --no-coverage

# バランスレポート（手動・重い。env ガードで通常 CI ではスキップ）
RUN_BALANCE_SIM=1 SIM_SEEDS=150 SIM_DIFFS=0,1,2,3 \
  npx jest src/features/primal-path/__sim__/balance-report.test.ts --no-coverage --silent=false
```

`simulateRun(config)` の `config`: `{ di, totemId, evoStrategy, seed, tree?, loopCount? }`。
`FULL_TREE` で全ツリー解放状態を再現。同一シードは完全に決定論的。
