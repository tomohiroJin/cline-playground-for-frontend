# 灰燼の城壁（ASHEN RAMPART）

## 概要

カードで砦を築くデッキ構築 × タワーディフェンスのローグライト。
手札からタワー・罠・呪文を配置して魔物のウェーブを凌ぎ、ウェーブクリア報酬でデッキを育てていく。
準備フェーズで盤面を組み立て、戦闘フェーズは決定的シミュレーションの「観戦」として再生される。

> **現状は P1（ウォーキングスケルトン）**。平原ステージ1面・3ウェーブのミニランと
> プレースホルダ UI を実装。3ステージ・分岐・ボス・飛行敵・Canvas 演出は今後のフェーズで追加予定。

## 遊び方

1. **準備フェーズ**: 手札のカードを選び、盤面のセルに配置する（マナを消費）
2. **戦闘フェーズ**: 配置した盤面で敵ウェーブを自動戦闘。結果を観戦する
3. **報酬フェーズ**: ウェーブをクリアすると 3 択の報酬カードから 1 枚をデッキに加える
4. 砦のライフ（初期 10）が 0 になる前に全ウェーブを凌ぎ切れば勝利

- **マナ**: 毎ウェーブ上限（初期 4）まで回復。カードのコストとして消費する
- **手札**: 毎ウェーブ引き直し。デッキ構築が手札の質を決める
- **ライフ**: 敵 1 体の漏れごとに -1。ステージをまたいで持ち越す

## ゲームシステム（P1）

### カードプール（8種）

| カード | 種別 | コスト | 効果 |
|--------|------|:---:|------|
| 弓兵の塔 | タワー | 2 | 単体を速射する基本タワー |
| 火砲台 | タワー | 3 | 着弾点の周囲にもダメージを与える重砲（攻撃間隔は長い） |
| 棘罠 | 罠 | 1 | 経路に仕掛ける棘。3 体まで傷つける |
| 落とし穴 | 罠 | 2 | 最初に踏んだ敵を確実に葬る一発罠 |
| 業火 | 呪文 | 2 | 次のウェーブの敵全員に、出現時 8 ダメージ |
| 時泥 | 呪文 | 2 | 次のウェーブの敵全員の足を 40% 遅くする |
| 補給 | 呪文 | 1 | ただちにマナを 2 得る |
| 鍛冶の加護 | 戦術 | 2 | このステージの間、全タワーの攻撃力 +15% |

- **初期デッキ（10枚）**: 弓兵の塔 ×6、業火 ×2、補給 ×1、棘罠 ×1
- **報酬プール**: 火砲台 / 落とし穴 / 時泥 / 鍛冶の加護 / 弓兵の塔 / 業火

### 敵（平原ステージ・地上3種）

| 敵 | HP | 速度 | 撃破スコア |
|----|:--:|:--:|:--:|
| 雑兵（grunt） | 20 | 0.10 | 10 |
| 俊足（runner） | 12 | 0.18 | 12 |
| 重装（brute） | 60 | 0.06 | 20 |

- **ウェーブ1**: 雑兵の小隊（雑兵 ×6）
- **ウェーブ2**: 雑兵 ×6 ＋ 俊足 ×4 の混成
- **ウェーブ3**: 雑兵 ×8 ＋ 重装 ×2 の大隊

ウェーブ構成は**乱数を使わず事前定義**。「盤面の答え合わせが運で崩れない」ことを保証する設計方針による。

## 盤面の空間パズル（本スライスで追加）

配置そのものを意思決定にするための3機構。すべて決定的シミュレーション内で完結する。

- **射程の希少化**: 弓兵の塔 射程1.6・火砲台1.5。1基で経路全体は覆えず、どのセグメントを守るかを選ぶ
- **地形**:
  - **高台（⛰）**: 設置スロットの一部。置いたタワーの火力 +30%。希少ゆえ争奪が起きる
  - **滞留ゾーン（🌫）**: 経路の一部セルで敵が減速（×0.6）。ここを覆うタワーは手数が増える高価値カバー域
- **かがり火（🔥）**: 攻撃しないオーラ塔。隣接（8近傍）の味方タワー火力を +25%。散らす（カバレッジ）か固める（バフ）かの緊張を生む
- **可読性**: タワーカード選択中にスロットへホバーすると、その射程で覆う経路セルが枠表示される

## 設計・技術詳細

### Clean Architecture + DDD

ゲーム固有のロジックを Feature モジュール内で 4 層に分離している。

```
src/features/ashen-rampart/
├── domain/                      # 純粋なゲームルール（外部依存なし）
│   ├── board/
│   │   ├── board-state.ts       #   盤面状態・配置可否判定（canPlaceTower / canPlaceTrap）
│   │   └── stage-map.ts         #   ステージのマップ（経路・配置可能セル）
│   ├── cards/
│   │   ├── card-definition.ts   #   カード型定義
│   │   ├── card-pool.ts         #   カードプール・初期デッキ・報酬プール
│   │   └── deck.ts              #   デッキ・手札・捨て札の状態遷移
│   ├── combat/
│   │   ├── enemies.ts           #   敵スペック
│   │   ├── waves.ts             #   ウェーブ定義（事前定義・乱数なし）
│   │   └── simulate-wave.ts     #   戦闘シミュレーション（決定的純粋関数）
│   ├── run/
│   │   ├── run-state.ts         #   ラン状態・定数（ライフ・マナ・スコア）
│   │   └── reward.ts            #   報酬選択肢の抽選
│   └── shared/
│       └── random.ts            #   RandomFn = () => number 型
├── application/                 # ユースケース（domain を組み立てる）
│   ├── ports/
│   │   └── random-port.ts       #   乱数のポート抽象
│   └── use-cases/               #   start-run / start-wave / play-card / choose-reward
├── infrastructure/
│   └── random/
│       └── seeded-random.ts     #   シード付き乱数の実装（RandomPort）
└── presentation/                # React UI（再生・オーケストレーション）
    ├── useAshenRampartGame.ts   #   ゲーム状態フック（dispatch）
    ├── AshenRampartGame.tsx     #   メインコンポーネント
    ├── BoardGrid.tsx            #   盤面グリッド
    ├── HandArea.tsx             #   手札エリア
    ├── RewardPanel.tsx          #   報酬選択パネル
    ├── ResultPanel.tsx          #   リザルトパネル
    └── StatusBar.tsx            #   ライフ・マナ・ウェーブ表示

src/pages/AshenRampartPage.tsx   # ページコンポーネント（薄いラッパー）
```

### 設計の要

- **戦闘は決定的純粋関数**: `(盤面, ウェーブ, 修飾子) → tick列` を乱数なしで算出。UI は結果を「再生」するだけ
- **依存方向の守り方**: `domain` は `RandomFn = () => number` のみを受け取り、`RandomPort` は `application/ports` に置く。乱数の実体（`infrastructure/random/seeded-random.ts`）を domain が知らない
- **カード ＝ 投資**: タワーは設置で盤面に永続。手札は毎ウェーブ引き直し、デッキ構築がラン全体の強さを決める
- **StrictMode 対応**: `dispatch` は「1 ハンドラ 1 回」の不変条件を守る純粋 updater として実装（二重実行に耐える）

### テスト

- Jest + React Testing Library。テストは対象と同じディレクトリに `*.test.ts(x)` で配置
- TDD（Red → Green → Refactor）で実装。domain の純粋関数テストを中核に、use-case・presentation を補完
