# スプリントノート（Sprint Note）ユーザーストーリー v2.0

## 凡例

- **優先度**: Must（必須）/ Should（重要）/ Could（あれば良い）/ Won't（今回は対象外）
- **規模**: S（小）/ M（中）/ L（大）/ XL（特大）
- **依存**: そのストーリーが前提とする他ストーリーのID
- **カテゴリ略称**: NARR（ナラティブ）、GOAL（ゴール）、CORE（コアループ）、PLAN（計画）、DEV（開発）、REL（リリース）、REV（レビュー）、RETRO（振り返り）、VOICE（ロールの声）、PARAM（パラメータ）、DIST（歪み）、EVT（イベント）、GROW（成長）、UI（演出）、RPL（リプレイ性）、META（メタ・管理）、ONBD（オンボーディング）

---

## ストーリーマップ概観

```
[プレイ外]                [1プレイ = 1プロジェクト]                              [プレイ外]
                ┌─────────────────────────────────────────────┐
 META-001 ──►   │ NARR-001 ─► NARR-002 ─► GOAL-001           │  ──► NARR-003
 メタ画面       │  PM提示      チーム結成    ゴール選択         │      リザルト
                │                                             │       │
 ONBD-001 ──►   │  ┌── Sprint 1 ────────────────────────┐     │  ──► GROW-001/002/003
 チュートリアル │  │ PLAN─►DEV─►REL─►REV─►RETRO─►EVT-003│     │      成長・アンロック
                │  └────────────────────────────────────┘     │
                │       × 5スプリント（Sprint 3後にGOAL-002） │  ──► UI-004
                └─────────────────────────────────────────────┘      称号付与

[常時稼働] PARAM-001/002, DIST-001/002/003, EVT-001/002, VOICE-001/002, UI-001/002
```

## Epic 1. オンボーディング

### ONBD-001：初回プレイのガイダンスを受ける

- **As a** 初めてプレイするプレイヤー
- **I want** ゲームの基本ルール（集合意思の操作・スプリントの流れ・ゴール選択の意味）を、プレイの流れの中で自然に理解したい
- **So that** 説明書を読まなくても最初の1プレイを楽しめる
- **優先度**: Must
- **規模**: L
- **依存**: NARR-001, NARR-002, GOAL-001, CORE-001
- **受入条件**:
  - 初回プレイ時のみ、各フェーズで簡潔なガイドテキストが表示される（2回目以降はスキップ可能）
  - ガイドは「説明」ではなくキャラクターの台詞やナレーションとして自然にゲーム内に組み込まれる
  - スプリント1の計画フェーズで「なぜ全部は選べないのか」をチームの会話として演出する
  - プレイを止めて教えるのではなく、判断しながら学ぶ形式にする
- **設計メモ**: チュートリアル専用シナリオは作らない。初回プレイ時にPM提示の難度を低めに固定し、ガイドテキストを差し込む方式。ゲームの「正解をなぞらない」思想を壊さないよう、ガイドは手順の案内に留め、判断の誘導はしない。

### ONBD-002：ゲームのコンセプトを伝える

- **As a** 初めてプレイするプレイヤー
- **I want** ゲーム開始前に「このゲームは何を体験するものか」を短い導入テキストで知りたい
- **So that** 「最適化ゲーム」ではなく「判断と結果のゲーム」であることを理解した上でプレイを始められる
- **優先度**: Should
- **規模**: S
- **依存**: なし
- **受入条件**:
  - 初回起動時に1画面で読める導入テキスト（3〜5文程度）が表示される
  - 「正解はない。あなたの判断が、チームの物語になる。」のようなトーンで伝える
  - スキップ可能で、メタ画面からいつでも再読可能

## Epic 2. プロジェクト開始・ナラティブ

### NARR-001：プロジェクト提示を受ける

- **As a** プレイヤー
- **I want** PMからプロジェクトの概要（作るもの・期限・期待値・制約条件）を提示されたい
- **So that** これから挑むプロジェクトの全体像と難度を把握し、心構えができる
- **優先度**: Must
- **規模**: M
- **依存**: なし
- **受入条件**:
  - PMからのテキスト演出でプロジェクト内容が提示される
  - 期限の長さ・期待値の高さ・要件の曖昧さ・ステークホルダーの圧の4軸で前提条件が決定される
  - プレイごとに提示内容が変化する（固定シナリオではない）
  - 4軸の値はプレイヤーに数値として見せず、PMの台詞や表現のニュアンスで伝える（例：「期限は……正直、厳しい」「上はかなり期待しているみたいだ」）
- **設計メモ**: PMは「敵」ではなく「条件を設定する外部の存在」として描写する。理不尽に押しつけるのではなく、PM自身も制約の中にいることが伝わるトーンが望ましい。

### NARR-002：チーム結成を体験する

- **As a** プレイヤー
- **I want** 新人5人（PdM・FE・BE・QA・デザイナー）のチームが結成される場面を見たい
- **So that** これから操作するチームの顔ぶれと役割を認識し、プロジェクトへの期待感を持てる
- **優先度**: Must
- **規模**: M
- **依存**: NARR-001
- **受入条件**:
  - 5人のロールと簡単な人物像がテキストで紹介される
  - 各メンバーの「得意なこと」と「新人ゆえの弱点」が台詞やナレーションで示唆される
  - プレイヤーが個人を直接操作するのではなく、チーム全体の意思決定者であることが自然に伝わる
  - ローグライク成長が反映される場合、メンバーの初期状態が前回より改善されていることがわかる演出がある
- **設計メモ**: 2回目以降のプレイでも結成シーンは表示するが、テキストのバリエーションを持たせて繰り返し感を軽減する。成長済みメンバーには「前より落ち着いている」等の描写を加える。

### NARR-003：プロジェクト終了の演出を受ける

- **As a** プレイヤー
- **I want** 全5スプリント完了後に「プロジェクト終了」の演出を見たい
- **So that** 自分の判断の積み重ねがどういう結末を生んだのかを実感できる
- **優先度**: Must
- **規模**: L
- **依存**: CORE-001, GOAL-001, PARAM-001, PARAM-002, UI-004
- **受入条件**:
  - 「勝利/敗北」の二値ではなく、「走り続けられる状態を作れたか」という観点で結末が語られる
  - 以下4つの視点からテキスト演出が提示される（各1〜3文）:
    - PMの一言（プロジェクト全体への評価）
    - ユーザーの反応（プロダクトに対する声）
    - ステークホルダーの態度（継続判断・信頼の温度感）
    - チームの空気（メンバーの状態・成長実感）
  - エンディング称号が付与される（UI-004参照）
  - 数値スコアは主役にならず、表示する場合も選択したゴールの評価軸のみに限定する
  - 「なぜこの結果になったのか」を振り返れるよう、プレイ中の主要な判断ポイントがダイジェストで表示される
- **設計メモ**: リザルト画面の情報量が多すぎると消化しきれない。テキスト演出→称号→（任意で展開）判断ダイジェスト、の段階的な構成にする。

### NARR-004：スプリントごとの物語の流れを感じる

- **As a** プレイヤー
- **I want** スプリント1〜5それぞれに異なる空気感・テーマがあると感じたい
- **So that** 5スプリントが単調な繰り返しではなく、物語として盛り上がりを感じられる
- **優先度**: Should
- **規模**: M
- **依存**: CORE-001
- **受入条件**:
  - 各スプリント開始時に、プロジェクトの「今の空気」を伝える短いナレーションが入る
  - スプリントの位置によってトーンが変化する:
    - Sprint 1：期待と手探り（「チームはまだ互いの力を知らない」）
    - Sprint 2：現実の壁（「見積もりの甘さが見えてくる」）
    - Sprint 3：中盤の正念場（「何かを諦めなければ前に進めない」）
    - Sprint 4：追い込み（「残り時間が見えてきた」）
    - Sprint 5：収束と決断（「最後に何を届けるか」）
  - ナレーションはチーム状態・歪みに応じてバリエーションが変わる

## Epic 3. ゴール設計

### GOAL-001：プロジェクトゴールを選択する

- **As a** プレイヤー
- **I want** スプリント0でプロジェクトの「勝ち筋（暫定ゴール）」を複数候補の中から選びたい
- **So that** 自分なりの戦略を立て、プロジェクト全体の方向性を定められる
- **優先度**: Must
- **規模**: M
- **依存**: NARR-001, NARR-002
- **受入条件**:
  - 以下4種類のゴールが提示される:
    - **安定稼働ゴール**：障害・重大バグを抑えて信頼を守る（評価軸：インシデント抑制度・信頼回復）
    - **価値最大化ゴール**：ユーザー価値の伸びを優先（評価軸：ユーザー反応の質・価値の到達度）
    - **納期死守ゴール**：期限達成を最優先（評価軸：マイルストーン達成・遅延の少なさ）
    - **品質文化ゴール**：品質・改善の土台を作る（評価軸：負債の抑制度・文化成熟度）
  - 各ゴールに達成難易度と固有の評価軸がテキストで説明される
  - ゴールは「暫定」であり後に変更可能であることが明示される
  - PM提示の前提条件によって、ゴールごとの難易度バランスが変動する（例：期限が短いプロジェクトでは納期死守が特に困難になる）
  - ゴール選択後、各ロールが選ばれたゴールに対して一言反応する
- **設計メモ**: ゴール選択はプレイの方向性を大きく決める判断なので、各ゴールの「何が評価され、何が犠牲になりやすいか」をプレイヤーが理解できる情報量にする。ただし最適解が読めてしまうほど詳細にはしない。

### GOAL-002：中盤でゴールを修正・切り替えする

- **As a** プレイヤー
- **I want** スプリント3終了後の判断ポイントで、当初のゴールを修正または別のゴールに切り替えたい
- **So that** 「最初の計画通りにいかない」現実に対応し、より良い着地を目指せる
- **優先度**: Should
- **規模**: M
- **依存**: GOAL-001, CORE-001
- **受入条件**:
  - スプリント3の振り返り後に「ゴール修正判断」フェーズが挿入される
  - 現在のプロジェクト状態（歪み・信頼度・進捗）を踏まえた上で判断できるよう、状況サマリーが表示される
  - 変更にはコストが伴う:
    - チーム信頼性の低下（「方針がブレた」と見なされる）
    - 手戻りコスト（スプリント4の実行タスク枠が1つ減る）
    - メンバーのモラル低下（個人歪みが微増する）
  - 「変更しない（現行ゴール維持）」も明示的な選択肢として存在し、維持にもメリットがある（信頼維持・チームの一貫性）
  - 変更後のゴールに応じてリザルトの評価軸が切り替わる
- **設計メモ**: 変更コストは「変更したら損」と感じるほど重くせず、「現状維持も変更も一長一短」になるバランスにする。アジャイルの「適応」を肯定する設計思想を反映する。

### GOAL-003：ゴール評価を内部で計算する

- **As a** ゲームシステム
- **I want** 選択されたゴールの評価軸に基づいてスプリントごとの達成度を内部計算したい
- **So that** リザルト演出やエンディング称号の決定に正確な根拠を持たせられる
- **優先度**: Must
- **規模**: M
- **依存**: GOAL-001, PARAM-001, PARAM-002
- **受入条件**:
  - 各ゴールの評価軸ごとにスプリント単位で達成度が加減算される
  - 評価値はプレイヤーに直接公開しない（裏パラメータ）
  - 評価値はリザルトのテキスト演出のバリエーション選択に使用される
  - ゴール変更（GOAL-002）が行われた場合、変更後のゴールの評価軸で残りスプリントが計算され、変更前の蓄積は部分的に引き継がれる

## Epic 4. スプリントのコアループ

### CORE-001：スプリントを繰り返し実行する

- **As a** プレイヤー
- **I want** 計画→実装→リリース→レビュー→振り返りのサイクルを繰り返したい
- **So that** 小さな価値を出し続けながらプロジェクトを前進させられる
- **優先度**: Must
- **規模**: XL
- **依存**: GOAL-001
- **受入条件**:
  - 1プレイは全5スプリントで構成される
  - 各スプリントは以下5フェーズを固定順で通過する:
    1. スプリントプランニング（PLAN-001, PLAN-002）
    2. インクリメント（DEV-001）
    3. リリース（REL-001）
    4. レビュー（REV-001）
    5. 振り返り（RETRO-001）
  - 1スプリント内でプレイヤーは3〜5回の意思決定を行う
  - 1プレイ全体で15〜25分に収まる
  - スプリント間のつなぎにスプリント予報（EVT-003）が表示される
  - スプリント3終了後にゴール修正判断（GOAL-002）が挿入される

### CORE-002：スプリント内の難度が後半に向けて上昇する

- **As a** プレイヤー
- **I want** スプリントが進むにつれて判断の難しさや緊張感が増してほしい
- **So that** 後半に向けて盛り上がる体験ができ、最終スプリントにクライマックス感がある
- **優先度**: Must
- **規模**: M
- **依存**: CORE-001, DIST-001, DIST-002
- **受入条件**:
  - スプリント1では歪みが少なく、タスク候補も比較的選びやすい
  - スプリントが進むにつれて以下が変化する:
    - 歪みの蓄積によりタスクの実行コストが増加する
    - イベントの発生率が状態に応じて上昇する
    - ステークホルダーの期待値が高まり、レビューの要求水準が上がる
    - タスク候補間のトレードオフがより鋭くなる
  - ただし、歪みを抑えたプレイでは後半も比較的穏やかに進む（チーム管理の成果が体感できる）
- **設計メモ**: 難度上昇は「時間経過で自動的に上がる」のではなく「歪みの蓄積結果として上がる」設計にする。うまく回せたプレイヤーは後半も快適、という因果関係を保つ。

## Epic 5. スプリントフェーズ詳細

### PLAN-001：スプリントプランニングでタスクを選択する

- **As a** プレイヤー
- **I want** 複数のタスク候補の中から今スプリントで取り組むものを選びたい
- **So that** 限られたリソースで何に注力するかを自分で決断できる
- **優先度**: Must
- **規模**: L
- **依存**: CORE-001, VOICE-001
- **受入条件**:
  - 毎スプリント、3〜4つのタスク候補が提示される
  - プレイヤーはそのうち2つを選択する（タスク枠=2が基本）
  - 各タスクには以下の情報がテキストで示される:
    - どのロールが主に関わるか
    - 期待される効果（ユーザー価値・品質・技術基盤など）
    - 潜在リスク（「ただし○○が不安定になる可能性がある」等）
  - タスク候補はプロジェクト状態・歪み・ゴール・前スプリントの結果によって動的に変化する
  - チーム歪みが高い場合、タスクの実行コストが増加していることがテキストで示される（例：「技術的負債が重く、いつもより時間がかかりそうだ」）
- **設計メモ**: 各ロールがそれぞれの立場から「このタスクを優先すべき理由」を「声」として提示する（VOICE-001）。これが「対立する正義の選択」の中心体験。選べなかったタスクは次スプリントのバックログに残る場合がある。

### PLAN-002：リファイメントにリソースを割くか判断する

- **As a** プレイヤー
- **I want** プランニング中に「次スプリントのバックログ整理に時間を割く」かどうかを選びたい
- **So that** 目先の成果と将来の準備のバランスを自分で判断できる
- **優先度**: Should
- **規模**: S
- **依存**: PLAN-001
- **受入条件**:
  - プランニングフェーズでタスク選択と同時に「リファイメントに時間を割く」選択肢が提示される
  - 選択すると今スプリントのタスク枠が2→1に減少する
  - 代わりに次スプリントで以下の効果が得られる:
    - タスク候補の情報量が増える（リスクがより正確に見える）
    - 状態依存イベントの発生確率が軽減される
  - リファイメントを選ばなくてもゲームは進行する（必須ではない）
  - 最終スプリント（Sprint 5）ではリファイメント選択肢は表示されない（次がないため）

### DEV-001：開発中にイベントへ対応する

- **As a** プレイヤー
- **I want** インクリメント（実装・テスト）フェーズ中に発生するイベントに対して対応方針を選びたい
- **So that** 計画通りにいかない状況で判断力が試される
- **優先度**: Must
- **規模**: L
- **依存**: PLAN-001, EVT-001, EVT-002
- **受入条件**:
  - イベントは状態依存（EVT-001）または外部要因（EVT-002）により発生する
  - イベント発生時、2〜3の対応選択肢が提示される
  - 各選択肢には短期的な効果と長期的な影響がある（長期的影響はテキストのニュアンスで示唆する程度に留め、明示しすぎない）
  - イベントが発生しないスプリントもあり得る（毎回必ず問題が起きるわけではない）
  - 複数イベントが同時発生する場合は最大2つまでとし、優先順位をつけて対応する
- **設計メモ**: イベント未発生のスプリントでは、開発フェーズはスキップまたは簡易テキスト（「開発は順調に進んだ」）で短く処理する。テンポを保つため、判断がないフェーズは引き伸ばさない。

### REL-001：リリース判断を行う

- **As a** プレイヤー
- **I want** スプリント終了時にインクリメントをリリースするかどうか、どの範囲で出すかを判断したい
- **So that** 「出す勇気」と「止める勇気」の両方を体験できる
- **優先度**: Must
- **規模**: M
- **依存**: PLAN-001, DEV-001
- **受入条件**:
  - リリースフェーズで以下のような選択肢が提示される:
    - **全機能リリース**：計画通りに出す（品質リスクがある場合は歪み増加の可能性）
    - **一部機能を削ってリリース**：安定性を優先（ユーザー期待を一部裏切る）
    - **リリース延期**：品質を万全にする（ステークホルダーの不満が増加）
  - 選択肢の内容・リスクはスプリント内の開発進捗やイベント結果に応じて動的に変化する
  - 各選択肢がチーム・ユーザー・ステークホルダーの三者に与える影響が異なる
  - リリース内容はレビューフェーズ（REV-001）の反応に直接影響する
  - 開発中にイベントが発生しなかった場合でも、リリース判断は必ず行う

### REV-001：レビューでフィードバックを受け取る

- **As a** プレイヤー
- **I want** リリース後にユーザーとステークホルダーからのフィードバックをテキスト演出で受け取りたい
- **So that** 自分の判断がどう受け止められたかを実感し、次の計画に活かせる
- **優先度**: Must
- **規模**: M
- **依存**: REL-001, PARAM-002
- **受入条件**:
  - フィードバックはテキストの「温度感」で表現される（数値ゲージではない）
  - ユーザーとステークホルダーの反応は別々に、それぞれ1〜3文で提示される
  - 反応内容は以下の要素から決定される:
    - リリース内容（何を出したか・何を削ったか）
    - チーム信頼性の現在値
    - 選択中ゴールとの整合性
    - 累積的な歪みの状態
  - まれに「称賛フィードバック」が入る（低確率だが発生するとポジティブ体験が強い）
  - リリース延期を選んだ場合、レビューフェーズは「デモなし」の演出になる（ステークホルダーの反応のみ表示）
- **設計メモ**: 称賛フィードバックの条件は「ゴールに沿った判断を連続で行っている」「歪みが低い状態でリリースできた」など。レアだが狙える位置に置く。

### RETRO-001：振り返りで改善アクションを選ぶ

- **As a** プレイヤー
- **I want** 各スプリント終了時の振り返りで、チームの働き方を改善するアクションを選びたい
- **So that** チームが少しずつ賢くなっていく実感を得られ、歪みに対抗する手段を持てる
- **優先度**: Must
- **規模**: M
- **依存**: REV-001, DIST-001, DIST-002
- **受入条件**:
  - 振り返りフェーズで2〜3の改善アクション候補が提示される
  - アクションには2種類がある:
    - **即効型**：次スプリントで効果が出る（例：「残業ルールの見直し → 次Sprint、疲労蓄積が軽減」）
    - **遅効型**：2スプリント後以降に効果が出る（例：「テスト自動化の導入準備 → Sprint後半でバグ発生率低下」）
  - 歪み（個人・チーム）の緩和に寄与するアクションが含まれる
  - アクション候補はチーム状態・スプリント内の出来事に応じて動的に変化する
  - 最終スプリント（Sprint 5）の振り返りは「プロジェクト全体の振り返り」となり、通常の改善選択ではなくナレーション演出に移行する

## Epic 6. ロールの「声」

### VOICE-001：ロールごとの「声」を選択肢に反映する

- **As a** プレイヤー
- **I want** 各判断場面で、5つのロール（PdM・FE・BE・QA・デザイナー）がそれぞれの立場から意見を述べるのを聞きたい
- **So that** 「誰が正しいか」ではなく「今回はどの正義を選ぶか」を決断する体験ができる
- **優先度**: Must
- **規模**: XL
- **依存**: PARAM-001
- **受入条件**:
  - 判断場面ごとに、関係するロール（2〜4人）の「声」がテキストで提示される
  - 全場面で全5ロールが発言するわけではない（場面に関係するロールのみ）
  - 各ロールの基本的な主張の方向性:
    - **PdM**：「ユーザーにとっての価値は何か」「優先順位はこうすべき」
    - **FE**：「ユーザーが触る部分を早く形にしたい」「体感速度が大事」
    - **BE**：「土台がしっかりしないと後で崩れる」「設計を急ぐな」
    - **QA**：「このまま出すのは危険」「テストが足りていない」
    - **デザイナー**：「導線が破綻している」「一貫性がないと信用を失う」
  - デザイナーは「美しさ」ではなく「伝わりやすさ・体験の破綻防止」の観点から発言する
  - 各ロールの主張はそれぞれ合理的で、互いに矛盾し得る

### VOICE-002：個人状態によって「声」が変化する

- **As a** プレイヤー
- **I want** メンバーの個人パラメータ（疲労・ストレス・スキル成熟度）に応じて、そのロールの「声」の質や内容が変化してほしい
- **So that** チームメンバーの状態がゲームの手触りとして直接伝わる
- **優先度**: Must
- **規模**: L
- **依存**: VOICE-001, PARAM-001, DIST-001
- **受入条件**:
  - 健康なメンバーは明確で建設的な提案をする
  - 疲労が高いメンバーは提案が消極的になる（「正直、もう余裕がない……」）
  - ストレスが高いメンバーは攻撃的・防衛的になる（「なんで毎回こっちにしわ寄せが来るんだ」）
  - スキル成熟度が低い段階では、提案の精度が低い（見積もりが甘い、リスク認識が浅い）
  - 個人歪みが極度に悪化すると、そのロールの「声」自体が消失する（判断材料が減る）
- **設計メモ**: 「声」の変化はゲームの情報伝達手段であると同時に、キャラクターへの愛着を生む演出でもある。疲弊したメンバーの台詞がプレイヤーに「休ませてあげたい」と思わせることが理想。

### VOICE-003：選択肢が三者に異なる影響を与える

- **As a** プレイヤー
- **I want** 各選択肢がチーム・ユーザー・ステークホルダーの三者に対して異なる影響を持つことを感じたい
- **So that** win–win–winを目指す難しさと面白さを体験できる
- **優先度**: Must
- **規模**: M
- **依存**: VOICE-001, PARAM-002
- **受入条件**:
  - 選択肢の影響はテキストの温度感で間接的に伝えられる（数値バーの上下ではない）
  - 全員が最大に満足する選択肢は基本的に存在しない
  - 特定の三者を優先し続けると、他の二者への歪みが蓄積するバランスになっている
  - ただし、稀にwin–win–winに近い結果が生まれる瞬間がある（ゾーン状態時など）

## Epic 7. パラメータシステム

### PARAM-001：個人パラメータを管理する

- **As a** ゲームシステム
- **I want** 各メンバー（5人）の個人パラメータを管理したい
- **So that** 個人の状態がロールの「声」の質やイベント発生率に影響を与えられる
- **優先度**: Must
- **規模**: L
- **依存**: なし
- **受入条件**:
  - 5人それぞれに以下の独立した個人パラメータが存在する:
    - **スキル成熟度**：判断の選択肢の質や効果に影響。初期値はローグライク成長で変動。
    - **疲労**：スプリントごとの作業負荷で蓄積。高いと提案が弱体化し、病欠リスクが上がる。
    - **ストレス**：不利な判断・イベント対応の負荷で蓄積。最大で脱退イベントが発生する。
    - **健康状態**：疲労・ストレスの複合で決まる総合指標。低いと病欠・不調が発生しやすい。
  - パラメータはプレイヤーに直接数値で見せず、ロールの「声」のトーン変化やUI演出（表情アイコン等）で間接的に伝える
  - 各パラメータの増減ロジック:
    - タスクに関わったロールは疲労が増加する
    - 自分のロールの意見が無視され続けるとストレスが増加する
    - 振り返りアクションで疲労・ストレスが軽減される

### PARAM-002：チームパラメータを管理する

- **As a** ゲームシステム
- **I want** チーム全体のパラメータを管理したい
- **So that** 個人の合算とは異なるチーム固有の状態がゲーム展開に影響を与えられる
- **優先度**: Must
- **規模**: L
- **依存**: PARAM-001
- **受入条件**:
  - 以下のチームパラメータが存在する:
    - **チーム信頼性**：ユーザー・ステークホルダーからの信用。レビューの反応基準、外部イベントの厳しさに影響。リリース判断・レビュー結果で増減する。
    - **開発文化の成熟度**：CI/CD・改善習慣の定着度。歪みの蓄積速度、選択肢の質に影響。振り返りアクション・文化アンロックで向上する。
    - **持続可能性**：仕組みによる耐久力。個人の欠けに対するチームの回復力を決定。文化アンロック・継続的な健全運営で向上する。
  - 個人パラメータの影響を受けるが完全には連動しない
  - チームパラメータもプレイヤーに直接数値で見せず、レビュー反応やイベントの厳しさとして間接的に体感させる

## Epic 8. 歪み（Distortion）システム

### DIST-001：個人歪みを蓄積・処理する

- **As a** ゲームシステム
- **I want** 個人由来の歪みを蓄積し、ゲーム状態に影響させたい
- **So that** プレイヤーの判断が個人レベルのリスクとして徐々に積み上がる
- **優先度**: Must
- **規模**: M
- **依存**: PARAM-001
- **受入条件**:
  - 個人歪みはスプリントごとの判断・イベント対応の結果として蓄積する
  - 蓄積の主な要因:
    - 該当ロールに負荷の高いタスクを連続で割り当てる
    - そのロールの意見を無視する判断が続く
    - イベント対応で「無理をさせる」系の選択をする
  - 蓄積量に応じた段階的影響:
    - 軽度：提案のトーンが消極的になる
    - 中度：提案の精度が低下する（見積もりが不正確になる）
    - 重度：提案自体が消失する（判断材料が減る）
    - 最大：病欠・脱退イベントが発生する
  - 緩和手段：振り返りアクション、タスク配分の分散、文化アンロック効果
  - 完全にゼロにはならない（蓄積は不可逆的に一部残る）

### DIST-002：チーム歪みを蓄積・処理する

- **As a** ゲームシステム
- **I want** チーム由来の歪みを蓄積し、全体に影響させたい
- **So that** チーム全体の健康がゲームの後半戦を左右するようになる
- **優先度**: Must
- **規模**: M
- **依存**: PARAM-002, DIST-001
- **受入条件**:
  - チーム歪みの主な種類:
    - **技術的負債**：品質を犠牲にした判断で蓄積。タスクの実行コストを増加させる。
    - **信頼負債**：リリース延期・品質問題で蓄積。ステークホルダーの要求が厳しくなる。
    - **文化負債**：振り返りを軽視・改善を怠ると蓄積。イベント対応力が低下する。
  - 個人歪みからチーム歪みへの波及ロジック:
    - 持続可能性が低いと、個人歪みがチーム歪みに波及しやすい
    - 持続可能性が高いと、個人の問題がチーム全体に影響しにくい
  - チーム歪みが高まると選択肢の実行コストが増加する
  - 振り返りアクション・文化アンロックで緩和可能だが、完全にはゼロにならない

### DIST-003：持続可能性と疲労の相互作用を処理する

- **As a** ゲームシステム
- **I want** 持続可能性と個人疲労の関係が構造的な違いとしてゲーム体験に表れるようにしたい
- **So that** 「仕組みで支える」か「個人に依存する」かの差がプレイを通じて実感できる
- **優先度**: Must
- **規模**: M
- **依存**: PARAM-001, PARAM-002, DIST-001, DIST-002
- **受入条件**:
  - **持続可能性が高いチーム**:
    - 1人が病欠・高疲労でも他メンバーへの負荷増加が軽微
    - 歪みの回復速度が速い
    - タスク枠の一時的な減少からの復帰が早い
  - **持続可能性が低いチーム**:
    - 1人の問題がチーム歪みに直結する（連鎖崩壊リスク）
    - 歪みの回復が遅い
    - 脱退発生時のダメージが大きい
  - この関係性はプレイヤーに明示的に説明されず、結果として体感的に理解される
- **設計メモ**: 持続可能性はゲーム後半の「保険」として機能する。序盤に仕組み作りに投資したプレイヤーが後半に報われる設計。

## Epic 9. イベントシステム

### EVT-001：状態依存イベントを発生させる

- **As a** ゲームシステム
- **I want** チーム・個人の状態に応じてイベントの発生率を変動させたい
- **So that** プレイヤーの過去の判断がイベントという形でフィードバックされる
- **優先度**: Must
- **規模**: XL
- **依存**: PARAM-001, PARAM-002, DIST-001, DIST-002
- **受入条件**:
  - 以下のイベントタイプと発生条件:
    - **バグ発生（品質起因）**
      - 発生条件：品質が低い・スコープが大きい・技術的負債が高い
      - 対応選択肢：即修正（タスク枠消費）／既知バグとしてリリース（信頼リスク）／機能トグルで切る（スコープ縮小）
    - **病欠・不調（健康起因）**
      - 発生条件：個人の疲労・健康が悪い
      - 対応選択肢：休ませる（タスク枠減少だが回復）／代替で回す（他メンバーの負荷増）／無理をさせる（短期的には回るが歪み急増）
    - **脱退（ストレス起因）**
      - 発生条件：ストレスが最大に到達
      - 処理：新人が補充されるが、該当ロールの成長が一部リセットされる（EVT-004参照）
    - **インフラトラブル（運＋成熟度起因）**
      - 発生条件：基本は低確率だが、インフラ成熟度・運用文化で軽減可能
      - 対応選択肢：即時復旧（コスト大）／原因調査（次回以降の発生率低下）／暫定回避（問題先送り）

### EVT-002：外部要因イベントを発生させる

- **As a** ゲームシステム
- **I want** チーム状態とは無関係に発生する外部要因イベントを用意したい
- **So that** プレイごとの展開が変化し、正解ルートが固定化されない
- **優先度**: Must
- **規模**: L
- **依存**: CORE-001
- **受入条件**:
  - 以下のイベントタイプ:
    - **神の一声**（仕様追加要求）：PMまたはステークホルダーから突然の要求。受け入れる（スコープ増・信頼向上）／交渉する（部分対応・信頼微減）／断る（スコープ維持・信頼低下）
    - **依存ライブラリ問題**（外部要因の手戻り）：既存タスクのコストが増加。対応に時間を使う／影響範囲を限定する／無視して進む（後で爆発リスク）
    - **災害級インシデント**（超低確率）：残りスプリントの行動が大幅に制限。救済措置として最低保証（リザルトで報酬補正）を前提に扱う
  - 外部要因イベントの頻度は状態依存イベントより低く設定する
  - 1プレイで災害級インシデントは最大1回まで

### EVT-003：スプリント予報で予兆を伝える

- **As a** プレイヤー
- **I want** 次スプリントで起こりやすい問題を「天気予報」のような形で事前に知りたい
- **So that** 失敗が「運ゲー」ではなく「準備不足だった」と納得できる
- **優先度**: Should
- **規模**: M
- **依存**: EVT-001, RETRO-001
- **受入条件**:
  - 各スプリント終了時（振り返り後）に、次スプリントの予報が表示される
  - 予報の精度は100%ではなく、傾向を示唆する程度:
    - 状態依存イベントの一部は予報に反映される（例：「負債の嵐に注意」＝技術的負債が高い）
    - 外部要因イベントは予報に出ないか、極めて曖昧にしか出ない（「外からの風が吹くかもしれない」程度）
  - 予報のフォーマット例：天気アイコン＋短いテキスト
  - リファイメント（PLAN-002）を実施した場合、予報の精度が向上する（より具体的な表現になる）
  - 最終スプリント（Sprint 5）終了後の予報は表示しない

### EVT-004：メンバー脱退と補充を処理する

- **As a** ゲームシステム
- **I want** メンバー脱退時に新人が補充される仕組みを持ちたい
- **So that** 脱退が「詰み」ではなく「痛いが続けられる」体験になる
- **優先度**: Must
- **規模**: M
- **依存**: EVT-001, PARAM-001, GROW-001
- **受入条件**:
  - 脱退イベント発生時、該当ロールに新人メンバーが補充される
  - 補充メンバーの初期パラメータ:
    - スキル成熟度はローグライク成長の現在値より低い（成長の一部リセット）
    - 疲労・ストレスはゼロからスタート
  - 脱退スプリントは該当ロールの「声」が不在になる（判断材料が一時的に減少する）
  - 補充メンバーは次スプリントから「声」に参加する
  - 脱退の事実はリザルト演出に反映される（称号にも影響する可能性がある）
  - 1プレイで脱退は最大2回まで（3人以上抜けるとゲーム体験が崩壊するため）

## Epic 10. ローグライク成長

### GROW-001：個人成長をプレイ間で引き継ぐ

- **As a** プレイヤー
- **I want** プロジェクト完了後、各ロールの個人成長が次回プレイに引き継がれてほしい
- **So that** 繰り返しプレイでチームメンバーが徐々に成熟していく実感を得られる
- **優先度**: Must
- **規模**: M
- **依存**: NARR-003, PARAM-001
- **受入条件**:
  - ロール別に以下の方向で成長する:
    - **PdM**：初期の優先順位判断の甘さが緩和される
    - **FE**：見た目優先の偏りが減り、パフォーマンス意識が育つ
    - **BE**：過剰設計の傾向が抑えられ、実用的な判断ができる
    - **QA**：過剰品質の傾向が抑制され、リスクベースの判断ができる
    - **デザイナー**：理想追求の傾向が和らぎ、現実的な落としどころを提案できる
  - 歪み耐性が向上する（疲労やストレスの影響を受けにくくなる）
  - 成長は「数値が上がる」ではなく「弱点が減る」方向で表現する
  - 成長度合いはプレイ中の経験（そのロールに関わる判断をどれだけ行ったか）に影響される
- **設計メモ**: 成長の上限を設け、完全に弱点がなくなることはない。「熟練しても判断は難しい」というゲームの基本体験を維持する。

### GROW-002：チーム文化をアンロックする

- **As a** プレイヤー
- **I want** 特定の条件を達成することでチーム文化をアンロックし、次回以降のプレイに反映させたい
- **So that** 繰り返しプレイの動機が生まれ、「チームが賢くなる」感覚を味わえる
- **優先度**: Must
- **規模**: L
- **依存**: NARR-003, UI-004, PARAM-002
- **受入条件**:
  - 以下の文化アンロック（例）と解放条件:
    - **CI/CD導入**：品質ゴール系称号を1回達成 → 効果：バグ発生率が全体的に軽減
    - **ペアプログラミング**：脱退ゼロでプロジェクトを2回完了 → 効果：1人欠けた時の影響が軽減（持続可能性ベースボーナス）
    - **心理的安全性**：脱退ゼロ かつ ストレス最大に達したメンバーなしで完了 → 効果：ストレス蓄積速度が低下、「声」の消失が起きにくくなる
    - **振り返り文化**：全スプリントで振り返りアクションを選択して3回完了 → 効果：振り返りアクションの選択肢が1つ増える
    - **インフラ成熟**：安定稼働ゴール系称号を1回達成 → 効果：インフラトラブル発生率が低下
  - アンロックされた文化は次回プレイ開始時から有効
  - 効果は構造的な改善として表現される（数値ボーナスの加算ではない）
  - アンロック済みの文化一覧がメタ画面（META-001）で確認できる
  - 未アンロック文化には解放条件のヒントが表示される（具体的すぎない程度に）

### GROW-003：選択肢の質を向上させる

- **As a** プレイヤー
- **I want** 繰り返しプレイにより、判断場面の選択肢が単純な二択から、より現実的な三択・折衷案に変化してほしい
- **So that** チームの成熟が「できることが増える」ではなく「判断の幅が広がる」として体験できる
- **優先度**: Should
- **規模**: L
- **依存**: GROW-001, GROW-002, VOICE-001
- **受入条件**:
  - 初回プレイでは二択が中心
  - 文化アンロックや個人成長に応じて、特定の判断場面で第三の選択肢（折衷案・創造的解決策）が出現する
  - 例：
    - 初期：「全機能リリース」vs「リリース延期」
    - 成長後：「全機能リリース」vs「リリース延期」vs「段階リリース（コア機能のみ先行、残りは次Sprint）」
  - 新たに出現した選択肢は万能ではなく、それぞれ異なるトレードオフを持つ
  - 選択肢の増加は判断場面の一部に限定し、すべてが三択になるわけではない（全場面が三択になると情報量過多で判断疲れを起こす）

### GROW-004：メタ成長の永続化を管理する

- **As a** ゲームシステム
- **I want** ローグライク成長のデータ（個人成長度・文化アンロック・称号コレクション）を永続的に保存したい
- **So that** プレイヤーがブラウザを閉じても成長の蓄積が失われない
- **優先度**: Must
- **規模**: M
- **依存**: GROW-001, GROW-002, UI-004
- **受入条件**:
  - 以下のデータがプレイ間で永続保存される:
    - 各ロールの個人成長度
    - アンロック済み文化の一覧
    - 獲得済みエンディング称号の一覧
    - プレイ回数・各ゴールの選択回数
  - データの保存はプロジェクト終了時（リザルト表示後）に自動で行われる
  - プレイ中断時（ブラウザを閉じた場合等）は成長データは更新されない（完了したプレイのみ反映）
  - データリセット機能を提供する（確認ダイアログ付き）

## Epic 11. 演出・UI

### UI-001：バーンダウンチャートで進捗を表示する

- **As a** プレイヤー
- **I want** スプリントの進捗をバーンダウンチャートとして常時表示してほしい
- **So that** 理想線との乖離がそのまま緊張感として感じられる
- **優先度**: Should
- **規模**: M
- **依存**: CORE-001, PLAN-001
- **受入条件**:
  - HPバーの代替として、バーンダウンチャートがUI上に常時表示される
  - 理想線（計画通りの進捗）と実績線が視覚的に比較できる
  - スプリント内の判断・イベントによって実績線がリアルタイムに変動する
  - チャートはスプリント単位でリセットされる（5スプリント全体のチャートではない）
  - プロジェクト全体の進捗は別のシンプルな指標（例：「Sprint 3/5」）で表示する

### UI-002：歪みをカード・タグとして可視化する

- **As a** プレイヤー
- **I want** 蓄積された歪みが「カード」や「タグ」として視覚的に表示されてほしい
- **So that** 現在のチーム状態を直感的に把握し、判断の材料にできる
- **優先度**: Should
- **規模**: M
- **依存**: DIST-001, DIST-002
- **受入条件**:
  - 個人歪みとチーム歪みが区別可能な形で表示される:
    - 個人歪み：メンバーアイコンの近くにタグとして表示（例：「FE：疲労」「QA：高ストレス」）
    - チーム歪み：画面の共有エリアにカードとして表示（例：「技術的負債：中」「信頼低下」）
  - 歪みの増減がスプリント間で視覚的にわかる（新規追加・解消時にアニメーションまたはハイライト）
  - 歪みカード/タグにはその歪みの影響を示す短い説明がある（例：「技術的負債：タスクの実行に余計な時間がかかる」）
  - 歪みが増えすぎると画面が圧迫される演出にすることで、「あふれる系」の視覚的プレッシャーを表現する

### UI-003：ゾーン（Flow）演出を表示する

- **As a** プレイヤー
- **I want** チームがうまく噛み合った瞬間に特別な演出（ゾーン）が発生してほしい
- **So that** 判断の連続の中で「うまく回った」と感じる明確な成功体験が得られる
- **優先度**: Could
- **規模**: M
- **依存**: DIST-001, DIST-002, REL-001
- **受入条件**:
  - 発動条件（いずれかを満たす）:
    - **意図的に狙える条件**：2スプリント連続で歪みを増やさずにリリース成功
    - **偶発的に揃う条件**：全ロールの提案が2スプリント連続で採用された（特定ロールを無視していない）
  - 発動中の効果（次スプリントに適用）:
    - タスクの実行効率が上昇（実質的にタスク枠が0.5増えるような効果）
    - 状態依存イベントの発生確率が低下
    - レビューでのフィードバックがポジティブに傾きやすい
  - 演出:
    - 発動時にテキスト演出（「チームが完全に噛み合っている」）
    - UI上にゾーンインジケーターが一時的に表示される
  - ゾーンは連続維持が可能だが、歪みが増えた時点で解除される

### UI-004：エンディング称号を付与する

- **As a** プレイヤー
- **I want** プロジェクト終了時にプレイ内容を象徴するエンディング称号を受け取りたい
- **So that** リプレイ時に「違う称号を目指そう」というモチベーションが生まれる
- **優先度**: Should
- **規模**: L
- **依存**: NARR-003, GOAL-003
- **受入条件**:
  - 称号はプレイ内容から決定される。決定要素:
    - 選択したゴール（最終ゴール）
    - 歪みの最終的な蓄積具合
    - イベント対応の傾向（品質重視か速度重視か等）
    - 特殊条件（脱退発生の有無、ゾーン発動回数、ゴール変更の有無等）
  - 称号の例（最低15種類を初期実装）:
    - **傷だらけの栄光**：歪みが大きいがゴール達成度が高い
    - **未完のユートピア**：理想を追いすぎて納期が厳しかった
    - **技術的破産**：技術的負債が限界を超えた
    - **静かなる船出**：安定稼働ゴールを低歪みで達成
    - **全員生還**：脱退・病欠ゼロで完了
    - **ピボットの英断**：ゴール変更して良い結果を得た
    - **ゾーンランナー**：ゾーンを複数回発動した
  - 同じゴールでも異なる称号が出る可能性がある
  - 称号はメタ画面（META-001）に蓄積され、コレクション要素として機能する
  - 一部の称号が文化アンロック（GROW-002）の解放条件になる

## Epic 12. リプレイ性

### RPL-001：プロジェクト提示条件を毎回変化させる

- **As a** プレイヤー
- **I want** プレイごとにPMから提示されるプロジェクトの条件が変化してほしい
- **So that** 同じ戦略が毎回通用せず、新鮮な判断体験が維持される
- **優先度**: Must
- **規模**: M
- **依存**: NARR-001
- **受入条件**:
  - プロジェクト提示条件の4軸がプレイごとにランダムまたは半ランダムで変動する:
    - 期限の長さ（余裕あり〜厳しい）
    - 期待値の高さ（控えめ〜野心的）
    - 要件の曖昧さ（明確〜不明瞭）
    - ステークホルダーの圧（穏やか〜高圧的）
  - 条件の組み合わせによって有効な戦略が変わる
  - 極端な組み合わせ（全軸最高難度など）は出現しないよう制御する
  - プレイヤーのメタ成長に応じて、徐々に高難度寄りの条件が出現する確率が上がる（ただし低難度も引き続き出る）

### RPL-002：状態依存イベントでプレイ内容に応じた変化を生む

- **As a** プレイヤー
- **I want** 自分の判断の積み重ねによって発生するイベントが変化してほしい
- **So that** 「同じことをしても同じ結果にならない」体験があり、リプレイが面白くなる
- **優先度**: Must
- **規模**: M（EVT-001に含まれる設計だが、リプレイ性としての確認）
- **依存**: EVT-001
- **受入条件**:
  - 品質を軽視するプレイではバグ系イベントが増加する
  - メンバーを酷使するプレイでは健康系イベントが増加する
  - 信頼を損なうプレイではステークホルダー介入系イベントが増加する
  - 状態依存と純ランダムの比率は約7:3（プレイヤーの判断に因果関係を感じさせるため）
  - 同じ判断パターンでもランダム要素により展開にばらつきがある

## Epic 13. メタ画面・管理

### META-001：メタ画面でプレイ間の進捗を確認する

- **As a** プレイヤー
- **I want** プレイ間にチームの成長状態を確認できる画面がほしい
- **So that** 繰り返しプレイの蓄積を実感し、次のプレイへのモチベーションが維持される
- **優先度**: Should
- **規模**: L
- **依存**: GROW-001, GROW-002, GROW-004, UI-004
- **受入条件**:
  - メタ画面に以下のセクションが存在する:
    - **チームメンバー**：各ロールの成長段階と現在の特徴がテキストで確認できる
    - **チーム文化**：アンロック済み・未アンロックの文化一覧。未アンロックは解放条件のヒントが見える（「品質に関する称号を取ると……？」程度）
    - **称号コレクション**：獲得済みエンディング称号の一覧。未獲得称号は存在だけ示される（シルエット等）
    - **プレイ履歴**：過去のプレイの概要（選択ゴール・称号・スプリント数）が簡易的に確認できる（直近5〜10プレイ分）
  - メタ画面からゲームを開始できる（「新しいプロジェクトを始める」ボタン）
  - データリセット機能がメタ画面内に存在する（確認ダイアログ付き）

### META-002：プレイ中に現在のチーム状態を確認する

- **As a** プレイヤー
- **I want** プレイ中にいつでもチームの現在状態（各メンバーの調子・歪みの蓄積・ゴール進捗）を確認できる画面を開きたい
- **So that** 判断に必要な情報を見落とさず、意図的な判断ができる
- **優先度**: Should
- **規模**: M
- **依存**: PARAM-001, PARAM-002, DIST-001, DIST-002, UI-002
- **受入条件**:
  - プレイ中に「チーム状態」パネルを開ける（判断フェーズ中でも参照可能）
  - 表示内容:
    - 各メンバーの状態（テキストと簡易アイコンで。数値は見せない）
    - 蓄積中の歪みカード/タグの一覧
    - 現在のゴールと評価軸のリマインド
    - 現在のスプリント番号と残りスプリント数
  - 表示はあくまで「今の状況の整理」であり、最適解を示すものではない

- プレイヤーがプロジェクト条件を自由に設定するモードは今回のスコープ外。

- プレイヤーが各メンバーの名前・見た目・初期パラメータを変更する機能は今回のスコープ外。

### WONT-004：実績・トロフィーシステム

- エンディング称号とは別の実績システムは今回のスコープ外。称号コレクションで代替する。
