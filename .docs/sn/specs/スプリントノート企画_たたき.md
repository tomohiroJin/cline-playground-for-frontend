# アジャイル開発チーム・シミュレーションゲーム企画（たたき）

## 仮タイトル

**和名：** スプリントノート

**英名：** Sprint Note

**略銘：**スプノ（SN）

## 1. 企画概要

新人だけで構成された開発チームが、PMから提示されるプロジェクトに挑み、限られた期間（スプリント）の中で「価値を出し続ける」ことを目指す、テキストベースのアジャイル開発シミュレーションゲーム。

プレイヤーは特定の個人ではなく、**開発チーム全体の集合意思**を操作する立場となり、チーム・ユーザー・ステークホルダーの三者が同時に破綻しない状態（win–win–win）を目指して意思決定を行う。

成功・失敗よりも「なぜその結果になったのか」を理解できることを重視し、繰り返しプレイによるチーム成長（ローグライク要素）を備える。

---

## 2. 主人公・登場人物（ペルソナ）

### 主人公：開発チーム（5人）

全員が新人。役割ごとの得意・不得意を持つ。

- **PdM**：方向性と価値の声
- **FE**：ユーザー体験を形にする
- **BE**：プロダクトの土台を支える
- **QA**：品質とリスクを守る
- **デザイナー**：意味と一貫性をつなぐ

プレイヤーは個別キャラを直接操作せず、各ロールの「声」が反映された選択肢を選ぶ。

#### デザイナーの「声」の例（対立する正義）

- デザイナー：「導線が破綻している。今直さないと『使いづらい』で信用を落とす」（体験・一貫性重視）
- FE：「動くものを先に出す。見た目は後で改善できる」（速度重視）
- PdM：「ユーザー価値として重要なのはどっち？」（意味重視）

このように、デザイナーは「美しさ」ではなく**伝わりやすさ／体験の破綻防止**として主張する。

### 準主人公・環境要因

- **PM**：プロジェクト提示役。期限・期待値・制約条件を外部から設定する存在（チーム外）
- **ユーザー**：集合体としてフィードバックを返す
- **ステークホルダー**：評価・継続判断を握る存在

※ **PdMはチーム内ロールとして「何を作るべきか」を提案し、PMは外部から「いつ・どこまで」を要求する**。役割は明確に分離される。

## 3. ナラティブフロー（1プレイの流れ）

1. **プロジェクト提示（PM）**

   - 作るもの・期限・期待値が提示される
   - ここで難度（前提条件）が決定される

2. **チーム結成（期待しかない）**

   - 新人5人でプロジェクト開始（PdM / FE / BE / QA / Designer）
   - 全員やる気はあるが、経験は浅い

3. **スプリント0（目標設定と“勝ち筋”の選択）**

   - チームは「何をゴールに掲げるか」を選ぶ（複数から選択）
   - ゴールは暫定であり、後に変更可能
   - プロジェクトの方向性と物語が語られる

4. **スプリント1〜最終スプリント（繰り返し）**

   - 計画 → 実装 → リリース → レビュー → 振り返り
   - 反応と歪みの蓄積、イベント対応

5. **中盤判断（ピボット）**

   - ゴール修正・切り替えの是非を判断（コスト付き）

6. **プロジェクト終了（持続可能な状態）**

   - 一定スプリント（**全5スプリント想定**）到達でゲーム終了
   - 「走り続けられる状態を作れたか」で評価される

7. **リザルト・成長**

   - エンディング称号と次回へのアンロック要素

## 4. ゲームの骨格（バックボーン）

1. プロジェクトを受ける（PM提示）
2. スプリント0：目標を定める（ゴール選択）
3. スプリントを計画する
4. 開発を進める（実装・テスト）
5. リリースする
6. レビューする（ユーザー／ステークホルダー反応）
7. 振り返る（働き方改善）
8. 次のスプリントへ進む
9. プロジェクトを終える（持続可能な状態で終了）

※全プレイで必ず通過する行動単位。

### 1スプリントの標準フロー（固定順）

1. **スプリントプランニング**（何をやるか決める）
2. **インクリメント**（実装・テスト＝作る）
3. **リリース**（出す）
4. **レビュー**（見せてフィードバックを得る）
5. **振り返り**（チームの働き方を改善する）

※スプリント中にイベントが割り込むことがある（後述）。

### リファイメント（バックログ整理）の扱い

リファイメントは独立フェーズにはせず、**プランニング内のリソース配分の選択肢**として扱う。

- 例：「次スプリントのバックログ整理に時間を割く（今回のタスク枠が1つ減るが、次回の見積もり精度が上がる）」

### 1スプリントの基本解像度（目安）

1スプリント内で、プレイヤーは**3〜5回の意思決定**を行う。

- 計画フェーズ：
  - 複数タスク候補から実行内容を選択（例：3つ提示され2つ選ぶ）
  - 上記に加え、リファイメントに時間を割くかを選べる
- 開発中イベント：
  - 状態やランダムにより問題発生への対応選択
- リリース判断：
  - 出す／削る／延期する等の最終判断

この解像度により、1プレイ（1プロジェクト）は **15〜25分程度**を想定する。

## 5. ゲームの特徴

- **判断ゲーム**：管理や操作ではなく意思決定が主
- **あふれる系構造**：時間・タスク・期待が常に足りない
- **即死しない失敗設計**：失敗は歪みとして蓄積
- **win–win–winが理想だが困難**：全員最大値にはならない
- **対立する正義の選択**：各ロール（PdM / FE / BE / QA / Designer）から、互いに正しそうだが矛盾する提案が提示される

プレイヤーは「誰が正しいか」ではなく、「今回はどの正義を選ぶか」を決断する。

### チームパラメータと個人パラメータの考え方

本作では、**チーム全体としての状態**と、**各メンバー個人の状態**を分けて管理する。

- **個人パラメータ（メンバー固有）**

  - スキル成熟度（役割ごとの得意度）
  - 疲労・ストレス
  - 健康状態

- **チームパラメータ（集合的・非属人的）**

  - チーム信頼性（ユーザー／ステークホルダーからの信用）
  - 開発文化の成熟度（CI/CD、改善習慣など）
  - 持続可能性（燃え尽きにくさ）

多くの場合、**チームパラメータは個人パラメータの合算・影響を受けるが完全には一致しない**。例えば、優秀な個人がいてもチーム信頼性が低い状況はあり得る。

### 歪み（Distortion）システム概要（改訂）

歪みは「個人由来」と「チーム由来」の2系統で蓄積し、相互に影響し合う。

- **個人歪み**（疲労・ストレス・スキル不足）

  - 該当ロールの提案が弱体化・減少
  - 病欠・脱退イベントの発生率上昇

- **チーム歪み**（信頼低下・文化未成熟）

  - 選択肢の実行コスト増加
  - チーム全体に影響する制約や追加要求が発生

#### 持続可能性と疲労の関係

- **疲労・ストレス**：個人の消耗度合い。高いほど短期的な事故が起きやすい。
- **持続可能性**：仕組みや文化による耐久力。個人が欠けても回る力。

個人疲労が高くても、持続可能性が高ければ即崩壊はしない。
逆に、持続可能性が低いチームは、1人の疲労が全体崩壊に直結する。

歪みは振り返りや文化アンロックによって**緩和はできるが、完全には消えない**。

## 6. 難度設計（プロジェクト提示＋スプリント0ゴール）

難度は2層で構成する。

### A. 前提条件の難度（PM提示）

PMからの提示内容で、プロジェクトの前提条件が決まる。

- 期限の長さ
- 期待値の高さ
- 要件の曖昧さ
- ステークホルダーの圧

### B. ゴール難度（スプリント0で選択）

スプリント0で「このプロジェクトの勝ち筋（暫定ゴール）」を選ぶ。ゴールは**達成難易度**と**評価軸**を持つ。

例（案）：

- **安定稼働ゴール**：障害・重大バグを抑えて信頼を守る（評価：インシデント抑制度／信頼回復）
- **価値最大化ゴール**：ユーザー価値の伸びを優先（評価：ユーザー反応の質／価値の到達度）
- **納期死守ゴール**：期限達成を最優先（評価：マイルストーン達成／遅延の少なさ）
- **品質文化ゴール**：品質・改善の土台を作る（評価：負債の抑制度／文化成熟度）

### ゴールは固定ではない（アジャイルらしさの担保）

中盤（例：スプリント3〜4）で、**ゴールを「修正／切り替え」できる判断ポイント**を設ける。

- 変更は無料ではない（信頼低下・手戻り・モラル低下などのコスト）
- ただし、変更できることで「最初の計画通りにいかない」体験を強化する

### 評価（スコア）の扱い方針（改訂）

- 内部的には各評価軸を裏で計算するが、
- リザルトは**単一スコアの数値表示を主役にせず**、
  - PMの一言
  - ユーザーの反応
  - ステークホルダーの態度
  - チームの空気
    といった**テキスト演出**で「終わりの質」を提示する。
- どうしても数字を見せる場合でも、**選んだゴールの評価軸のみ**を表示する（最適化ゲー化を抑える）。

## 7. ローグライク的成長要素（案）

1プレイ＝1プロジェクト。

### 成長の方向性

数値の単純な上昇ではなく、「個人の成熟」と「チーム文化の獲得」を組み合わせて表現する。

- **個人成長（ロール別）**

  - 初期デメリットの緩和
  - 歪み耐性の向上

- **チーム文化のアンロック（解放条件例）**

  - 特定エンディング称号の獲得
  - 特定の判断を複数回選択

  例：

  - CI/CD導入：品質ゴール系称号を1回達成
  - 心理的安全性：脱退ゼロでプロジェクト完了

- **選択肢の質の向上**

  - 単純な二択が、より現実的な三択・折衷案に変化

繰り返しプレイすることで、チームは「強くなる」というより\*\*「賢く、壊れにくくなる」\*\*。

---

## 8. 目指す体験

- 正解をなぞるゲームではない
- 現実を単純化・誇張したフィクション
- IT技術者が「わかる……」と言いたくなる
- 判断の連続の中で、時折「うまく回った」と感じられる小さな成功体験がある
- チーム・ユーザー・ステークホルダーの状態は**数値ではなくテキストの温度感**で伝えられる
- 1回では終わらず、何度も試したくなる

## 9. 演出・UIコンセプト（案）

- **バーンダウンチャートUI**：
  スプリントの進捗をHPバーの代わりに常時表示。理想線との乖離がそのまま緊張感になる。

- **歪みの可視化**：
  技術的負債や疲労は「カード」や「タグ」として蓄積し、後のスプリントで行動を阻害する。

- **スプリント予報（リスク予兆）**：
  次スプリントで起こりやすい問題を、天気予報のように薄く示唆する。

  - 例：「次スプリント予報：負債の嵐に注意」「ステークホルダー介入の恐れ」
    これにより、失敗が運ゲーではなく「準備不足だった」という納得につながる。

- **ゾーン（Flow）演出**：
  チームが噛み合った瞬間を明確に褒める演出。

  **発動条件例：**

  - 3スプリント連続で歪みを増やさずにリリース成功
  - 全ロールの提案が連続して採用された

  効果：

  - 次スプリント中、選択肢効果が上昇／事故が起きにくくなる
    うまく回った瞬間を明確に褒める演出。条件を満たすと一時的にボーナスが発生する。

## 10. イベントシステム（割り込み・トラブル・外圧）

スプリント中に割り込みで発生し、計画や歪みに影響を与える。**実装上は1つのイベントシステム**として扱い、発生ロジックで2分類する。

### A. 状態依存イベント（歪みによって起きる）

状態（品質／疲労／ストレス／文化成熟度など）に応じて発生率が増減する。

1. **バグ発生（品質起因）**

   - 品質が低い・スコープが大きいほど発生しやすい
   - 対応：即修正／既知バグで出す／機能を切る（トグル）など

2. **病欠・不調（健康起因）**

   - 個人歪み（疲労・健康）が悪いほど発生しやすい
   - 対応：休ませる／代替で回す／無理をさせる（短期＋、長期−）

3. **脱退（ストレス起因）**

   - ストレス最大で退職イベント
   - 新人が補充されるが、該当ロールの成長が一部リセットされる

4. **インフラトラブル（運＋成熟度起因）**

   - 基本は運だが、「インフラ成熟度」や「運用文化」で軽減できる
   - 対応：即時復旧／原因調査／暫定回避

### B. 外部要因イベント（純ランダム／外圧）

歪みとは別に起こり得る。ただし頻度は低めにし、スプリント予報で予兆を出せる場合がある。

- **神の一声**：突然の仕様追加要求
- **依存ライブラリ問題**：外部要因による想定外の手戻り
- **災害級インシデント（超低確率）**：
  ごく稀に発生。残りスプリントの行動が大幅に制限される。
  「ほぼ理不尽」になりやすいので、救済（最低保証・報酬補正など）を前提に扱う

## 11. リプレイ性と“気持ちよさ”の設計（案）

- **正解の固定化を防ぐ要素**：

  - プロジェクト提示条件（期限・期待値・曖昧さ・圧）が毎回変わる
  - 状態依存イベントがプレイ内容により変化する
  - 中盤のゴール修正判断が展開を分岐させる

- **ポジティブな成功体験（カタルシス）**：

  - 「ゾーン（Flow）」：チームが噛み合った瞬間に一時的ボーナス
  - レビューでの“称賛フィードバック”が入る（まれだが強い）

- **エンディング称号**（例）：

  - 傷だらけの栄光／未完のユートピア／技術的破産 など



※本企画は実在の手法・組織を再現するものではなく、アジャイル開発をモチーフにしたフィクションである。
