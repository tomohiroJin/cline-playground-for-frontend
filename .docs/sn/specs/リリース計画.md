# スプリントノート（Sprint Note）リリース計画 v2.0

## 本ドキュメントの目的

ユーザーストーリー（v2.0）をもとに、E2E（End-to-End）で動作するインクリメントとしてMVPおよび後続リリースを定義する。各インクリメントは単体で「最初から最後までプレイ可能」であることを原則とし、段階的にゲーム体験を厚くしていく。

---

## 分割方針

### 分割の原則

1. **E2Eの原則**：どのインクリメントもプロジェクト開始→スプリント→終了の一連のフローが通る
2. **1仮説1検証の原則**：各インクリメントで検証する仮説は1つに絞る
3. **依存順の原則**：後続インクリメントが前段の基盤に乗る構成にする
4. **体験層の原則**：骨格→意味→因果→物語→深み→成長→磨きの順に層を重ねる

### インクリメント全体像

```
Inc.1  Loop        機械的なループが回る
  │
Inc.2  Conflict    判断に「迷い」が生まれる
  │
Inc.3  Cause       判断の結果が跳ね返ってくる
  │
Inc.4  Arc         5スプリントの物語が成立する
  │
Inc.5  Disruption  想定外が割り込み、適応を迫られる
  │
Inc.6  Growth      繰り返し遊ぶ理由がある
  │
Inc.7  Polish      人に勧められる完成度になる
```

## Inc.1：Loop — 機械的なループが回る

### 検証する仮説

> 計画→開発→リリース→レビュー→振り返りの5フェーズを繰り返すサイクルは、ゲームの骨格として機能するか？

### スコープ概要

3スプリントの短縮版。イベント・歪み・ロールの声はなし。純粋に「タスクを選び、リリースし、結果を見て、改善を選ぶ」という機械的なループだけを通す。

### 含めるストーリーと対応方針

| ID | ストーリー名 | 対応方針 |
|---|---|---|
| NARR-001 | プロジェクト提示 | 固定の1テンプレート。PMの台詞で概要を提示。 |
| NARR-002 | チーム結成 | 5人のロール名と一言紹介のみ。 |
| NARR-003 | プロジェクト終了演出 | 良い/普通/悪いの3パターンのテキスト。称号なし。 |
| GOAL-001 | ゴール選択 | 4種から選択可能。ただし評価への影響は最小限（リザルトテキストの分岐に使う程度）。 |
| GOAL-003 | 評価計算 | ゴールごとに1軸の簡易スコアリング。 |
| CORE-001 | スプリント実行 | 3スプリント。5フェーズ固定順。 |
| PLAN-001 | タスク選択 | 3候補から2つ選択。候補は固定プールからランダム。各タスクに簡単な説明文のみ。 |
| REL-001 | リリース判断 | 「出す」「一部削って出す」「延期」の3択。 |
| REV-001 | レビュー | ユーザー・ステークホルダーの反応を1文ずつ表示。バリエーション最小限。 |
| RETRO-001 | 振り返り | 2つの改善候補から1つ選択。効果は次スプリントに即反映。 |
| PARAM-002 | チームパラメータ | チーム信頼性の1パラメータのみ。リリース判断で増減。 |

### 含めないストーリー

ロールの声（VOICE系）、個人パラメータ（PARAM-001）、歪みシステム（DIST系）、イベント（EVT系）、ローグライク（GROW系）、演出UI（UI系）、オンボーディング（ONBD系）、メタ画面（META系）はすべて対象外。

### Inc.1完了時の姿

- プレイヤーはプロジェクトを受け、ゴールを選び、3スプリントを回して結末を見る
- 各スプリントで「何を作るか」「何をリリースするか」「何を改善するか」を判断する
- リザルトで結末テキストが表示される
- **1プレイ 5〜8分**
- まだ「面白い」は期待しない。**ループの流れが自然か、テンポは適切か**を確認する

### 検証すること

- 5フェーズの順序と遷移は自然か？
- 1スプリント2〜3回の判断（タスク選択・リリース判断・振り返り選択）でテンポは保てるか？
- フェーズ間のテキスト量は適切か？（多すぎて読み飛ばされないか、少なすぎて意味不明にならないか）
- リザルトのテキストに「なるほど」感があるか？

## Inc.2：Conflict — 判断に「迷い」が生まれる

### 検証する仮説

> ロールの「声」が加わることで、判断が「作業」から「迷い」に変わるか？

### スコープ概要

Inc.1のループに、ロールの「声」と三者（チーム・ユーザー・ステークホルダー）への影響差を追加する。ここで「対立する正義の選択」が初めて体験可能になる。歪みシステムはまだ入れない。

### Inc.1からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| VOICE-001 | ロールの声 | プランニング・リリース判断の2フェーズで、関連ロール2〜3人が「声」を出す。 |
| VOICE-003 | 三者への影響 | 選択肢ごとにチーム・ユーザー・ステークホルダーへの影響差を実装。テキストの温度感で間接表現。 |
| PARAM-001 | 個人パラメータ | 疲労の1パラメータのみ導入。タスクに関わったロールの疲労が増加する。 |
| PLAN-001 | タスク選択 | 各タスクに「どのロールが関わるか」「期待効果」「リスク」の情報を追加。 |
| REV-001 | レビュー | 三者への影響に基づく反応バリエーションの拡充。 |

### Inc.2完了時の姿

- プランニングで各ロールが「自分のタスクを優先すべき理由」を語り、プレイヤーが迷う
- リリース判断でも「出すべきか止めるべきか」にロールの声が割れる
- 選択の結果がユーザー・ステークホルダー・チームの反応としてそれぞれ返ってくる
- 「全員を満足させるのは無理」という感覚が芽生える
- **1プレイ 8〜12分**

### 検証すること

- ロールの声はプレイヤーの判断を実際に迷わせるか？
- 各ロールの主張は「どちらも正しい」と感じられるか？（一方が明らかに正解に見えないか）
- 三者への影響差はテキストの温度感だけで伝わるか？
- 5ロールのうち、存在意義が薄いロールはないか？（特にデザイナー）

## Inc.3：Cause — 判断の結果が跳ね返ってくる

### 検証する仮説

> 歪みの蓄積と状態依存イベントにより、「過去の判断が今の苦しさの原因だ」と感じられるか？

### スコープ概要

歪みシステム（個人・チーム）と状態依存イベント（バグ発生・病欠）を追加。判断の結果が数スプリント後に跳ね返ってくる因果関係を作る。

### Inc.2からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| PARAM-001 | 個人パラメータ | ストレスを追加（疲労+ストレスの2パラメータ）。意見を無視され続けるとストレス増加。 |
| DIST-001 | 個人歪み蓄積 | 疲労・ストレスの蓄積→ロールの声の弱体化。段階的影響（軽度・中度まで。重度・最大はInc.5）。 |
| DIST-002 | チーム歪み蓄積 | 技術的負債の1種類。品質を犠牲にした判断で蓄積→タスク実行コスト増加。 |
| VOICE-002 | 声の状態変化 | 疲労で消極化、ストレスで攻撃化する声のトーン変化を実装。消失はまだなし。 |
| EVT-001 | 状態依存イベント | バグ発生（品質起因）と病欠（疲労起因）の2種類。各2〜3択の対応。 |
| DEV-001 | 開発中イベント対応 | インクリメントフェーズ中にイベントが割り込む処理。イベントなしのスプリントもあり得る。 |
| RETRO-001 | 振り返り | 歪み緩和に寄与するアクションを追加。候補を3つに拡張。 |

### Inc.3完了時の姿

- 品質を削ればバグが増え、人を酷使すれば病欠が出る
- 技術的負債が溜まると同じタスクに余計な時間がかかる
- 疲弊したメンバーの声が弱々しくなり、「休ませてあげたい」と感じる
- 振り返りで歪みに対抗する手段を選べる
- 「あの時無理させたからこうなった」という因果の実感がある
- **1プレイ 10〜15分**

### 検証すること

- 歪みの因果関係はプレイヤーに伝わるか？（「自分のせいだ」と感じるか）
- 歪みの蓄積速度は適切か？（3スプリントで効果を体感できるか、逆に早すぎて詰まないか）
- イベントの発生頻度は適切か？（毎回起きると煩わしい、全く起きないと意味がない）
- 振り返りの改善アクションは「対処手段がある」安心感を与えるか？

## Inc.4：Arc — 5スプリントの物語が成立する

### 検証する仮説

> 3スプリントから5スプリントに拡張し、中盤のゴール修正やスプリントごとのナラティブが加わることで、1プレイが「物語」として感じられるか？

### スコープ概要

スプリント数を3→5に拡張。ゴール修正判断、スプリントごとのナラティブ、難度上昇カーブを実装。リファイメント選択とスプリント予報も追加し、「先を見据えた判断」の体験を作る。

### Inc.3からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| CORE-001 | スプリント実行 | 3→5スプリントに拡張。1プレイ15〜25分の本来想定。 |
| CORE-002 | 難度上昇 | 歪み蓄積に応じたタスクコスト増加・イベント発生率変動。後半に向けた緊張感の上昇。 |
| NARR-004 | スプリントの物語 | 各スプリント開始時のナレーション。Sprint位置によるトーン変化（期待→現実→正念場→追い込み→収束）。 |
| GOAL-002 | ゴール修正 | Sprint 3終了後のゴール修正判断。変更コスト（信頼低下・タスク枠減・歪み増）の実装。 |
| PLAN-002 | リファイメント選択 | プランニングでリファイメントに時間を割く選択肢。タスク枠-1の代わりに次Sprint情報向上。 |
| EVT-003 | スプリント予報 | 振り返り後に次Sprintの予報を表示。リファイメント実施で精度向上。 |
| PLAN-001 | タスク選択 | タスク候補をプロジェクト状態・歪み・前Sprint結果で動的生成に拡張。 |
| RETRO-001 | 振り返り | 即効型・遅効型の両方を実装。Sprint 5の振り返りをナレーション演出に変更。 |
| NARR-003 | 終了演出 | テキストバリエーション拡充。判断ダイジェスト表示を追加。 |
| REV-001 | レビュー | 称賛フィードバック（低確率）を追加。リリース延期時のデモなし演出。 |
| VOICE-001 | ロールの声 | イベント対応フェーズでもロールが発言するよう拡張（計3フェーズ）。 |

### Inc.4完了時の姿

- 5スプリントの起承転結が「物語」として感じられる
- Sprint 1の期待感からSprint 5の収束まで、空気感が変化する
- 中盤でゴールを変えるか維持するかのドラマチックな判断がある
- スプリント予報で「次に何が来そうか」を見て備える戦略性がある
- リファイメントに投資するか否かの長期/短期トレードオフがある
- **1プレイ 15〜25分**（本来想定）

### 検証すること

- 5スプリントで中だるみしないか？（特にSprint 2〜3）
- ゴール修正判断は面白い分岐か？（「変えても変えなくても一理ある」になっているか）
- スプリント予報は判断材料として機能するか？（見て行動を変えるか）
- リファイメントの「今を犠牲にして未来に投資する」判断は体感できるか？
- 遅効型の振り返りアクションは効果を実感できるか？
- 5スプリント通しての難度カーブは自然か？

## Inc.5：Disruption — 想定外が割り込み、適応を迫られる

### 検証する仮説

> 外部イベント、脱退、フルスペックのパラメータが加わることで、「計画通りにいかないが、適応する面白さ」が生まれるか？

### スコープ概要

外部要因イベント、メンバー脱退・補充、個人パラメータのフルセット化、持続可能性メカニクスを追加。ゲームシステムの完全版。

### Inc.4からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| EVT-002 | 外部要因イベント | 神の一声・依存ライブラリ問題・災害級インシデントの実装。 |
| EVT-004 | 脱退と補充 | ストレス最大で脱退発生。新人補充、成長リセット、声の一時不在。 |
| PARAM-001 | 個人パラメータ | スキル成熟度・健康状態を追加（4パラメータのフルセット）。 |
| PARAM-002 | チームパラメータ | 開発文化成熟度・持続可能性を追加（3パラメータのフルセット）。 |
| DIST-001 | 個人歪み蓄積 | 重度（声の消失）・最大（脱退トリガー）の段階を追加。フル実装。 |
| DIST-002 | チーム歪み蓄積 | 信頼負債・文化負債を追加（3種類のフルセット）。 |
| DIST-003 | 持続可能性と疲労 | 持続可能性による波及抑制/連鎖崩壊ロジックの実装。 |
| GOAL-003 | 評価計算 | 各ゴール本来の複数評価軸。ゴール変更時の引き継ぎ計算。 |
| DEV-001 | イベント対応 | 複数イベント同時発生（最大2つ）への対応。 |
| EVT-001 | 状態依存イベント | インフラトラブルを追加（3種類+脱退で4種のフルセット）。 |

### Inc.5完了時の姿

- 突然の仕様追加に「受けるか断るか」を迫られる
- メンバーが限界を超えると脱退し、チームに穴が空く
- 持続可能性が高いチームは1人欠けても回るが、低いチームは連鎖崩壊する
- 「仕組みで支える」か「個人の頑張りに頼る」かの構造的差異が体感できる
- 歪みの全段階（軽度→最大）とイベントの全種類が揃い、ゲームシステムが完成する
- **1プレイ 15〜25分**（ゲームシステム完全版）

### 検証すること

- 外部イベントは「理不尽」ではなく「適応の面白さ」として受け取られるか？
- 脱退イベントのインパクトは適切か？（痛いが詰みではないか）
- 持続可能性の高低でプレイ体験に明確な差が出るか？
- パラメータ・歪み・イベントのフルセットでバランスは崩壊していないか？
- 災害級インシデントの救済措置は納得感があるか？

## Inc.6：Growth — 繰り返し遊ぶ理由がある

### 検証する仮説

> ローグライク成長・称号・条件ランダム化により、「もう1回やりたい」と思えるか？

### スコープ概要

ローグライク成長（個人成長・文化アンロック・選択肢の質向上）、エンディング称号、プロジェクト条件のランダム化、メタ画面を実装。プレイ間の蓄積と変化を作る。

### Inc.5からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| GROW-001 | 個人成長引き継ぎ | ロール別の弱点緩和・歪み耐性向上。プレイ内容に応じた成長度計算。 |
| GROW-002 | 文化アンロック | 5種類以上の文化アンロック。解放条件・効果の実装。 |
| GROW-003 | 選択肢の質向上 | 成長に応じた第三の選択肢（折衷案）の出現。 |
| GROW-004 | 成長永続化 | 成長データの永続保存・リセット機能。 |
| UI-004 | エンディング称号 | 最低15種の称号。決定ロジック。コレクション機能。文化アンロック条件との連携。 |
| RPL-001 | 提示条件の変化 | プロジェクト条件4軸のランダム化。メタ成長に応じた難度調整。 |
| RPL-002 | イベント変化 | プレイスタイルに応じたイベント傾向変化。 |
| NARR-001 | プロジェクト提示 | 固定→ランダム生成に拡張。4軸をPM台詞で表現。 |
| NARR-002 | チーム結成 | 成長済みメンバー描写。繰り返し感の軽減。 |
| META-001 | メタ画面 | チーム・文化・称号・履歴のメタ画面。ゲーム開始導線。 |
| META-002 | 状態確認パネル | プレイ中に開けるチーム状態パネル。 |

### Inc.6完了時の姿

- プレイを重ねるとチームが「賢く、壊れにくく」なる実感がある
- 文化アンロックで以前は不可能だった選択肢や緩和手段が解放される
- プロジェクト条件が毎回変わり、同じ戦略が通用しない
- エンディング称号をコレクションするモチベーションがある
- メタ画面で蓄積を眺める楽しさがある
- **ゲームとしての内容面の完成形**

### 検証すること

- 2回目・3回目のプレイ動機は十分か？
- 文化アンロック条件は達成感を生むか？
- 成長でゲームが簡単になりすぎていないか？
- 条件ランダム化はプレイ変化として十分機能するか？
- 第三の選択肢（折衷案）はゲーム体験を豊かにしているか？

## Inc.7：Polish — 人に勧められる完成度になる

> 演出・UI・オンボーディングの磨き込みで、「人に勧めたい」プロダクトになるか？

### スコープ概要

演出系UI、オンボーディング、テキストバリエーションの大量追加。体験の完成度を引き上げる。

### Inc.6からの差分

| ID | ストーリー名 | 対応内容 |
|---|---|---|
| ONBD-001 | 初回ガイダンス | 初回プレイ時のガイドテキスト。キャラクター台詞による自然な導入。 |

| UI-001 | バーンダウンチャート | スプリント進捗のリアルタイム可視化。理想線と実績線。 |
| UI-002 | 歪みカード可視化 | 個人タグ・チームカードの視覚表示。増減アニメーション。 |
| UI-003 | ゾーン演出 | 発動条件判定・ボーナス効果・テキスト/ビジュアル演出。 |

### Inc.7完了時の姿

- 初めてのプレイヤーがスムーズにゲームに入れる
- 歪みの蓄積が視覚的「圧迫感」として伝わる
- バーンダウンチャートが固有の緊張感を生む
- ゾーン演出で「うまく回った」瞬間が褒められる
- テキストの被りが少なく、繰り返しプレイでも新鮮さがある
- **リリース可能なプロダクト**

### 検証すること

- IT未経験者でもオンボーディングでルールを理解できるか？

## リスクと注意点

### Inc.1〜3 共通リスク

- **3スプリントの制約**: Inc.1〜3は3スプリントで検証する。3スプリントでは体感しにくい要素（遅効型の改善、長期的な歪み蓄積）はInc.4以降に意図的に先送りしている。3スプリントの検証結果だけでゲーム全体の面白さを判断しないよう注意。

### Inc.4 のリスク

- **3→5スプリント拡張時のバランス崩壊**: Inc.3までの3スプリントで調整したパラメータが、5スプリントでは破綻する可能性がある。拡張時にバランスの再調整時間を見込む。
- **中だるみ**: Sprint 2〜3の緊張感維持がNARR-004（物語）とGOAL-002（ゴール修正）に依存する。テキストの質が低いと機能しない。

### Inc.5 のリスク

- **システムの複雑化**: パラメータ・歪み・イベントがフルセットになることで、プレイヤーが「何が起きているかわからない」状態になるリスク。META-002（状態確認パネル）をInc.6ではなくInc.5に前倒すことを検討してもよい。

### Inc.6 のリスク

- **成長によるゲーム崩壊**: 個人成長と文化アンロックでゲームが簡単になりすぎる可能性。RPL-001（難度調整）との連動バランスが鍵。
- **アンロック条件の到達不能**: 全アンロックの到達可能性をプレイテストで確認する。

### Inc.7 のリスク

- **演出過多**: バーンダウンチャート・歪みカード・ゾーン演出・予報が全て画面に乗ると情報過多の可能性。UIの情報設計を慎重に行う。
- **テキスト量**: バリエーション追加は際限がない。「繰り返しプレイで被りが気にならない最低ライン」を先に定義する。

### 全体リスク

- **テキスト作成の工数**: 全インクリメント共通で最大のリスク。PM台詞・ロールの声・レビュー反応・ナレーション・リザルトなど、テキスト量がゲーム体験の質を直接左右する。各Inc開始時にテキスト作成の工数を必ず見積もる。
