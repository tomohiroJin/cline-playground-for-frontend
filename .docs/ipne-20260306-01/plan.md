# IPNE ブラッシュアップ計画

## 目標

**ユーザーに「面白い」と思ってもらえるゲーム体験を実現する。**

具体的には以下の4つの柱で改善を行う：

1. **画面サイズの最適化** — レスポンシブ化で大画面を活かす
2. **キャラクターのカッコよさ向上** — 32x32スプライトで描き込み量を大幅増加
3. **戦闘演出のエスカレーション** — 強くなるにつれて派手になる演出
4. **ゲーム体験全体の質の向上** — 爽快感・没入感・リプレイ意欲

---

## 現状分析

### 良い点（維持すべき要素）

| 要素 | 評価 |
|------|------|
| ゲームシステム | 5ステージ制、職業選択、成長選択と基盤は十分 |
| ストーリー | SF的世界観、ステージ間の物語展開が完成している |
| 音声 | 22種SE + 4BGMパターンが実装済み |
| アーキテクチャ | DDD風レイヤード構成で拡張性が高い |
| エフェクト | パーティクルシステム（8種）、画面シェイク、低HPビネット |

### 改善が必要な点

| 課題 | 詳細 | 優先度 |
|------|------|--------|
| 画面が小さい | Canvas 固定720x528px。フルHDで画面の37%しか使わない | **最高** |
| ホームボタンとHUDが重なる | 左上のFloatingHomeButtonとHPバーが同じ位置 | **最高** |
| キャラクターが地味 | 16x16スプライトが小さく情報量が少ない | **高** |
| 攻撃エフェクトが一律 | Lv1でもLv20でも同じパーティクル数・色・範囲 | **高** |
| 爽快感の不足 | 雑魚敵撃破時の反応が薄い、連続撃破の達成感がない | **高** |
| ボス戦の盛り上がり不足 | 通常敵とエフェクトの差が小さい | **中** |
| BGMが全ステージ同一 | 進行による雰囲気変化が音楽に反映されない | **中** |
| ダメージ表現が視認しにくい | 数値表示がなく、敵に与えたダメージが分かりにくい | **中** |
| レベルアップの体感不足 | 能力は上がるが、視覚的な成長実感が薄い | **高** |

## フェーズ構成

### Phase 0A: レスポンシブCanvas化 + HUDレイアウト再設計

**目的**: 画面を大きく使い、HUDの視認性と操作性を改善する

#### 現状の問題

```
現在:
  Canvas = 720×528px（固定）
  tileSize = 48px（固定）
  padding-top = 80px（ヘッダー非表示なのに空白）
  FloatingHomeButton: top:12px, left:12px
  HPバー: top:1rem, left:1rem → 重なる
```

#### 0A-1. レスポンシブCanvas

タイルサイズを画面サイズに応じて動的に計算する。表示タイル数（15×11）は固定。

```
Canvas幅  = min(利用可能幅, tilesX × maxTileSize)
Canvas高さ = min(利用可能高さ, tilesY × maxTileSize)
tileSize  = min(Canvas幅 / tilesX, Canvas高さ / tilesY)
```

**デバイス別の予測値（表示タイル数 15×11 固定）**:

| デバイス | 画面 | 利用可能領域 | tileSize | Canvas | スプライト倍率 |
|---------|------|------------|---------|--------|-------------|
| フルHD PC | 1920×1080 | 1880×1020 | 92px | 1380×1012 | 2.88x |
| WQHD PC | 2560×1440 | 2520×1380 | 125px | 1875×1375 | 3.91x |
| MacBook Air | 1440×900 | 1400×840 | 76px | 1140×836 | 2.38x |
| iPad | 1024×768 | 984×708 | 64px | 960×704 | 2.00x |
| iPhone 14 | 390×844 | 370×780 | 24px | 360×264 | 0.75x |

> **最小tileSize**: 32px（これ以下ではモバイルUIに切り替え）
> **最大tileSize**: 128px（大画面での過度な拡大を防止）

#### 0A-2. HUDレイアウト再設計

ホームボタンとの干渉を解消し、情報を整理する。

```
変更前:
┌─[🏠]────────────────────────────┐
│ [HP ████████░░]  [Lv.15]        │
│                  [⚔️3 🏹2 🏃5] │
│          [05:23]                 │
│                                  │
│           (ゲーム画面)           │
│                                  │
│ [🔑]                  [🗺️] [❓] │
└──────────────────────────────────┘

変更後:
┌─[🏠]──────[STAGE 3]──────[Lv.15]─┐
│              [05:23]    [⚔️3 🏹2] │
│ [HP ████████░░░░]                  │
│                                    │
│            (ゲーム画面)            │
│                                    │
│                                    │
│ [🔑]                    [🗺️] [❓] │
└────────────────────────────────────┘
```

**変更点**:
- HPバーをホームボタンの**下**に移動（y座標を下げる）
- ステージ表示を上部中央に移動
- タイマーをステージ表示の直下に配置
- レベル・ステータスは右上を維持
- `padding-top: 80px` を `padding-top: 0` に変更（ヘッダー非表示時）

#### 0A-3. viewport.ts のリファクタリング

```typescript
// 変更前: 固定値
export const VIEWPORT_CONFIG = {
  tilesX: 15,
  tilesY: 11,
  tileSize: 48,  // ← 固定
} as const;

// 変更後: タイル数のみ固定、tileSize は動的計算
export const VIEWPORT_CONFIG = {
  tilesX: 15,
  tilesY: 11,
  minTileSize: 32,
  maxTileSize: 128,
} as const;

export function calculateTileSize(
  availableWidth: number,
  availableHeight: number
): number;

export function getCanvasSize(tileSize: number): {
  width: number;
  height: number;
};
```

#### 0A-4. リサイズ対応

- `window.resize` イベントでタイルサイズを再計算
- `ResizeObserver` でコンテナサイズの変化を検知
- デバウンス処理（200ms）でパフォーマンスを確保
- リサイズ時にスプライトキャッシュをクリア（`SpriteRenderer.clearCache()`）

### Phase 0B: スプライト解像度アップ（16x16 → 32x32）

**目的**: ドット絵の情報量を4倍にし、装飾・表情・武器のディテールを描き込む

#### 0B-1. サイズ変更マトリックス

| スプライト | 現在 | 変更後 | 比率(対base) |
|-----------|------|--------|-------------|
| 通常キャラ・敵 (base) | 16 | **32** | 1.00x |
| アイテム (item) | 8 | **16** | 0.50x |
| ミニボス (miniBoss) | 20 | **40** | 1.25x |
| ボス (boss) | 24 | **48** | 1.50x |
| メガボス (megaBoss) | 28 | **56** | 1.75x |

> すべて8の倍数。ドット絵として扱いやすく、スケーリング時の品質も良い。

#### 0B-2. パレット拡張（8色→12色）

32x32 の広いキャンバスを活かし、影・装飾・ディテール用の色を4色追加。

| 2 | 暗部 | #4c51bf | #6d28d9 |
| 3 | メインボディ | #667eea | #a78bfa |
| 4 | ハイライト | #818cf8 | #c4b5fd |
| 5 | 最明部 | #c7d2fe | #ddd6fe |
| 6 | 白・武器刃 | #f5f5f5 | #f5f5f5 |
| 7 | 肌 | #d4a574 | #d4a574 |

| 9 | 肌影 (新) | #b8845a | #b8845a |

| 11 | ベルト・靴 (新) | #4a3728 | #4a3728 |

#### 0B-3. 32x32 で追加できるディテール

**戦士**:
- 兜: 角飾り・バイザーの形状、兜の装飾ライン
- 顔: 目のハイライト（パレット10）、口元の影
- 胴: 鎧のリベット、胸板の立体感、肩パッドの段差
- 腕: 篭手のディテール、シールドの紋章
- 武器: 剣の鍔・柄・刃の反射光
- 脚: レッグアーマーの関節部分、ブーツの厚み
- マント: 背面で翻るマントの質感

**盗賊**:
- フード: 尖った形状、影に隠れた顔、目だけが光る
- 胴: クロークの重なり、ベルトのポーチ
- 腕: 素手の手首ガード、ダガーの柄の巻き
- 武器: ダガーの反り、投げナイフのホルスター
- 脚: 軽装ブーツ、足音を立てない柔らかい形状
- クローク: 左右非対称のたなびき

### Phase 1: キャラクタービジュアル強化（見た目を変える）

**目的**: プレイヤーキャラクターの存在感とカッコよさを向上させる

#### 1-1. パワーオーラシステム

レベルに応じてキャラクターの周囲にオーラエフェクトを常時表示する。

| レベル帯 | オーラ | 色 | 演出 |
|---------|--------|-----|------|
| Lv1-4 | なし | - | 初期状態 |
| Lv5-9 | 微発光 | 淡い白 | キャラ周囲に微かな発光（1px） |
| Lv10-14 | 小オーラ | 職業色（戦士:青/盗賊:紫） | 脈動する発光リング（2px） |
| Lv15-19 | 中オーラ | 職業色+金 | パーティクル付き発光（3px） |
| Lv20+ | 大オーラ | 金+白 | 炎のような揺らぎエフェクト |

#### 1-2. 武器エフェクト強化

攻撃力に応じて武器の見た目が変化する。

| 攻撃力 | 武器演出 |
|--------|---------|
| 1-3 | 通常（現状維持） |
| 4-6 | 武器に淡い光の軌跡 |
| 7-9 | 武器が発光、攻撃時に弧を描く光跡 |
| 10+ | 武器が強く発光、攻撃時に衝撃波エフェクト |

#### 1-3. ステージ進行による見た目の変化

各ステージクリア報酬で選んだ能力に応じて、視覚的なアクセントを追加する。

- 最大HP強化 → キャラクター外枠に防御的な輝き
- 攻撃力強化 → 武器の輝き強化
- 移動速度強化 → 足元に残像エフェクト
- 攻撃速度強化 → 手元に回転エフェクト
- 回復量強化 → 定期的な緑の回復パーティクル

### Phase 2: 戦闘演出エスカレーション（強くなるにつれ派手に）

**目的**: プレイヤーの成長を戦闘演出で実感させる

#### 2-1. ダメージ数値表示（フローティングダメージ）

攻撃ヒット時に与えたダメージ数値をポップアップ表示する。

| 要素 | 仕様 |
|------|------|
| 表示位置 | 敵の頭上 |
| 動き | 上方向にフロート（0.8秒で消滅） |
| フォント | bold, ドット風（Canvas描画） |
| 色 | 白（通常）、金（クリティカル）、赤（敵→プレイヤー） |
| サイズ | ダメージ量に応じて変化（基本12px、高ダメージで最大20px） |

#### 2-2. 攻撃ヒットエフェクトのスケーリング

プレイヤーのレベル/攻撃力に応じて、攻撃ヒットエフェクトを段階的に強化する。

| レベル帯 | パーティクル数 | サイズ | 追加演出 |
|---------|-------------|--------|---------|
| Lv1-4 | 4個 | 小 | なし |
| Lv5-9 | 8個 | 中 | 衝撃波リング |
| Lv10-14 | 12個 | 大 | 衝撃波+画面フラッシュ |
| Lv15-19 | 16個 | 特大 | 衝撃波+フラッシュ+残光 |
| Lv20+ | 24個 | 極大 | フル演出+画面シェイク |

#### 2-3. コンボシステム

短時間に連続で敵を撃破すると、コンボカウンターが表示され、演出が強化される。

| 要素 | 仕様 |
|------|------|
| コンボ時間窓 | 3秒（最後の撃破から次の撃破まで） |
| 表示位置 | 画面上部中央 |
| 表示 | 「x2」「x3」...「x10+」 |
| 演出強化 | コンボ数に応じてヒットエフェクトが追加で1.5倍増幅 |
| 最大コンボ | 表示上限なし |
| コンボSE | コンボ数に応じてピッチが上がる撃破SE |

#### 2-4. 敵撃破演出の強化

雑魚敵撃破時の爽快感を向上させる。

| 要素 | 現状 | 改善後 |
|------|------|--------|
| 撃破パーティクル | なし（即消滅） | 敵色の破片パーティクル（6-10個） |
| 撃破SE | 共通SE | 変更なし（既に22種あるため十分） |
| 画面効果 | なし | 高レベル撃破時に微画面シェイク |
| 敵消滅アニメ | 即消滅 | 縮小→破裂→パーティクル拡散（300ms） |

#### 2-5. ボス戦演出強化

ボス戦をより特別な体験にする。

| 要素 | 仕様 |
|------|------|
| ボス登場演出 | ボスに初めて接近時、画面が暗転+「WARNING」テキスト（1秒） |
| ボスHP残量演出 | HP50%以下で赤いオーラ、25%以下で激しい点滅 |
| ボス攻撃演出 | 攻撃前の溜めモーション（300ms）+ 衝撃波パーティクル |
| ボス撃破演出 | 大規模パーティクル（48個）+ 1秒間スローモーション + 画面フラッシュ |
| メガボス追加 | 上記に加え、多段階パーティクル爆発（3波） |

### Phase 3: ゲーム体験の全体的な向上

**目的**: ゲーム全体の没入感と満足度を高める

#### 3-1. ステージ別BGM

各ステージに固有のBGMパターンを割り当て、進行に応じた雰囲気変化を実現する。

| ステージ | BGMトーン | 波形 | テンポ |
|---------|----------|------|--------|
| 1 | 探索的・穏やか | triangle | 中速 |
| 2 | やや緊張・神秘 | triangle+sine | 中速 |
| 3 | 緊張・不安 | sawtooth+triangle | やや速い |
| 4 | 重厚・威圧 | sawtooth | やや遅い |
| 5 | 激しい・クライマックス | square+sawtooth | 速い |

ボス戦突入時にBGMが専用パターンに変更される。

#### 3-2. レベルアップ演出強化

| 要素 | 仕様 |
|------|------|
| 全画面フラッシュ | 金色、200ms |
| パーティクル | 螺旋上昇パーティクル（24個、金+白） |
| テキスト | 「LEVEL UP!」+ 「Lv.X」の表示（1.5秒） |
| SE | 既存のレベルアップSEを使用 |

#### 3-3. 探索報酬の視覚的フィードバック

| イベント | 追加演出 |
|---------|---------|
| HP回復アイテム | 緑のパーティクル上昇 + HPバーのフラッシュ |
| レベルアップアイテム | 金のパーティクル + 画面フラッシュ |
| 鍵の取得 | 金の螺旋パーティクル + 「KEY GET!」テキスト |
| マップ開示 | マップ全体がフラッシュ |

#### 3-4. 画面遷移演出の改善

| 遷移 | 演出 |
|------|------|
| ステージ開始 | 暗転からフェードイン + ステージ番号テキスト |
| ゲームオーバー | スローモーション（0.5秒）→ 暗転 → ゲームオーバー画面 |
| ステージクリア | 画面フラッシュ → パーティクル → リザルト表示 |

## 技術的アプローチ

### 既存アーキテクチャへの統合方針

1. **レスポンシブCanvas** → `viewport.ts` を固定値から動的計算に変更、`useCanvasSize` フック新規作成
2. **HUDレイアウト** → `IpnePage.styles.ts` の配置変更、`Game.tsx` のCanvas内HUD描画位置調整
3. **32x32スプライト** → `config.ts` の値変更 + 全スプライトファイルの再設計
4. **エフェクトスケーリング** → `effectManager.ts` の `addEffect` に `powerLevel` パラメータ追加
5. **コンボシステム** → 新規ファイル `combo.ts` をドメイン層に追加
6. **ステージBGM** → `bgm.ts` に5パターン追加

| `particleSystem.ts` | スケーリング対応パーティクル生成 |
| `combat.ts` | ダメージ値の返却、コンボトリガー |
| `bgm.ts` | ステージ別BGMパターン |
| `useGameLoop.ts` | エフェクトトリガーの拡張 |
| `useGameState.ts` | ステージBGM切り替え |

| 新規: `combo.ts` | コンボシステムロジック |
| 新規: `aura.ts` | オーラ描画ロジック |
| 新規: `floatingText.ts` | フローティングテキスト描画 |

### 後方互換性

- 既存の `EffectType` に新値を追加（既存値は変更しない）
- `addEffect` の新パラメータはすべてオプショナル
- BGMの切り替えロジックは既存の `playGameBgm()` を拡張
- テストは追加のみ（既存テストの変更は最小限）

## リスクと対策

| リスク | 対策 |
|--------|------|
| レスポンシブ化でのレイアウト崩れ | minTileSize/maxTileSize で範囲制限、段階的テスト |
| 大画面でのパフォーマンス低下 | Canvas描画領域が増えるが、ビューポート内タイル数は固定のため影響は限定的 |
| 32x32スプライトの作成工数 | 最重要スプライト（プレイヤー下向き）から着手し、段階的に完成させる |
| スプライトキャッシュのメモリ増加 | リサイズ時のキャッシュクリア、不要エントリの定期削除 |
| パーティクルの相対サイズ変化 | tileSize に対する比率で指定（絶対px値ではなく） |
| 演出が過剰で視認性低下 | エフェクト透明度の適切な調整、重要情報はHUDで確保 |
| コンボシステムによるバランス崩壊 | コンボは演出のみ（ダメージ倍率に影響しない） |
