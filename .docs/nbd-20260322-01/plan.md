# Non-Brake Descent リファクタリング計画

## 1. 現状分析

### 1.1 プロジェクト概要

Non-Brake Descent は React + TypeScript + SVG で実装されたハイスピード・ランニングアクションゲームです。
ブレーキなしで100個のランプを高速下降し、障害物を回避しながらハイスコアを目指します。
DOM ベース（SVG + styled-components）でレンダリングしており、Canvas は未使用です。

### 1.3 検出された問題点

#### アーキテクチャ上の問題

1. **`NonBrakeDescentGame.tsx` の肥大化（632行）**: 状態管理（25個以上の useState）、ゲームループ、入力処理、衝突判定コールバック、UI レンダリングが1コンポーネントに集約
2. **ゲームループ内の副作用**: `setInterval` 内で多数の `setState` が連鎖的に呼ばれ、レンダリング効率が低い
3. **衝突ハンドラのミューテーション**: `createCollisionHandlers` 内で `obs.t = ObstacleType.DEAD` のように引数を直接変更している
4. **`CollisionHandlerResult` の型**: `boolean | 'slow'` という不明確な型が使用されている
5. **ドメイン層の不完全な分離**: `domains/` に一部分離されているが、ゲームループ内のロジック（遷移処理、スコア加算、コンボ判定）がコンポーネントに残存

#### 設計原則違反

6. **SRP 違反**: `NonBrakeDescentGame.tsx` が状態管理・ゲームループ・入力処理・描画の全責務を持つ
7. **OCP 違反**: 新しい障害物タイプを追加する場合、`createCollisionHandlers` の switch 的マップを直接変更する必要がある
8. **DRY 違反**: `addParticles` + `addScorePopup` のパターンが衝突コールバック内で繰り返し出現
9. **DbC 不在**: `physics.ts` のパラメータチェックがない（`math-utils.ts` には DbC あり）
10. **マジックナンバー散在**: `obs.t = ObstacleType.TAKEN`（副作用）、`-2`（敵キル減速値）、`0.4`（バウンス倍率）等

#### テストの問題

11. **テストカバレッジの偏り**: ドメインテスト（71行）のみ。`physics.ts`、`entities.ts`、`generators.ts`、`audio.ts`、`particles.ts` のテストが皆無
12. **テスト構造の非統一**: AAA パターンが明示されていない
13. **統合テストの不足**: ゲームループ全体の動作、画面遷移のテストがない
14. **レンダラーテストの不足**: 各レンダラーコンポーネントのテストがない

#### コンポーネントの問題

15. **インラインスタイルの多用**: メインコンポーネントで styled-components と混在
16. **画面コンポーネントの未分離**: タイトル画面・プレイ画面・リザルト画面が1つのコンポーネントに混在
17. **`Audio` モジュールのグローバル参照**: IIFE でシングルトン化されており、テスタビリティが低い

## 2. リファクタリング方針

### 2.1 レイヤードアーキテクチャの導入

```
domain/           # ドメイン層（純粋なビジネスロジック、外部依存なし）
  entities/       # エンティティ（値オブジェクト含む）
  services/       # ドメインサービス（衝突判定、スコア計算等）
  events/         # ドメインイベント定義

application/      # アプリケーション層（ユースケース・オーケストレーション）
  game-loop/      # ゲームループ処理
  collision/      # 衝突処理オーケストレーション

infrastructure/   # インフラ層（外部依存）
  audio/          # オーディオシステム
  storage/        # localStorage 操作

presentation/     # プレゼンテーション層（React コンポーネント）
  screens/        # 画面コンポーネント（Title, Play, Result）
  components/     # 共通UIコンポーネント（既存 renderers を移行）
  hooks/          # React フック
```

### 2.2 デザインパターンの適用

| パターン | 適用箇所 | 目的 |
|---------|---------|------|
| Strategy | 衝突ハンドラ（障害物タイプ別） | OCP 準拠、switch 的マップの排除 |
| Factory Method | エンティティ生成（既存を強化） | 生成ロジックの型安全性強化 |
| Observer/Event | オーディオ・パーティクルイベント | 副作用の分離、ゲームループの純粋化 |
| State | ゲーム画面遷移 | 画面状態管理の明確化 |
| Command | 入力処理 | 入力デバイスの抽象化 |

### 2.3 E2E テストの判断

**結論: E2E テストを導入する**

理由:
- SVG ベースの DOM レンダリングのため、Playwright で要素の検出・操作が可能
- ゲーム状態遷移（TITLE → COUNTDOWN → PLAY → CLEAR/OVER）の検証に適している
- キーボード入力のシミュレーションが容易
- ランダム要素は `Math.random` のシードで制御可能

スコープ:
- 画面遷移フロー（タイトル → カウントダウン → プレイ開始）
- 基本的なキーボード操作の応答確認
- スコア表示・UI 要素の表示確認

## 3. Phase 構成

### Phase 1: 基盤整備（テスト強化 + テストインフラ整備）
既存の動作を壊さずにテストを拡充し、安全にリファクタリングを進める土台を作る。
テストヘルパー・ファクトリの作成、不足テストの追加、テスト構造の統一。

### Phase 2: ドメイン層の強化
既存の `domains/` を `domain/` に再編成し、純粋関数化とドメイン境界を確立する。
衝突ハンドラのロジック抽出、ドメインイベントの定義。

### Phase 3: アプリケーション層の抽出
`NonBrakeDescentGame.tsx` からゲームループ・衝突処理のオーケストレーションを分離。
ゲーム状態をイミュータブルな構造に統合。

### Phase 4: インフラ層の分離
オーディオ・ストレージをインターフェースで抽象化し、テスタビリティを向上させる。

### Phase 5: プレゼンテーション層のリファクタリング
メインコンポーネントの分割、画面コンポーネントの整理、フックの責務分割。

### Phase 6: E2E テスト追加とテストリファクタリング・最終確認
E2E テストの追加、テストコードのリファクタリング、カバレッジ確認。

## 4. リスク管理

| リスク | 対策 |
|-------|------|
| リファクタリング中のデグレ | 各 Phase で既存テストが全パスすることを確認 |
| ゲームバランスの崩壊 | ロジック変更時は定数・計算式を変えない |
| パフォーマンス劣化 | Phase 完了ごとに 60fps が維持されているか確認 |
| 過度な抽象化 | ゲーム特化のシンプルな抽象化にとどめる |
| ゲームループの破壊 | useReducer 移行前に既存挙動のテストを十分に書く |

## 5. 成功基準

- すべての既存テストがパスすること
- 新規テストカバレッジ: ドメインロジック 90% 以上
- `NonBrakeDescentGame.tsx` を 150行以内に縮小
- 新しい障害物タイプの追加が既存ファイルの変更なしに可能（OCP 準拠）
- E2E テストで画面遷移フローが検証可能
- Lint エラーがゼロであること
