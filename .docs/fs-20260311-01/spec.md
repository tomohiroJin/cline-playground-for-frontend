# Falldown Shooter ブラッシュアップ仕様書

## 1. レスポンシブ対応

### 1.1 対応デバイス・画面サイズ

| デバイス | 画面幅 | セルサイズ | 備考 |
|---------|--------|----------|------|
| スマートフォン縦 | ~480px | 幅・高さ両方から動的計算 | メインターゲット、画面に収まる最大サイズ |
| スマートフォン横 | ~896px | 高さベースで動的計算 | 横レイアウト対応 |
| タブレット | ~1024px | 幅・高さ両方から動的計算（上限なし） | iPad等、画面に合わせて拡大・縮小 |
| デスクトップ | 1024px~ | 動的計算（下限30px・上限50px） | Phase 1 実装後のフィードバックにより固定値から動的計算に変更 |

### 1.2 レイアウト仕様

#### スマートフォン縦向き（portrait）
```
┌──────────────────┐
│ Header（コンパクト）│
│ SkillGauge        │
│ StatusBar         │
├──────────────────┤
│                  │
│   GameBoard      │
│  （画面幅に合わせ） │
│                  │
├──────────────────┤
│ Controller       │
│ （画面下部固定）   │
└──────────────────┘
```

- セルサイズ: `min((画面幅 - 32px) / 12, (画面高さ - UI高さ) / 18)`
  - 幅と高さの両方を考慮し、画面に収まる最大サイズを算出
  - UI高さ = Header + SkillGauge + StatusBar + Controller（約200px）
- コントローラーは画面下部に固定（`position: sticky; bottom: 0`）
- ゲーム画面が画面内に収まるよう幅・高さ両方で制約

#### スマートフォン横向き（landscape）
```
┌────────────────────────────────┐
│ Header（1行コンパクト）          │
├──────────┬─────────────────────┤
│ GameBoard│ HUD + Controller    │
│          │ ┌───────────────┐  │
│          │ │ SkillGauge    │  │
│          │ │ StatusBar     │  │
│          │ │ PowerUp       │  │
│          │ ├───────────────┤  │
│          │ │ ← 🔫 →       │  │
│          │ └───────────────┘  │
└──────────┴─────────────────────┘
```

- セルサイズ: `(画面高さ - 80px) / 18`
- ゲーム画面とHUD/コントローラーを横並び（`flex-direction: row`）
- `@media (orientation: landscape)` で切り替え

#### タブレット（~1024px）
- 既存レイアウトを維持（縦積み中央寄せ）
- セルサイズ: `min((画面幅 - 32px) / 12, (画面高さ - UI高さ) / 18)`
- 画面に合わせて拡大・縮小（上限なし）

#### デスクトップ（1024px~）
- 既存レイアウトを維持（縦積み中央寄せ）
- セルサイズ: 動的計算（下限30px・上限50px）
  - ※ 当初30px固定としたが、フィードバックにより大画面でもゲームが拡大表示されるよう動的計算に変更

### 1.3 ブレイクポイント

```typescript
const BREAKPOINTS = {
  mobile: 480,      // スマートフォン
  tablet: 768,      // タブレット
  desktop: 1024,    // デスクトップ
} as const;
```

### 1.4 useResponsiveSize フック改修

現在の実装を拡張し、以下を返す:

```typescript
interface ResponsiveLayout {
  cellSize: number;          // セルのピクセルサイズ
  isLandscape: boolean;      // 横向きかどうか
  isMobile: boolean;         // モバイルかどうか
  controllerPosition: 'bottom' | 'side'; // コントローラー位置
}
```

#### セルサイズ計算ロジック

```
横向き（width > height かつ width <= 1024px）:
  cellSize = (height - 80) / 18

縦向き（上記すべて、デスクトップ含む）:
  widthBased  = (width - 32) / 12
  heightBased = (height - UI高さ) / 18
  cellSize    = min(widthBased, heightBased)

全共通:
  cellSize = clamp(cellSize, 30, 50)  // 下限30px・上限50px
```

**重要**: すべての画面サイズで動的にセルサイズを計算する。
下限30px（小画面でもゲームが小さくなりすぎない）、上限50px（大画面でもセルが大きくなりすぎない）でクランプする。

※ 当初デスクトップは30px固定としたが、フィードバックにより大画面でゲームが小さすぎるとの指摘を受け、全画面サイズで動的計算に変更。

## 3. コンボシステム

### 3.1 基本仕様

| パラメータ | 値 | 説明 |
|----------|---|------|
| コンボウィンドウ | 2秒 | この時間内の連続ライン消しでコンボ成立 |
| 最大コンボ倍率 | 5.0x | コンボ倍率の上限 |
| 倍率増加量 | +0.5x / コンボ | 1コンボごとの倍率増加 |
| スキルゲージボーナス | +10% / 5コンボ | 5コンボごとにゲージボーナス |

### 3.2 コンボ倍率テーブル

| コンボ数 | 倍率 | 表示テキスト |
|---------|------|------------|
| 0 | 1.0x | （非表示） |
| 1 | 1.0x | （非表示） |
| 2 | 1.5x | "2 COMBO!" |
| 3 | 2.0x | "3 COMBO!!" |
| 5 | 3.0x | "5 COMBO!!!" |
| 8 | 4.5x | "8 COMBO!!!!" |
| 10+ | 5.0x | "MAX COMBO!!!!!" |

### 3.3 UI仕様

- **コンボ表示**: ゲーム画面中央上部にフローティング表示
- **フォント**: 太字、サイズはコンボ数に応じて拡大（1.5rem → 2.5rem）
- **色**: コンボ数に応じてグラデーション（白 → 黄 → 赤 → 紫）
- **アニメーション**: ポップイン（scale 0→1.2→1.0）+ フェードアウト
- **表示時間**: 2秒（コンボ継続中はリセット）

### 3.4 実装方針

新規フック `use-combo-system.ts` を作成:

```typescript
interface ComboState {
  count: number;          // 現在のコンボ数
  multiplier: number;     // 現在の倍率
  isActive: boolean;      // コンボ継続中かどうか
  lastHitTime: number;    // 最後にブロックを破壊した時刻
}
```

## 9. テストモード

### 9.1 概要

ゲームの動作確認を効率化するための裏技モード。
通常プレイでは見えず、隠しコマンドでのみ起動できる。

### 9.2 起動方法

- **条件**: タイトル画面（`status === 'idle'`）でキーボードから「jin」を3回タイプ
- **検知ロジック**: キー入力バッファに直近の入力を保持し、「jinjinjin」（9文字）が揃った時点で有効化
- **有効化後**: セッション中は維持。ブラウザリロードでリセット（`sessionStorage` 等は使わず、React state のみ）
- **無効化**: テストモード中にもう一度「jin」を3回タイプすると無効化（トグル動作）

### 9.3 テストモードインジケーター

- 画面右上に小さなバッジ「TEST」を表示（半透明、ゲームプレイの邪魔にならない位置）
- テストモードが有効な間だけ表示

### 9.4 デバッグパネル

テストモード有効時、ゲーム画面の下部（縦向き）または右側パネル下部（横向き）にデバッグパネルを表示。
パネルは折りたたみ可能（デフォルトは展開）。

#### 9.4.1 グリッド操作

| 操作 | 説明 | ボタンラベル |
|------|------|------------|
| 行を埋める（1行） | 最下段をプレイヤーの列だけ空けて埋める | 「1行セット」 |
| 行を埋める（2行） | 最下段から2行分を同様に埋める | 「2行セット」 |
| 行を埋める（3行） | 最下段から3行分を同様に埋める | 「3行セット」 |
| 行を埋める（4行） | 最下段から4行分を同様に埋める | 「4行セット」 |
| グリッドクリア | グリッド上のブロックを全消去 | 「グリッドクリア」 |

**「行を埋める」の動作詳細**:
1. グリッドの最下段（プレイヤー行の1つ上）から指定行数分を対象とする
2. 対象行の全セルを色付きで埋める（プレイヤーの現在X座標のセルだけ空ける）
3. プレイヤーが弾を撃ち、空いたセルが埋まるとライン消しが発動
4. 複数行セットすれば同時消しのテストが可能

```
行を2行セットした場合の例（プレイヤーX = 6）:

行16: ■■■■■■□■■■■■  ← 6列目だけ空き
行17: ■■■■■■□■■■■■  ← 6列目だけ空き
行18: （プレイヤー行）
```

#### 9.4.2 エフェクト操作

| 操作 | 説明 | ボタンラベル |
|------|------|------------|
| 爆弾シェイク | bombShake を手動発火 | 「Bomb」 |
| ブラストシェイク | blastShake を手動発火 | 「Blast」 |
| ラインシェイク | lineShake を手動発火 | 「Line」 |
| ゲームオーバーシェイク | gameOverShake を手動発火 | 「G.Over」 |
| ハイスコア演出 | HighScoreEffect を3秒間表示 | 「Hi-Score」 |

#### 9.4.3 スコア・ステージ操作

| 操作 | 説明 | ボタンラベル |
|------|------|------------|
| スコア加算 | 現在スコアに +1000 | 「+1000」 |
| スキルゲージ満タン | スキルゲージを 100% にする | 「Skill MAX」 |
| ステージスキップ | 次のステージに即座に進む | 「Next Stage」 |

#### 9.4.4 状態表示

デバッグパネル上部に以下の内部状態をリアルタイム表示:

```
Combo: 3 (x2.0) | Skill: 45% | Score: 1250 | Stage: 2
```

### 9.5 実装方針

#### フック構成

```typescript
// 隠しコマンド検知 + テストモード状態管理
interface UseTestModeReturn {
  isTestMode: boolean;
  // テストモードOFF時は何もしないのでゲーム本体への影響なし
}

// 隠しコマンド検知: キー入力をバッファリングし「jinjinjin」を検出
// idle状態でのみ検知（プレイ中のキー入力は無視）
```

#### コンポーネント構成

```
TestModeIndicator  — 「TEST」バッジ（右上）
TestModePanel      — デバッグパネル（折りたたみ可能）
  ├─ GridControls    — 行セット・グリッドクリアボタン群
  ├─ EffectControls  — シェイク・ハイスコア演出ボタン群
  ├─ StateControls   — スコア加算・スキル・ステージ操作ボタン群
  └─ StateDisplay    — 内部状態リアルタイム表示
```

#### ゲーム本体への影響の最小化

- `FalldownShooterGame.tsx` に `useTestMode` フックを追加し、`isTestMode` が `true` の時のみ `TestModePanel` を描画
- テストモードの操作は既存フックの公開API（`gameState.updateState`、`shake.bombShake` 等）を呼ぶだけで、既存コードの内部には手を入れない
- テストモード関連のコードは `components/TestModePanel.tsx` と `hooks/use-test-mode.ts` に集約し、他のファイルへの変更は `FalldownShooterGame.tsx` のみ

## 6. コンボトリガーの修正

### 6.1 変更概要

コンボの発動条件を「弾でブロックを壊した回数」から「ライン消し回数」に変更する。

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| トリガー | 弾がブロックに当たるたび | ラインが消えるたび |
| コンボウィンドウ | 2秒 | 2秒（変更なし） |
| カウント単位 | ブロック1個 = 1ヒット | 1回のライン消し = 消えたライン数分のヒット |

### 6.2 設計意図

- ゲームの目的は「ブロックを揃えてライン消しすること」であり、弾でブロックを壊すことではない
- コンボがゲームの目的と連動することで、プレイヤーの行動指針が明確になる

## 7. 同時消しボーナス

### 7.1 基本仕様

複数ラインを同時に消した場合、ライン数に応じたボーナス倍率を適用する。
同時消しの方が連続消しより高得点になるようにし、ゲームバランスを保つ。

### 7.2 同時消しボーナス倍率テーブル

| 同時消しライン数 | ボーナス倍率 | 基本スコア計算 | スコア例（ステージ1） |
|----------------|------------|--------------|-------------------|
| 1ライン | 1.0倍 | 100 × 1 × 1.0 | 100点 |
| 2ライン | 1.5倍 | 100 × 2 × 1.5 | 300点 |
| 3ライン | 2.0倍 | 100 × 3 × 2.0 | 600点 |
| 4ライン | 3.0倍 | 100 × 4 × 3.0 | 1200点 |

### 7.3 スコア計算式

```
ラインスコア = CONFIG.score.line × 消したライン数 × 同時消しボーナス倍率 × ステージ × 難易度倍率 × コンボ倍率
```

### 7.4 バランス比較

連続1ライン消し（コンボあり） vs 2ライン同時消し（コンボなし）の比較：

| パターン | 計算 | スコア |
|---------|------|--------|
| 1ライン × 2回（コンボ2、倍率1.5x） | 100 + 100×1.5 | 250点 |
| 2ライン同時（コンボなし、倍率1.0x） | 100×2×1.5 | 300点 |

→ 同時消しの方が高得点となり、ゲームルールとして自然。

### 7.5 実装方針

同時消しボーナス倍率を定数として定義：

```typescript
const SIMULTANEOUS_LINE_BONUS: Record<number, number> = {
  1: 1.0,
  2: 1.5,
  3: 2.0,
  4: 3.0,
};
```

`use-game-loop.ts` のライン消しスコア計算にボーナス倍率を組み込む。

## 8. ライン消しスコアへのコンボ倍率適用

### 8.1 変更概要

現在コンボ倍率（`comboMultiplier`）は弾ヒットスコアにのみ適用されているが、
ライン消しスコアにも適用する。

### 8.2 変更前後の比較

| スコア種別 | 変更前 | 変更後 |
|-----------|--------|--------|
| 弾ヒット | `score × scoreMultiplier × comboMult` | `score × scoreMultiplier`（コンボ倍率を外す） |
| ライン消し | `cleared × 100 × stage × scoreMultiplier` | `cleared × 100 × 同時消しボーナス × stage × scoreMultiplier × comboMult` |

### 8.3 設計意図

- コンボはライン消しで発動するため、コンボ倍率もライン消しに適用するのが自然
- 弾ヒットスコアからコンボ倍率を外すことで、弾を連射するだけの稼ぎを防止
