# Racing Game ファミコン風クリア型ブラッシュアップ — 計画

- **対象**: `src/features/racing-game/`
- **着手日**: 2026-05-08
- **ブランチ**: `feature/racing-famicom-brushup-20260508`
- **ステータス**: 計画策定中

---

## 1. 目的

Racing Game に **「最後まで遊ぶと終わる」キャンペーンモード** を追加し、現状の単発レース／2P 対戦中心の構造から、**ファミコン時代のクリア型レースゲームの感触** を持つタイトルへブラッシュアップする。

リーダーボードやデイリーチャレンジといった live-service 的な要素は **意図的に採用しない**。「始まりがあって、終わりがある」体験に振り切る。

## 2. ターゲット体験

プレイヤーが体験すべき感触を 5 つの言葉で定義する:

1. **「全 8 ステージを駆け抜ける緊張感」** — タイマーが減る、チェックポイントで延長される、ファミコンらしい時間との戦い
2. **「次のステージに何があるかワクワクする」** — ステージ間の短いナラティブと舞台変化
3. **「ゴールしたときの達成感」** — クリア後にエンディングテキストとクレジットロール
4. **「もう一度上手く走り切りたい」** — タイムランク（GOLD/SILVER/BRONZE）でやり込み余地
5. **「あの頃のゲームっぽい」** — 8bit 的タイポ、チップチューン、シンプルな演出

## 3. 参考にしたファミコン時代のゲーム

実コードに照合しながら採用する要素を選定した。

| 参考タイトル | 年 | 採用する要素 | 採用理由 |
|------------|----|------------|---------|
| **Rad Racer** | 1987 | 全 8 ステージ進行 / **チェックポイント通過で残り時間延長** / ステージ後半ほど時間が厳しくなる | 現状のチェックポイント実装と相性が最良 |
| **Mach Rider** | 1985 | 10 セクター進行 / **各ステージの分岐ルート（A/B 選択）** / クリアでエンディング | 既存コース 6 種を 8 ステージに引き伸ばすのに分岐が有効 |
| **Outrun**（NES 移植） | 1989 | ステージ間のドライバー描写 / 舞台変化（昼→夕→夜） | キャンペーンに物語性を付与 |
| **Excitebike** | 1984 | **タイム目標による格付け（GOLD/SILVER/BRONZE）** / 短いステージ | やり込み動機を残しつつ周回強制を避ける |
| **F-1 Race** / **Top Gear** | 1984/1992 | チャンピオンシップ的なステージ通し進行 | キャンペーンの全体構造のメタファ |

不採用の要素（一応の検討記録）:
- Rad Racer の「擬似 3D スケーリングロード」 → 現状はトップダウン 2D。技術的飛躍が大きすぎるため見送り
- Mach Rider の「武器でバイクから敵を撃つ」 → ジャンルが変わる。コア体験がブレる
- Outrun の「ハイスコア順位を画面に登場させる」 → モダン Web では情報過多

## 4. 全体方針

### 4.1 既存資産の扱い

**極力既存ドメインを再利用する**。新ドメインの追加は最小限に絞る。

| 既存資産 | 扱い |
|----------|------|
| 6 コース（Forest/City/Mountain/Beach/Night/Snow） | **全て活用**。8 ステージのうち 2 ステージは既存コースの高難度バリアント |
| ドラフトカード（14 種） | **キャンペーンでは無効化**（既存 `RaceConfig.cardsEnabled = false` フラグで対応可）。フリー対戦・2P・CPU 戦では従来どおり使える |
| HEAT / ドリフト / 衝突 / 物理 | **無変更**で再利用。既存の純粋関数ドメインに手を入れない |
| チェックポイント（4 個 / コース） | 仕組みは流用。チェックポイント通過時に **時間延長イベント** を新たに発火する |
| ハイライト検出 | キャンペーンの結果画面でも引き続き利用（"FASTEST LAP" 等の表示） |
| ソロ／2P／CPU 戦 | **そのまま残す**。メニューに「キャンペーン」を追加する形 |
| score-repository | キャンペーン用に拡張（ステージごとのベストタイム＋ランク保存） |

### 4.2 新規追加するドメイン（最小限）

```
domain/race/
├── stage.ts          # ステージ定義（型・カタログ）
├── stage-progress.ts # ステージ進行ロジック（時間延長・クリア判定）
└── time-limit.ts     # 残り時間管理（純粋関数）
```

```
application/use-cases/
├── start-campaign-stage.ts
├── advance-stage-time.ts
├── checkpoint-time-bonus.ts
└── complete-campaign.ts
```

> 既存の `application/race-handler.ts` は **キャンペーン専用の更新処理を呼び出す薄い分岐** を加えるのみ。コアの race フェーズ更新は流用する。

### 4.3 アーキテクチャ原則

- 新規 Domain は **すべて純粋関数**（既存規約継承）
- Use Case は 1 ファイル 1 関数の単一責任
- Presentation は既存 `RacingGameNew.tsx` を拡張せず、**`RacingGameCampaign.tsx`** を別ファイルで追加する選択肢を残す
- 既存テストは壊さない。新規追加は TDD（Red → Green → Refactor）

## 5. スコープ

### 5.1 含むもの（IN SCOPE）

- キャンペーンモードの追加（メニュー → ステージ選択 → ステージ実走 → 結果 → 次ステージ or エンディング）
- 全 8 ステージの構成・データ
- 残り時間 + チェックポイント時間延長
- 残機 3 機制（GAME OVER で再挑戦）
- ステージアンロック方式（クリアしたステージから自由に再挑戦可能）
- タイムランク（GOLD / SILVER / BRONZE）
- 分岐ルート（一部ステージ：A コース / B コース 選択）
- ステージ間の 1〜2 行ナラティブ
- エンディング演出（テキスト + クレジットロール）
- ファミコン風 UI 演出（大きい英大文字、チップチューン BGM、シンプルなフェード）
- localStorage によるキャンペーン進捗保存

### 5.2 含まないもの（OUT OF SCOPE）

意図的に **やらないこと** を明示する:

- ❌ オンライン対戦・リーダーボード
- ❌ デイリー／ウィークリーチャレンジ
- ❌ ガチャ／スキン／マイクロトランザクション
- ❌ リプレイ機能
- ❌ 車両カスタマイズ（既存の単一車両を維持）
- ❌ 真エンディング / 隠しステージ（将来検討余地として残すが、本計画には含めない）
- ❌ 既存ドラフトカードの再設計（無効化のみ）
- ❌ 物理エンジンのフレームレート可変対応（既知の見送り）

## 6. フェーズ計画

### Phase 1 — コアキャンペーン（必須）

最小で「8 ステージ走ってエンディングに到達する」体験を成立させる。

- ステージカタログ（8 ステージのデータ）
- ステージ進行ドメイン + ユースケース
- 残り時間／チェックポイント延長／GAME OVER 判定
- ステージアンロック保存
- メニューにキャンペーン項目追加
- ステージ選択画面（ロック中はグレー表示）
- ステージ開始・終了・ゲームオーバー・エンディングの最低限の UI

### Phase 2 — 演出強化（推奨）

「ファミコン的」感触を肉付けする。

- ステージ間ナラティブ（intro テキスト 1〜2 行）
- タイムランク（GOLD / SILVER / BRONZE）算出と表示
- ステージタイトルの大型タイポ表示（フェーズ間トランジション）
- BGM 統合（ステージ別 + GAME OVER + エンディング）
- エンディングのクレジットロール
- 結果画面の演出強化（既存ハイライト機能を流用）

### Phase 3 — 分岐ルートと仕上げ（任意）

- ステージ 3 / 5 / 7 の分岐ルート（A/B 選択）
- 難易度選択（NORMAL / HARD）— Phase 1 のステージ時間を係数で調整
- 既知バグ確認・修正（2P 入力周辺、フレームレート可変）
- E2E テスト（Playwright）でキャンペーン全 8 ステージ完走シナリオ

## 7. リスクと対応

| リスク | 兆候 | 対応 |
|--------|------|------|
| キャンペーン追加で既存モードが壊れる | 既存テスト失敗 | キャンペーン追加は **既存ハンドラへの分岐追加** ではなく **新規ハンドラ + 設定フラグ** で隔離。既存 28 テストは無変更で通すことを必須条件にする |
| 8 ステージのバランス調整が長引く | プレイテストで「ステージ 3 が異様に厳しい」等の報告 | 時間係数を **データ駆動** にして JSON / TS 配列で一元管理。コード変更不要で調整可能に |
| ステージ間遷移のフェーズ管理が複雑化 | バグの巣になる | 既存の `GamePhase` Sum 型に `stage_intro` / `stage_clear` / `game_over` / `ending` を追加。状態遷移図を spec に明記し、テストで網羅 |
| ファミコン的演出が安っぽくなる | 「ただ大きい文字が出るだけ」 | フェード時間 / フォント / 音 / 振動の組み合わせを spec で固定。Excitebike / Rad Racer の動画を参照しつつ調整 |
| 分岐ルートの実装が膨らむ | Phase 3 が長期化 | 分岐は **コース ID を切り替えるだけ** の最小実装に留める。新コース生成は禁止 |

## 8. 成功条件

Phase 1 完了時:
- メニューから「キャンペーン」を選び、ステージ 1〜8 を順番に完走できる
- 制限時間切れで GAME OVER になり、再挑戦できる
- 全 8 ステージクリアでエンディング画面に到達する
- 既存のソロ／2P／CPU モードが従来どおり動作する（既存テスト全 28 件パス）

Phase 2 完了時:
- 各ステージにタイトル＋ナラティブ表示が入る
- ベストタイムに応じて GOLD/SILVER/BRONZE のランクが表示される
- BGM がステージ別に切り替わる
- 「ファミコン世代に響く」プレイテスト被験者の主観評価で過半数の好評を得る（被験者数は最低 3 名）

Phase 3 完了時:
- 分岐ルートが少なくとも 2 ステージで機能する
- 難易度 2 種で全クリア可能
- E2E でキャンペーン全 8 ステージ完走テストが緑

## 9. 進め方

- **PR は Phase 単位**で切る（Phase 1 を 1 PR、Phase 2 を 1〜2 PR、Phase 3 を 1〜2 PR）
- Phase 1 の中はさらに「ドメイン」「ユースケース」「UI」のサブ PR に分けてレビューしやすくしてもよい
- 各 PR の Description は本計画書の該当セクションへリンクする
