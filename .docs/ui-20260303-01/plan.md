# Game Platform UI/UX ブラッシュアップ計画 v2

## 1. 背景と目的

### 1.1 背景

前回の UI/UX 改善（`ux-20260301-01`）でプラットフォームの基盤整備（法的ページ、音声停止、ナビゲーション、アイコン設定など）を実施した。今回は残存する課題と、さらなるビジュアル面の向上を目的とする。

### 1.2 課題一覧

| # | 課題 | 影響度 | カテゴリ |
|---|------|--------|---------|
| 1 | 原始進化録の BGM がページ遷移後も鳴り続ける | 高 | バグ修正 |
| 2 | ホームページの背景が単調（グラデーションアニメーションのみ） | 中 | ビジュアル |
| 3 | ヘッダーと HeroSection で「Game Platform」が重複表示 | 中 | UX |
| 4 | SEO 対策に改善余地がある | 中 | SEO |
| 5 | ページ全体の演出がまだ物足りない | 低 | ビジュアル |

### 1.3 目的

1. 原始進化録の音声停止バグを根本解決する
2. 3層パララックス背景でホームページの没入感を大幅に向上させる
3. ヘッダーのデザインを洗練し、重複表示を解消する
4. SEO のベストプラクティスを追加適用する
5. ページ全体を彩る演出でプラットフォームの魅力を高める

### 1.4 スコープ外

- 各ゲーム内部のロジック変更（音声停止バグ修正は PrimalPath の audio.ts のみ対象）
- 新規ゲームの追加
- ユーザー認証・アカウント機能
- バックエンドの変更

## 3. フェーズ構成

全5フェーズで段階的に実装する。フェーズ1は他フェーズのブロッカーではないが、優先度が最も高い。

### フェーズ 1: 原始進化録 音声停止バグ修正

**目的:** ページ遷移時に原始進化録の BGM が確実に停止するようにする。

| 項目 | 内容 |
|------|------|
| 1-1 | `audio.ts` に `cleanup()` メソッドを追加 |
| 1-2 | `PrimalPathGame.tsx` にアンマウント時の cleanup useEffect を追加 |
| 1-3 | 動作確認テストの追加 |

**根本原因:** `BgmEngine.play()` が `setInterval` で BGM のノートを順次再生しているが、コンポーネントのアンマウント時に `clearInterval` と `BgmEngine.stop()` が呼ばれない。さらに、`setInterval` のコールバック内で `initAudio()` を呼び出しており、これが `AudioContext.resume()` を実行するため、`GamePageWrapper` の `stopAllAudio()` による `suspend()` が即座に打ち消されてしまう。

**修正方針:** Non-Brake Descent の実装パターン（`Audio.cleanup()` + `useEffect` cleanup）を踏襲する。

### フェーズ 2: 3層パララックス背景

**目的:** ホームページに迷宮の残響のタイトルバックと同様の3層パララックス背景を実装し、没入感を向上させる。

| 項目 | 内容 |
|------|------|
| 2-1 | ホームページ用 `HomeParallaxBg` コンポーネントの作成 |
| 2-2 | マウス追従ロジックの実装 |
| 2-3 | `GameListPage` への統合 |
| 2-4 | モバイル対応（タッチデバイスは自動ドリフトのみ） |
| 2-5 | `prefers-reduced-motion` 対応 |
| 2-6 | AI 生成用画像プロンプトの作成 |

**実装方針:**
- 迷宮の残響の `TitleScreen.tsx` のマウス追従方式を採用
- 3層（Far / Mid / Near）で異なるパララックス速度
- `willChange: 'transform'` による GPU 加速
- WebP フォーマットで軽量化

## 4. フェーズ間の依存関係

```
フェーズ 1（音声バグ修正）  ← 独立・最優先
フェーズ 2（パララックス背景）← 独立
フェーズ 3（ヘッダー洗練）  ← 独立
フェーズ 4（SEO 強化）     ← 独立
フェーズ 5（ページ演出）   ← フェーズ 2, 3 完了後が望ましい（背景・ヘッダーとの整合性）
```

- フェーズ 1〜4 は相互に独立して実装可能
- フェーズ 5 はフェーズ 2（背景）とフェーズ 3（ヘッダー）のデザインが確定してから実装するのが望ましい

## 6. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| パララックス背景のパフォーマンス低下 | フレームレート低下、バッテリー消費 | `will-change` ヒント、`prefers-reduced-motion` 対応、画像の適正サイズ管理 |
| AI 生成画像の品質がデザインに合わない | 視覚的統一感の欠如 | 複数のプロンプトバリエーションを用意し、選定の余地を残す |
| ヘッダー変更で既存ユーザーが混乱 | UX への不満 | 段階的な変更、既存のグラスモーフィズムデザインとの一貫性を保つ |
| SEO 変更がインデックスに影響 | 検索順位の一時的変動 | `canonical` URL を維持、リダイレクトは使わない |
| 演出過多でページが重くなる | Core Web Vitals 悪化 | `IntersectionObserver` で viewport 外のアニメーション停止 |

## 7. 検証方法

### 自動テスト

- `npm run build` でビルドエラーがないこと
- 既存テスト + 新規テストが全て通ること
- Lighthouse スコアのパフォーマンス / SEO カテゴリ確認

### 手動検証

- 原始進化録のゲームプレイ後にページ遷移し、音が停止することを確認
- 再度原始進化録に入った際に音声が正常に再生されること
- パララックス背景のマウス追従が滑らかに動作すること
- モバイル端末でパララックスがドリフトアニメーションに切り替わること
- ヘッダーのスクロール縮小アニメーションが自然であること
- Google Search Console でインデックス状況を確認
- `prefers-reduced-motion` 有効時にアニメーションが無効化されること

## 8. 実装順序の推奨

1. **フェーズ 1（音声バグ修正）** — ユーザー体験に直接影響するバグのため最優先
2. **フェーズ 4（SEO 強化）** — コード変更量が少なく、効果が継続的
3. **フェーズ 3（ヘッダー洗練）** — フェーズ 2 の背景との整合性を先に検討
4. **フェーズ 2（パララックス背景）** — AI 画像生成の待ち時間を考慮
5. **フェーズ 5（ページ演出）** — 他フェーズのデザインが確定してから実装
