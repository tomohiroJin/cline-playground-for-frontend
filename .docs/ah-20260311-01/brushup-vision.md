# Air Hockey ブラッシュアップ計画 v3

## 現状の到達点

前回計画（v2）のMVPは**全13項目が実装完了**。
- 台・マレット・パックの立体的な見た目 ✅
- ヒットストップ・ゴールスローモーション ✅
- ストーリーモード第1章（3ステージ）完全動作 ✅
- キャラクターアイコン画像（7人分） ✅
- VS画面・ダイアログ・ステージ選択 ✅

## ゴール

**「もう1回遊びたい」から「友達に薦めたい」へ引き上げる。**

そのために必要な3つの柱:
1. **没入感**: ストーリーをテキストだけでなく「見せる演出」にする
2. **ボリューム**: キャラクターの深みと世界観で「もっと知りたい」を生む
3. **共有体験**: 2人で遊べることで「一緒にやろう」と誘える理由を作る

---

## 独自分析: ゲームを面白くするために足りないもの

### A. ストーリー演出の薄さ（優先度: ★★★★★）

**現状の課題**:
- ダイアログはテキスト+アイコンのみ。背景も暗転一色で場面の雰囲気が伝わらない
- VS画面は名前とアイコンだけで、「対決感」が弱い
- 試合前後の高揚感・緊張感が足りない
- ステージクリア時の達成感演出が控えめ

**ユーザーが離脱する理由**:
> 「ストーリーを読んでるだけで、体験していない」

### B. ストーリーのボリューム不足（優先度: ★★★★☆）

**現状の課題**:
- 第1章3ステージのみ。30分もかからず全クリア可能
- キャラクターの背景（なぜエアホッケーをしているのか等）が薄い
- クリア後のモチベーションが「フリー対戦に戻る」しかない
- 世界観が断片的で、プレイヤーが感情移入しきれない

### C. リプレイ性の弱さ（優先度: ★★★☆☆）

**現状の課題**:
- 一度クリアしたストーリーを再プレイする動機が薄い
- 実績・デイリーチャレンジはあるが、ストーリーモードとの連動が弱い
- 「上手くなった実感」を得る仕組みが足りない（タイムアタック、スコアランク等）

### D. 対人戦の不在（優先度: ★★★☆☆）

**現状の課題**:
- エアホッケーは本来2人で遊ぶゲーム。対人戦がないのは根本的な欠落
- 「友達に見せたい」と思っても一緒に遊べない

### E. キャラクター表現の限界（優先度: ★★★☆☆）

**現状の課題**:
- キャラクターが128x128アイコンでしか視覚化されていない
- 対戦中のリアクションが吹き出しテキストのみ
- キャラクター固有の「戦い方の個性」が視覚的に伝わらない

---

## ユーザー提案の評価と批評

### 案1: 2人協力プレイ・対戦プレイ ★★★☆☆（優先度: 中）

**良い点**:
- エアホッケーの本質は対人戦。ジャンルとの相性は最高
- 「友達と一緒に遊べる」は最強の共有動機
- 協力プレイ（2v1 CPU）は新しい遊び方を提供できる

**批評・懸念点**:
- **技術的コスト大**: マルチタッチの入力分離（特にiOS Safari）、キーボード同時入力（WASD + 矢印）の実装は複雑
- **対戦モードは「見た目の差」が出にくい**: 対人戦は仕組みがシンプルなぶん、ストーリーや演出の改善ほど「面白い」の印象に直結しない
- **協力プレイは仕様が曖昧**: 2人で1つのマレットを操作？ 2つのマレットで2v1？ どちらもバランス調整が難しい
- **テストが困難**: 2人同時入力のテストは自動化しづらく、手動テストの負荷が高い

**結論**: 面白さの根幹に関わるが、先にストーリー演出を充実させてからの方が「2人で一緒にストーリーを進める」等の展開も見えてくる。**Phase 3以降で着手を推奨**。

---

### 案2: ストーリーを絵で飾った派手な見た目にする ★★★★★（優先度: 最高）

**良い点**:
- 現状最大の弱点（テキストだけのストーリー体験）を直接解決する
- 「見た目がいい」はスクリーンショット映え→SNS共有に直結
- ゲーム体験の「記憶に残る度」が格段に上がる
- 既存のストーリーフレームワーク（ダイアログ、VS画面）にビジュアルを追加するだけで効果が出る

**批評・懸念点**:
- **画像アセットの制作コスト**: コマ割り漫画風にすると画像枚数が膨大になる
- **スタイルの統一**: AI生成画像のスタイル一貫性を保つのが難しい
- **ロード時間**: 画像が増えるとモバイルでの読み込みが遅くなる可能性

**具体的にできること**:
1. **ダイアログ背景画像**: 場面ごとの背景（部室、体育館、屋上等）を追加
2. **キャラクター立ち絵**: ダイアログ時にアイコンではなく半身像を表示
3. **VS画面の演出強化**: キャラの全身イラスト + 炎・雷などのエフェクト
4. **ステージクリア時のカットイン**: 勝利時に特別なイラストを表示
5. **チャプタータイトル画像**: 「第1章 はじめの挑戦」のタイトルカード

**結論**: 最もインパクトが大きく、既存の仕組みに乗せやすい。**Phase 1で最優先着手を推奨**。

---

### 案3: キャラ設定の深掘り・紹介ページ・設定ドキュメント ★★★★☆（優先度: 高）

**良い点**:
- キャラクターの深みは「次の章が楽しみ」という動機を強化する
- 設定を先に作ることで、第2章以降のストーリー展開が自然に広がる
- 紹介ページはゲーム内のコレクション要素になる（キャラ図鑑）

**批評・懸念点**:
- **設定だけでは面白くならない**: 設定資料がどんなに良くても、ゲーム内で体験できなければプレイヤーには伝わらない
- **紹介ページは「読むコンテンツ」**: ゲーム体験との温度差が出やすい。プロフィールを読むだけでは面白みに欠ける
- **設定の肥大化リスク**: 凝りすぎると本編のストーリーとの整合性維持が大変になる

**より効果的なアプローチ**:
1. **設定はドキュメントとして先に作る**（開発用・将来参照用）
2. **ゲーム内では「キャラクター図鑑」として段階的に開放**
3. **紹介ページは対戦前に自然に見せる**（VS画面にプロフィールを少しずつ表示）
4. **設定はストーリー内で「見せる」**（説明的にせず、ダイアログや行動で表現）

**結論**: 設定ドキュメント作成は**案2（ビジュアル強化）の前提作業として有効**。ビジュアルを作る際にキャラの立ち位置・背景が明確だと絵の指示も出しやすい。**Phase 1の前段階（Phase 0）として着手を推奨**。

---

## 優先順位の総合判定

```
優先度: 案2（ビジュアル強化） > 案3（設定深掘り） > 案1（2人対戦）
```

**理由**:
1. **案2が最優先**: 「見て面白い」は「遊んで面白い」の入口。現時点で最も足りていないのは視覚的な没入感
2. **案3は案2の下地**: キャラ設定が先にあれば、ビジュアル制作の方向性が定まる。ただしゲーム内実装よりドキュメント整備を先行
3. **案1は安定してから**: 2人対戦はストーリー演出が完成した後なら「2人でストーリーを進める」という発展形も設計可能

## 実装フェーズ

### Phase 0: 世界観・キャラ設定の整備（案3の前半）
**目的**: ビジュアル制作の土台を作る

- キャラクター詳細設定（性格、背景、戦い方の特徴、ビジュアルイメージ）
- 世界観設定（エアホッケー部の位置づけ、学校の雰囲気）
- 第2章以降のプロット概要
- 画像生成用のプロンプトガイドライン
- **成果物**: 設定ドキュメント（`.docs/` 配下）

### Phase 1: ストーリービジュアル強化（案2）
**目的**: 「見て面白い」を実現する

1. **ダイアログ背景画像の追加**
   - 場面別背景（部室、体育館等）のAI生成＋配置
   - DialogueOverlay への背景表示機能追加
2. **キャラクター立ち絵の導入**
   - 各キャラの半身像（表情差分: 通常・喜び・怒り・驚き）
   - ダイアログでアイコン→立ち絵に切り替え
3. **VS画面の演出強化**
   - キャラクターの全身 or 半身イラスト配置
   - 背景エフェクト（炎、雷、光線等）
   - 登場時アニメーション（スライドイン）
4. **勝利カットイン**
   - ストーリーステージクリア時の特別イラスト表示
   - 「TO BE CONTINUED...」画面の演出強化
5. **チャプタータイトルカード**
   - 「第1章 はじめの挑戦」等のタイトル画像

### Phase 2: キャラクター図鑑・ゲーム内紹介（案3の後半）
**目的**: 「もっと知りたい」を満たす

1. **キャラクター図鑑画面**
   - タイトル画面に「キャラクター」ボタン追加
   - 各キャラのプロフィールカード（立ち絵 + 設定情報）
   - ストーリー進行で段階的にアンロック
2. **VS画面へのプロフィール表示**
   - 「得意技」「一言」等の短い情報を VS 画面に追加
3. **リザルト画面のキャラ演出**
   - 勝利/敗北時のキャラクター表情差分表示

### Phase 3: 2人対戦モード（案1）
**目的**: 「一緒に遊ぼう」を実現する

1. **ローカル2人対戦（キーボード）**
   - 1P: 矢印キー / 2P: WASD
   - タイトル画面に「2P対戦」ボタン追加
2. **ローカル2人対戦（タッチ）**
   - 画面上半分/下半分で入力分離
   - マルチタッチ対応
3. **2人協力プレイ（2v1 CPU）** ← 仕様検討が必要
   - 2つのマレットで1つのCPUに挑戦
   - 専用のバランス調整

### Phase 4: 統合テスト・ストーリー第2章準備
**目的**: 品質担保と将来の基盤

- 全フェーズの統合テスト
- パフォーマンス計測（画像読み込み最適化）
- 第2章のストーリー・キャラ設計
- バランス調整

---

## リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| 画像アセットの量が膨大 | 制作期間が伸びる | 最小限の背景3枚 + 立ち絵4キャラ×2表情でスタート |
| AI生成画像のスタイル不統一 | 世界観が崩れる | Phase 0でスタイルガイド策定、同一プロンプト構造で生成 |
| 画像読み込みのパフォーマンス | モバイルで遅延 | WebP形式 + 遅延読み込み + プリロード戦略 |
| 2人対戦のタッチ入力問題 | iOS Safariで動作不良 | Phase 3ではキーボード対戦を先行、タッチは後追い |
| 設定の肥大化 | ストーリーとの矛盾 | 「核になる設定」と「装飾的設定」を分離して管理 |
| 立ち絵とアイコンのギャップ | キャラの印象が変わる | アイコンと同じAI生成プロンプトをベースに拡張 |
