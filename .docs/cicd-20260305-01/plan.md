# CI/CD パイプライン整備計画

## 1. 背景と目的

### 1.1 背景

Game Platform は React 19 + TypeScript で構築されたフロントエンドアプリケーションであり、13種類のブラウザゲームを収録している。現在、テスト（Jest）、リント（ESLint）、ビルド（Webpack）のツールチェーンは整備済みだが、以下の課題がある。

| # | 課題 | 影響度 | カテゴリ |
|---|------|--------|---------|
| 1 | PR に対する自動品質チェックが存在しない | 高 | CI |
| 2 | lint / test / build のゲートがないまま main にマージ可能 | 高 | 品質保証 |
| 3 | デプロイが手動で、手順が標準化されていない | 中 | CD |
| 4 | 本番ビルドの完全性を保証する仕組みがない | 中 | ビルドシステム |
| 5 | デプロイ時に全ファイルを転送しており非効率 | 低 | パフォーマンス |

### 1.2 目的

1. PR マージ前に lint / test / build が自動で実行され、全て通過しなければマージできない状態にする
2. デプロイ前に完全な品質チェック（型チェック含む）が通った状態を保証する
3. コンテナ環境へのデプロイを、認証付きの差分デプロイで効率化する
4. シェルスクリプトベースの再現可能なデプロイパイプラインを構築する

### 1.3 スコープ外

- バックエンド / API の CI/CD
- CDN の設定・キャッシュ戦略
- ステージング環境の構築
- E2E テスト（Playwright / Cypress）の導入
- Docker イメージのビルド・プッシュ（既存コンテナ環境を前提）

## 2. 現状の技術スタック

| カテゴリ | 技術 | バージョン / 備考 |
|---------|------|------------------|
| フレームワーク | React | 19.0.0 |
| 言語 | TypeScript | strict モード |
| バンドラ | Webpack | 5.x（ts-loader 使用） |
| テスト | Jest + Testing Library | ts-jest, jsdom 環境 |
| リント | ESLint | v9 Flat Config |
| フォーマッタ | Prettier | .prettierrc 設定済み |
| パッケージ管理 | npm | package-lock.json |
| デプロイ先 | コンテナ環境 | サーバー認証あり |

### 2.1 既存の npm スクリプト

```json
{
  "build": "NODE_ENV=production webpack --mode production",
  "test": "jest",
  "test:coverage": "jest --coverage",
  "lint": "eslint src"
}
```

### 2.2 不足しているスクリプト

| スクリプト | 用途 | 追加理由 |
|-----------|------|---------|
| `typecheck` | `tsc --noEmit` による型チェック | ビルド（ts-loader）とは独立した型安全性の検証 |
| `lint:ci` | CI 用リントコマンド（フォーマットチェック含む） | CI での再現性確保 |
| `ci` | lint + typecheck + test + build の一括実行 | デプロイ前の完全検証用 |

## 3. フェーズ構成

全4フェーズで段階的に実装する。

### フェーズ 1: npm スクリプトの整備

**目的:** CI/CD の基盤となるコマンドを追加・整理する。

| 項目 | 内容 |
|------|------|
| 1-1 | `typecheck` スクリプトを追加（`tsc --noEmit`） |
| 1-2 | `lint:ci` スクリプトを追加（`eslint src --max-warnings 0`） |
| 1-3 | `ci` スクリプトを追加（一括検証） |
| 1-4 | 各スクリプトの動作確認 |

**詳細:**
- `typecheck`: `tsc --noEmit` で型エラーを検出する。ビルドの ts-loader は型エラーがあっても警告で通過する場合があるため、独立した型チェックが必要
- `lint:ci`: `--max-warnings 0` で警告も許容しない厳格モード。PR のゲートとして使用
- `ci`: `npm run lint:ci && npm run typecheck && npm test && npm run build` を順次実行

---

### フェーズ 2: GitHub Actions CI ワークフロー

**目的:** PR に対して自動的に lint / typecheck / test / build を実行し、全て通過しなければマージ不可にする。

| 項目 | 内容 |
|------|------|
| 2-1 | `.github/workflows/ci.yml` の作成 |
| 2-2 | Lint ジョブの設定 |
| 2-3 | TypeCheck ジョブの設定 |
| 2-4 | Test ジョブの設定（カバレッジレポート付き） |
| 2-5 | Build ジョブの設定（ビルド成果物のアーティファクト保存） |
| 2-6 | ジョブ間の依存関係と並列実行の最適化 |

**ワークフロー設計:**

```
PR / push to main
    ├── lint（独立・最速）
    ├── typecheck（独立）
    ├── test（独立、カバレッジ付き）
    └── build（lint + typecheck + test 通過後）
```

- Lint / TypeCheck / Test は並列実行（互いに独立）
- Build は上記3ジョブが全て成功した後に実行（品質保証済みのコードのみビルド）
- Node.js バージョン: 20.x（LTS）
- npm キャッシュ: `actions/cache` で `node_modules` をキャッシュ

**トリガー条件:**
- `pull_request` → main ブランチへの PR
- `push` → main ブランチへの直接プッシュ（緊急時用、通常は PR 経由）

---

### フェーズ 3: ビルドシステムの完全性保証

**目的:** デプロイ前に「完全に完璧な状態」を保証するビルドパイプラインを構成する。

| 項目 | 内容 |
|------|------|
| 3-1 | プリデプロイ検証スクリプトの作成（`scripts/pre-deploy.sh`） |
| 3-2 | ビルド成果物の検証（dist/ の存在確認、ファイルサイズチェック） |
| 3-3 | ソースマップ除外の確認（本番ビルドに devtool: false 設定済み） |
| 3-4 | 依存関係の整合性チェック（`npm audit` 統合） |

**プリデプロイ検証フロー:**

```
scripts/pre-deploy.sh
    ├── 1. 依存関係の整合性確認（npm ci）
    ├── 2. リント実行（npm run lint:ci）
    ├── 3. 型チェック実行（npm run typecheck）
    ├── 4. テスト実行（npm test）
    ├── 5. 本番ビルド実行（npm run build）
    ├── 6. ビルド成果物の検証
    │   ├── dist/index.html の存在確認
    │   ├── dist/*.bundle.js の存在確認
    │   ├── 各ファイルサイズの上限チェック
    │   └── ソースマップファイルが含まれていないことの確認
    └── 7. 成功ログの出力
```

---

### フェーズ 4: デプロイパイプライン

**目的:** コンテナ環境への認証付き差分デプロイを実現する。

| 項目 | 内容 |
|------|------|
| 4-1 | デプロイスクリプトの作成（`scripts/deploy.sh`） |
| 4-2 | 環境設定ファイルのテンプレート作成（`.env.deploy.example`） |
| 4-3 | サーバー認証の仕組み（SSH 鍵 / トークン認証） |
| 4-4 | rsync による差分デプロイの実装 |
| 4-5 | デプロイ後のヘルスチェック |
| 4-6 | ロールバック手順の文書化 |

**デプロイフロー:**

```
scripts/deploy.sh
    ├── 1. 環境変数の読み込み（.env.deploy）
    ├── 2. プリデプロイ検証の実行（scripts/pre-deploy.sh）
    ├── 3. サーバー認証
    │   ├── SSH 鍵認証（推奨）
    │   └── または トークン認証（コンテナ環境の認証方式に依存）
    ├── 4. rsync による差分デプロイ
    │   ├── --checksum: ファイル内容の変更のみ転送
    │   ├── --delete: サーバー上の不要ファイルを削除
    │   ├── --exclude: 除外パターン（.env, .git 等）
    │   └── --compress: 転送時の圧縮
    ├── 5. デプロイ後ヘルスチェック
    │   ├── HTTP ステータスコード確認（200 OK）
    │   └── index.html の応答確認
    └── 6. デプロイログの記録
```

**差分デプロイの仕組み:**

rsync の `--checksum` オプションにより、ファイルのチェックサムを比較し、変更があったファイルのみを転送する。Webpack の `[contenthash]` チャンク命名と組み合わせることで、変更のないチャンクファイルはスキップされる。

**認証方式の検討:**

| 方式 | メリット | デメリット | 推奨度 |
|------|---------|----------|--------|
| SSH 鍵認証 | 安全、標準的 | 鍵管理が必要 | 推奨 |
| トークン認証 | コンテナ向き | トークンの定期更新が必要 | 代替 |
| VPN + パスワード | シンプル | セキュリティリスク | 非推奨 |

※ 実際の認証方式はコンテナ環境の構成に依存するため、デプロイスクリプトは認証部分をプラグイン可能な設計とする。

## 4. フェーズ間の依存関係

```
フェーズ 1（npm スクリプト整備）  ← 全フェーズの前提
    ↓
フェーズ 2（GitHub Actions CI）   ← フェーズ 1 完了後
フェーズ 3（ビルド完全性保証）    ← フェーズ 1 完了後
    ↓（フェーズ 2, 3 とも完了後）
フェーズ 4（デプロイパイプライン）← フェーズ 3 完了必須
```

- フェーズ 1 は全フェーズの土台
- フェーズ 2 と フェーズ 3 はフェーズ 1 完了後に並列実装可能
- フェーズ 4 はフェーズ 3 のプリデプロイスクリプトに依存

## 6. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| CI の実行時間が長い | 開発効率の低下 | npm キャッシュ、ジョブ並列化で最短化 |
| 既存テストの不安定さ（flaky test） | CI が不当に失敗 | `forceExit: true` 設定済み、問題発生時に個別対応 |
| デプロイ先サーバーへの接続失敗 | デプロイ不可 | リトライ機構、ヘルスチェック、ロールバック手順 |
| rsync の権限エラー | ファイル転送失敗 | 事前の権限確認スクリプト、`--chmod` オプション |
| Node.js バージョンの不一致 | ビルド結果の差異 | `.nvmrc` ファイルの追加、CI で明示的にバージョン指定 |
| 機密情報（SSH 鍵等）の漏洩 | セキュリティインシデント | GitHub Secrets 使用、`.env.deploy` を gitignore |

---

## 7. 実装順序の推奨

1. **フェーズ 1（npm スクリプト整備）** — 最小限の変更で全フェーズの土台を構築
2. **フェーズ 2（GitHub Actions CI）** — PR の品質ゲートを即座に有効化
3. **フェーズ 3（ビルド完全性保証）** — デプロイ前の安全装置を構築
4. **フェーズ 4（デプロイパイプライン）** — 最終的なデプロイ自動化

---

## 8. 成功基準

| 基準 | 測定方法 |
|------|---------|
| PR 作成時に CI が自動実行される | GitHub PR ページで CI ステータスが表示 |
| CI 失敗時に PR がマージ不可になる | ブランチ保護ルール設定後に確認 |
| lint / typecheck / test / build が全て通過する | CI の全ジョブが緑色 |
| デプロイスクリプトが差分のみ転送する | rsync の `--dry-run` 出力で変更ファイル数を確認 |
| デプロイ前に完全な検証が実行される | `pre-deploy.sh` が全チェック通過後にのみビルド成功 |
| デプロイ後にサイトが正常応答する | ヘルスチェックで HTTP 200 を確認 |
