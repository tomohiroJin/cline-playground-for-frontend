# Step 1: フィールドリサイズ — 実装計画

## 概要

エアホッケーの内部解像度を 450×900 から 600×1200 に拡大する。
マレット・パックのサイズは据え置きとし、フィールドに対する相対サイズを小さくすることで
「ゴール前無敵」問題を解消し、将来のペアマッチに必要な広さを確保する。

## 目標

1. **Canvas 内部解像度の拡大**: 450×900 → 600×1200（1.33x、アスペクト比 1:2 維持）
2. **要素サイズの最適化**: マレット/パック据え置き、ゴール・障害物はスケーリング
3. **全既存モードの非破壊**: フリー対戦・ストーリー・2P 対戦・デイリーが正常動作
4. **レンダラーの対応**: 描画のハードコード座標を定数ベースに修正
5. **物理演算の整合**: 速度定数・AI パラメータの再調整

## 設計原則

### スケーリング方針

「フィールドは広く、プレイヤーは相対的に小さく」を実現するため、以下の方針を取る:

| カテゴリ | スケーリング | 理由 |
|---------|-------------|------|
| Canvas サイズ | 1.33x (450→600, 900→1200) | フィールドを広くする |
| マレット半径 | 据え置き (42px) | 相対的に小さくして無敵問題を解消 |
| パック半径 | 据え置き (21px) | マレットとの比率を維持 |
| アイテム半径 | 据え置き (24px) | 視認性を維持 |
| ゴール幅 | 1.33x (120→160px 等) | フィールド幅に対する比率を維持 |
| 障害物座標 | 1.33x | フィールド内の相対位置を維持 |
| 障害物半径 | 据え置き | マレット/パックとの衝突バランスを維持 |
| 外枠・装飾 | 1.33x | フィールドに対する比率を維持 |

### 速度・物理パラメータの調整

フィールドが大きくなるため、以下の調整が必要:

| パラメータ | 現在値 | 調整方針 |
|-----------|--------|---------|
| KEYBOARD_MOVE_SPEED | 6 | 8 に増加（移動距離/フィールドの比率を維持） |
| CPU maxSpeed (easy/normal/hard) | 1.5/3.5/6.0 | 2.0/4.7/8.0 に増加（同上） |
| PHYSICS.MAX_POWER | 12 | 16 に増加（パック速度の上限） |
| PHYSICS.MIN_SPEED | 1.5 | 2.0 に増加（停止判定の閾値） |
| SPEED_NORMAL / SPEED_FAST | 6/10 | 8/13 に増加（演出閾値） |
| AI predictionFactor | 各種 | 1.33x（予測距離がフィールドに比例） |
| AI wobble | 各種 | 1.33x（ブレ幅がフィールドに比例） |

> **注意**: 最終値はプレイテストで微調整する。上記は 1.33x を基準とした初期値。

### CSS スケーリング

アスペクト比 1:2 は変更しないため、CSS の `aspect-ratio: 1 / 2` はそのまま。
`max-width` を 450px → 600px に変更し、大画面で広く表示されるようにする。

### Phase S1-1: 定数・型定義の更新

**目的**: フィールドサイズの変更を全コードに波及させる起点を作る

#### S1-1-1: Canvas サイズ定数の変更

- `core/constants.ts` の `CONSTANTS.CANVAS` を `{ WIDTH: 600, HEIGHT: 1200 }` に変更
- コメントを更新: `// 内部解像度 600x1200 固定`

#### S1-1-2: ゴール・障害物座標のスケーリング

- `core/config.ts` のフィールド定義を更新:
  - ゴールサイズ: 各フィールドの `goalSize` を 1.33x
  - 障害物座標: 各障害物の `x`, `y` を 1.33x
  - 障害物半径: 据え置き（マレット/パックとの衝突バランス維持）
- コメントを更新: `// 障害物座標は 600x1200 解像度基準`

#### S1-1-3: 物理・速度パラメータの調整

- `core/constants.ts`:
  - `PHYSICS.MAX_POWER`: 12 → 16
  - `PHYSICS.MIN_SPEED`: 1.5 → 2.0
  - `CPU`: easy 1.5→2.0, normal 3.5→4.7, hard 6.0→8.0
- `core/keyboard.ts`:
  - `KEYBOARD_MOVE_SPEED`: 6 → 8
- `infrastructure/renderer/renderer-utils.ts`:
  - `SPEED_NORMAL`: 6 → 8
  - `SPEED_FAST`: 10 → 13

**成果物**: 更新された constants.ts, config.ts, keyboard.ts, renderer-utils.ts
**完了条件**: `tsc --noEmit` で型エラーなし

### Phase S1-2: レンダラーの対応

**目的**: 描画がリサイズ後のフィールドで正しく表示される

#### S1-2-1: 描画関数のハードコード座標を定数化

- `renderer.ts` 内のハードコードされた座標値を `consts.CANVAS.WIDTH` / `consts.CANVAS.HEIGHT` ベースの計算に置換
- 主な対象:
  - 外枠の strokeRect のオフセット（6, 12 等）
  - 中央ラインの y 座標（H/2 で計算済みの箇所は OK）
  - 中央サークルの半径（60px → フィールド幅の 10% として計算）
  - ゴール LED の描画位置
  - 照明効果の放射グラデーション

#### S1-2-2: 外枠・装飾・HUD の座標調整

- 外枠のストローク幅（12px）はフィールド幅に対する比率で調整するか、据え置きか判断
- カウントダウン・ゴールエフェクトのテキストサイズ確認
- ポーズオーバーレイの座標確認

#### S1-2-3: エンティティ描画の確認

- マレット描画: サイズは `consts.SIZES.MALLET` 参照なので変更不要（確認のみ）
- パック描画: 同上
- アイテム描画: 同上
- 障害物描画: 座標は `obstacle.x/y` 参照、半径は `obstacle.r` 参照（確認のみ）

**成果物**: 更新された renderer.ts, entity-renderer.ts
**完了条件**: ビルド成功、Canvas 描画が正しい

## リスク管理

| リスク | 影響度 | 対策 |
|--------|--------|------|
| レンダラーのハードコード座標が多すぎて修正漏れ | 高 | renderer.ts を全文検索し、数値リテラルを洗い出す |
| フィールド拡大でゲームバランスが崩れる | 中 | 速度パラメータの初期値は 1.33x を基準とし、プレイテストで微調整 |
| テスト内のハードコード座標が多い | 中 | テスト修正を最後にまとめて実施 |
| CSS スケーリングで小画面デバイスの表示が崩れる | 低 | max-height 制約があるため自動的に縮小される |
| 既存の localStorage データとの互換性 | 低 | ゲーム状態は毎回新規生成、保存データは影響なし |
