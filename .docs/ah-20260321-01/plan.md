# Air Hockey Phase 3: 2P 対戦モード — 実装計画

## 概要

ブラッシュアップビジョン Phase 3 に基づき、ローカル 2P 対戦機能を実装する。
エアホッケー本来の「対面で遊ぶ楽しさ」を提供し、「一緒にやろう」と誘える体験を実現する。

## 目標

1. **キーボード 2P 対戦**: 1P（矢印キー）/ 2P（WASD）で同時操作
2. **タッチ 2P 対戦**: 画面上下分割のマルチタッチ操作
3. **キャラクター選択**: 2P 対戦時に各プレイヤーがキャラを選択
4. **対戦演出**: VS 画面・リザルト画面の 2P 対応
5. **既存モードの非破壊**: フリー対戦・ストーリーモードに影響なし

## 設計原則

### 既存アーキテクチャとの整合

DDD リファクタリング（R1-R5）で整備されたレイヤー構成を活用する。

| レイヤー | 変更内容 |
|---------|---------|
| **domain** | `PlayerInput` 型、`GameMode` に `'2p-local'` 追加、2P スコアリング |
| **application** | `TwoPlayerBattleUseCase` の追加 |
| **infrastructure** | 入力アダプタの拡張（キーボード 2P、マルチタッチ） |
| **presentation** | キャラ選択画面、2P 対戦ボタン、VS/リザルト画面改修 |

### 入力の抽象化

現在の入力は「1P = マウス/タッチ/キーボード」「2P = CPU AI」で固定されている。
これを以下のように抽象化する:

```
InputSource (抽象)
  ├── MouseInputSource     (マウス/タッチ — 1P 用)
  ├── KeyboardInputSource  (矢印キー — 1P 用)
  ├── WasdInputSource      (WASD — 2P 用)
  └── TouchZoneInputSource (画面上下分割 — 1P/2P 用)
```

ゲームループは `InputSource` を 2 つ受け取り、各プレイヤーのマレット位置を更新する。
CPU 対戦時は 2P 側の `InputSource` を AI に差し替える（既存と互換）。

### 段階的リリース

キーボード対戦を先行実装し、タッチ対戦は後追いで実装する。
キーボード対戦だけでも完全な対戦体験が成立するため、途中で止めても価値がある。

### Phase 3-1: 2P 入力システム

**目的**: 2 人のプレイヤーが同時に独立したマレット操作を行える入力基盤を構築する

#### 3-1-1: 入力抽象化

- `domain/contracts/input.ts` に `InputSource` インターフェースを定義
  ```typescript
  // プレイヤー入力の抽象インターフェース
  interface InputSource {
    // 現在の入力位置を取得（Canvas 座標系）
    getPosition(): Vector | undefined;
    // 入力がアクティブかどうか
    isActive(): boolean;
    // フレーム開始時の更新処理
    update(deltaTime: number): void;
    // リソース解放
    dispose(): void;
  }
  ```
- `PlayerSlot` 型の定義（`'player1' | 'player2'`）
- 入力とプレイヤーの紐づけ管理

#### 3-1-2: WASD キーボード入力

- 既存の `useKeyboardInput.ts` を拡張
- WASD キー（W:上, A:左, S:下, D:右）の入力マッピング
- 1P（矢印キー）と 2P（WASD）の独立した `KeyboardState` 管理
- 同時押し時の干渉防止

#### 3-1-3: マルチタッチ入力

- `useMultiTouchInput.ts` を新規作成
- 画面を上下 2 分割し、各領域のタッチを独立して追跡
- `Touch.identifier` でタッチポイントを区別
- 各タッチゾーンの座標をマレット位置に変換
- iOS Safari のマルチタッチ制約への対応（`touch-action: none`, `preventDefault`）

#### 3-1-4: 既存入力の適合

- 現在のマウス/タッチ入力（`useInput.ts`）を InputSource 形式に適合
- 現在のキーボード入力（`useKeyboardInput.ts`）を InputSource 形式に適合
- 後方互換を維持し、既存の CPU 対戦フローに影響を与えない

**成果物**: `domain/contracts/input.ts`, 拡張された入力フック群
**完了条件**: 1P + 2P のキーボード入力が同時に動作、既存テスト全パス

### Phase 3-2: ゲームモード拡張

**目的**: ゲームループが CPU 対戦と 2P 対戦の両方をサポートする

#### 3-2-1: GameMode に '2p-local' を追加

- `core/types.ts` の `GameMode` 型に `'2p-local'` を追加
  ```typescript
  type GameMode = 'free' | 'story' | 'daily' | '2p-local';
  ```
- `ScreenType` に `'characterSelect'` を追加（2P 対戦設定画面）
- 型定義の影響範囲を確認し、関連コードを更新

#### 3-2-2: ゲームループの 2P 対応

- `presentation/hooks/useGameLoop.ts` の改修:
  - `GameLoopConfig` に `gameMode` と `player2Input` を追加
  - `gameMode === '2p-local'` 時は AI 更新をスキップし、2P 入力を適用
  - CPU マレットの位置更新を 2P 入力に差し替える分岐
- 物理演算・衝突判定は変更なし（マレット位置の入力元が変わるだけ）
- アイテムシステムも変更なし

#### 3-2-3: 2P 対戦ユースケース

- `application/use-cases/two-player-battle.ts` を新規作成
  ```typescript
  class TwoPlayerBattleUseCase {
    start(config: TwoPlayerConfig): void;
    getResult(): BattleResult;
    getMatchStats(): MatchStats;
  }
  ```
- `TwoPlayerConfig`: フィールド、勝利スコア、各プレイヤーのキャラ選択
- 実績・アンロック判定は 2P 対戦では無効化（CPU 相手の実績のみ有効）

#### 3-2-4: スコア・勝敗管理の 2P 対応

- スコア表示を「Player vs CPU」から「1P vs 2P」に変更（2P モード時）
- キャラ名が選択されている場合はキャラ名で表示
- 勝敗判定ロジックは変更なし（winScore 到達で終了）

**成果物**: 改修された useGameLoop, 新規 TwoPlayerBattleUseCase
**完了条件**: キーボード 2P 対戦が動作、CPU 対戦に影響なし

### Phase 3-3: キャラクター選択

**目的**: 2P 対戦時に各プレイヤーがキャラクターを選び、マレットに反映する

#### 3-3-1: キャラクター選択画面

- `components/CharacterSelectScreen.tsx` を新規作成
  - 1P / 2P それぞれのキャラ選択パネル
  - アンロック済みキャラのアイコン + 名前をグリッド表示
  - 選択済みキャラにハイライト表示
  - 同キャラ選択可能（ミラーマッチ）
  - フィールド・勝利スコアの設定
  - 「対戦開始！」ボタン（両プレイヤーがキャラを選択したら有効化）
- 表示するキャラクター:
  - 図鑑でアンロック済みのキャラクター（ストーリー進行依存）
  - 初期選択: 1P = player（アキラ）、2P = hiro（ヒロ）

#### 3-3-2: マレットのキャラクター反映

- マレット描画時にキャラクターのテーマカラーを適用
- マレット上にキャラアイコンを小さく表示（オプション）
- 1P / 2P のマレットを視覚的に区別（色 + ラベル）

#### 3-3-3: 2P 対戦設定フロー

- タイトル画面 → キャラ選択画面 → VS 画面 → ゲーム → リザルト の遷移定義
- 各画面間の状態の受け渡し（選択キャラ、設定値）

**成果物**: CharacterSelectScreen, マレット描画改修
**完了条件**: キャラ選択 → 対戦 → リザルトの一連フローが動作

### Phase 3-4: 画面・演出対応

**目的**: 2P 対戦に合わせた UI と演出を提供する

#### 3-4-1: タイトル画面に 2P 対戦ボタン追加

- TitleScreen に「2P 対戦」ボタンを追加
- ボタンスタイル: オレンジ系グラデーション（既存ボタンと差別化）
- ボタン位置: ストーリー/キャラクターボタンの下
- `onTwoPlayerClick` コールバック Props の追加

#### 3-4-2: VS 画面の 2P 対応

- VS 画面で 2 人のキャラクター情報を表示
- 「CPU」ラベルの代わりに「2P」ラベルを表示
- 選択キャラの VS 画像・アイコンを使用
- 演出は既存 VS 画面と共通

#### 3-4-3: リザルト画面の 2P 対応

- 勝者を「Player Win!」ではなく「1P Win!」/「2P Win!」で表示（2P モード時）
- キャラ名がある場合は「{キャラ名} Win!」
- 勝者キャラの happy 表情 + 敗者キャラの normal 表情
- 「もう1回」ボタンで同設定で再戦（キャラ選択からやり直しも可能）
- 2P 対戦では実績・アンロック通知は表示しない

#### 3-4-4: ゲーム中の 2P 表示

- スコア表示のラベルを「Player / CPU」→「1P / 2P」（2P モード時）
- キャラ名がある場合はキャラ名で表示
- ゲーム開始カウントダウンは変更なし

**成果物**: 改修された TitleScreen, VS 画面, ResultScreen, ゲーム中 UI
**完了条件**: 2P 対戦の全画面遷移が正しく動作

## リスク管理

| リスク | 影響度 | 対策 |
|--------|--------|------|
| WASD と矢印キーの同時入力が一部ブラウザで制約 | 中 | 主要ブラウザ（Chrome, Firefox, Safari）で動作確認、`keydown`/`keyup` ベースで実装 |
| マルチタッチが iOS Safari で不安定 | 高 | キーボード対戦を先行リリース、タッチ対戦は別フェーズでも可 |
| ゲームループの 2P 分岐による複雑化 | 中 | InputSource 抽象化で分岐を最小限に。AI と 2P 入力を同じインターフェースで扱う |
| キャラ選択画面の UX 複雑化 | 低 | 初期選択を自動設定し、最小操作で対戦開始できるようにする |
| 既存テストの破損 | 中 | 各フェーズ完了時に全テスト実行、後方互換を維持 |
