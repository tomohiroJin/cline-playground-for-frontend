# Air Hockey Phase 3 — ナラティブフロー

## 前提: ここまでの到達点

プレイヤー（アキラ）はエアホッケー部の仲間と出会い（Phase 0）、
彼らの姿を立ち絵・背景・VS演出で「目で見て体験」し（Phase 1）、
図鑑で各キャラクターの深みに触れた（Phase 2）。
DDD リファクタリング（R1-R5）により、コードは拡張しやすい土台が整った。

しかし、ここまでの体験は **「ひとりの時間」** で完結している。

---

## 物語: 「もうひとりの手が、テーブルの向こう側に」

### 起: 画面越しのさびしさ

> アキラは強くなった。ストーリーで仲間と出会い、CPU 相手に腕を磨いてきた。
> でもどこかで気づいている——
> 「本物の緊張感は、隣にいる誰かと一打を競う瞬間にある」。

**ユーザーにとっての課題:**
- 「面白い！」と思っても、その感動を隣にいる人と分かち合えない。
- CPU は学習しないが、人間の友達は予測不能。そのドキドキが欠けている。
- 「一緒にやろう」と誘えないゲームは、共有体験を生まない。

---

### 承: テーブルの両側に立つ

エアホッケーは本来、**2人が向かい合うスポーツ**。
ゲーム内でも、プレイヤー同士が同じ画面・同じキーボードで対戦できるようになる。

#### 体験 1: ローカル 2P 対戦（キーボード）

> ヒロが言う。「アキラ、ちょっと俺と勝負してみろよ」
> 画面には「2P 対戦」ボタン。隣に座った友達が WASD に手を置く。

- **1P: 矢印キー** / **2P: WASD** でマレットを操作
- タイトル画面に「2P 対戦」ボタンが現れる
- 難易度・フィールド・勝利スコアは自由に選べる
- 対戦中は CPU の AI がオフになり、2P がマレットを直接操作
- リザルト画面で勝者が表示され、再戦を誘う

この体験は **「隣にいる人と今すぐ遊べる」** という最も原始的な楽しさを提供する。

#### 体験 2: ローカル 2P 対戦（タッチ）

> スマホを横に置いて、画面の上半分と下半分で指を動かす。
> まるで本物のエアホッケー台のように。

- 画面を上下に分割し、それぞれのタッチ領域でマレットを操作
- マルチタッチ対応で 2 本の指が同時に動ける
- モバイルでの体験がキーボードなしでも完結する

#### 体験 3: キャラクター選択

> 「俺はタクマでいく」「じゃあ私ミサキね」

- 2P 対戦時、それぞれのプレイヤーがキャラクターを選べる
- 選んだキャラのアイコンとテーマカラーがマレットに反映される
- VS 画面で選んだキャラ同士の対決演出が入る
- リザルト画面でキャラの表情差分が表示される

---

### 転: 壁を乗り越える

2P 対戦を実現するには、いくつかの壁を越える必要がある。

**壁 1: 入力の分離**
現在の入力は「プレイヤー 1 人 + CPU」前提。
2P では 2 つの独立した入力を同時に処理する必要がある。
→ キーボード入力の拡張（WASD 対応）から着手し、タッチは後追い。

**壁 2: ゲームモードの拡張**
ゲームループが「CPU vs Player」で固定されている。
→ `gameMode` に `'2p-local'` を追加し、AI の代わりに 2P 入力を注入。

**壁 3: テストの困難さ**
2 人同時入力のテストは自動化しづらい。
→ ドメイン層のユニットテスト + ユースケース結合テストで振る舞いを保証。
→ UI レベルの 2 入力同時テストはキーボードイベントのシミュレーションで対応。

---

### 結: 「一緒にやろう」が自然に生まれる

2P 対戦が実装されたことで:

- ストーリーで知ったキャラを **「自分で」選んで対戦できる**
- 「面白いゲーム見つけた！一緒にやろう」が自然に言える
- **隣にいる人との笑い・悔しさ・もう一回!** がゲーム体験の核になる
- エアホッケーというジャンルが本来持つ「対面で遊ぶ楽しさ」が蘇る

> ミサキが微笑む。
> 「ゲームって結局、誰かと一緒の時が一番楽しいでしょ？」

## スコープ整理

### Phase 3 で実装する（本スコープ）

| 優先度 | 内容 | 理由 |
|--------|------|------|
| ★★★★★ | ローカル 2P 対戦（キーボード） | 最小コストで対人戦を実現。1P 矢印 / 2P WASD |
| ★★★★☆ | キャラクター選択画面 | 図鑑で育てた愛着をゲームプレイに接続する |
| ★★★★☆ | 2P 用 VS 画面・リザルト画面 | 対決感と再戦動機の強化 |
| ★★★☆☆ | ローカル 2P 対戦（タッチ） | モバイルでも遊べる体験を提供 |

### Phase 3 では実装しない

| 内容 | 理由 |
|------|------|
| オンライン対戦 | インフラコストが高く、ローカル対戦で十分な体験が得られる |
| 2P 協力（2v1 CPU）| バランス調整が複雑、仕様検討が不十分 |
| ストーリー第2章 | Phase 4 で着手予定 |
| presentation/components の完全移動 | スコープを限定するため次フェーズ以降 |

### Phase 2 残課題（Phase 3 で併せて対応）

| 内容 | 対応方針 |
|------|---------|
| プロフィールカードの閉じるフェードアウト | 2P 実装の合間にパッチ対応 |
| club フィールドの表示検討 | プロフィールカード改修時に判断 |

```
  → スコア管理の 2P 対応

  → キャラクター選択画面コンポーネント
  → 選択キャラのマレットへの反映
  → 2P 対戦設定画面
      ↓
Phase 3-4: 画面・演出対応
  → VS 画面の 2P 対応
  → リザルト画面の 2P 対応
  → タイトル画面に 2P 対戦ボタン追加
      ↓
Phase 3-5: テスト・品質保証
  → 2P 入力のユニットテスト
  → ゲームモード結合テスト
  → 既存モードの非破壊確認
```
